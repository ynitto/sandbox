# 中級者ガイド — Agent Skills を使いこなす

> **対象読者**: 基本的なスキルの利用方法を理解している方、チームでの運用やカスタマイズに関心がある方
>
> **前提**: [初心者ガイド](guide-beginners.md) の内容を把握していること

---

## この資料でわかること

1. 自分だけのスキルを作る
2. フィードバックでスキルを育てる
3. チームでスキルを共有する（ノードフェデレーション）
4. バージョン管理とロールバック
5. プロファイルによるスキルセットの切り替え
6. 既存コードベースからスキルを自動生成する

## 1. 自分だけのスキルを作る

### スキルの構造

スキルは **SKILL.md を中心としたフォルダ** です。

```
my-skill/
├── SKILL.md          ← エージェントへの指示書（必須）
├── scripts/          ← 補助スクリプト（任意）
├── references/       ← 参考ドキュメント（任意）
└── assets/           ← テンプレート・リソース（任意）
```

### skill-creator でスキルを作成する

ゼロから手書きする必要はありません。`skill-creator` に依頼すれば対話的にスキルを生成できます。

```
「Python の FastAPI バックエンド開発を支援するスキルを作って」
「このプロジェクトの命名規則をチェックするスキルを作って」
```

skill-creator は以下の手順で動作します：

1. スキルの目的・対象・制約をヒアリング
2. SKILL.md のドラフトを生成
3. ユーザーの確認を経て `.github/skills/` に配置

### SKILL.md の書き方のコツ

SKILL.md はエージェントが読む「指示書」です。以下のポイントを押さえると品質が上がります。

| ポイント | 良い例 | 悪い例 |
|---------|--------|--------|
| **目的を明確に** | 「REST API のエンドポイントを OpenAPI 3.0 仕様で設計する」 | 「API を作る」 |
| **手順を段階的に** | 「Step 1: 既存のエンドポイントを一覧化…」 | 手順なしで結果だけ書く |
| **出力形式を指定** | 「出力は YAML の OpenAPI スキーマとする」 | フォーマット指定なし |
| **禁止事項を明示** | 「認証トークンをログに出力しないこと」 | 禁止事項なし |

### ワークスペーススキルとユーザースキルの違い

| 種類 | 配置場所 | 用途 |
|------|---------|------|
| ワークスペーススキル | `.github/skills/` | このプロジェクト専用 |
| ユーザースキル | `<AGENT_HOME>/skills/` | 全プロジェクト共通 |

同名のスキルが両方にある場合、**ワークスペーススキルが優先** されます。

## 2. フィードバックでスキルを育てる

### フィードバックの仕組み

スキルは使い捨てではありません。フィードバックを記録することで、スキルの品質を定量的に追跡・改善できます。

```
「react-frontend-coder に ok を付けて」
「code-reviewer に needs-improvement を記録して。セキュリティ指摘が甘い」
「systematic-debugging に broken を記録して」
```

### verdict（判定）の使い分け

| verdict | 意味 | いつ使う？ |
|---------|------|-----------|
| `ok` | 期待通りに動作した | スキルの出力が実用的だったとき |
| `needs-improvement` | 改善の余地がある | 動くが品質や手順に不満があるとき |
| `broken` | 動作しなかった | エラーで完了できなかったとき |

### メトリクスの確認

フィードバックは自動集計されてレジストリに記録されます。

```json
{
  "metrics": {
    "total_executions": 15,
    "ok_rate": 0.87,
    "last_executed_at": "2026-02-27T09:00:00Z"
  }
}
```

- `ok_rate` が高いスキルは安定して使える
- `ok_rate` が低いスキルは改善が必要 → `skill-evaluator` が改善案を自動提案

### スキルの評価と改善

蓄積されたフィードバックを基に、スキルを評価・改善できます。

```
「試用中スキルを確認して」          → skill-evaluator がフィードバックを分析
「このスキルを改善して」            → skill-creator が SKILL.md を更新
```

## 3. チームでスキルを共有する

### 全体像：ノードフェデレーション

Agent Skills は **各メンバーのローカル環境（ノード）** と **中央リポジトリ** を連携させるフェデレーションモデルを採用しています。

```
中央リポジトリ（信頼された知識の集積）
    ↓ pull（自動スナップショット → いつでも戻せる）
ノード（あなたのローカル環境）
    ├─ スキルを試用・改善
    ├─ フィードバックを蓄積
    └─ 条件が揃ったら中央に貢献（push → PR）
中央リポジトリ（レビュー・統合）
```

### 3-1. スキルを取得する（pull）

```
「スキルをpullして」
```

- pull 前に自動でスナップショットが保存される
- ローカルで改善済みのスキルは上書きされない（`protect_local_modified: true`）

### 3-2. スキルを共有する（push）

ローカルで作成・改善したスキルを中央リポジトリに共有できます。

```
「スキルをpushして」
「react-frontend-coder を中央リポジトリに共有して」
```

push はブランチを切って PR を出すだけなので、**完成していなくても気軽に共有できます**。

### 3-3. スキルを昇格する（promote）

ワークスペース内のスキル（`.github/skills/` にあるもの）を、ユーザー領域（`<AGENT_HOME>/skills/`）にコピーして他のプロジェクトでも使えるようにします。

```
「ワークスペースのスキルを他のプロジェクトでも使えるようにして」
「react-frontend-coder を昇格して」
```

### 3-4. ノードアイデンティティ

チームで運用する場合、各メンバーの環境に一意の ID が付与されます。これにより「誰の改善か」を追跡できます。

レジストリの `node` フィールドで確認できます：

```json
{
  "node": {
    "id": "node-abc123",
    "name": "tokyo-team-dev",
    "created_at": "2026-02-27T00:00:00Z"
  }
}
```

### 3-5. 自動昇格通知

フィードバックが蓄積されると、昇格ポリシーに基づいて自動で通知されます。

**昇格条件（デフォルト）：**

- ok 件数 ≥ 3
- 問題率 ≤ 10%
- ローカルで改善済み（`local_modified: true`）

条件を満たすと「このスキル、中央にあげませんか？」と提案されます。

## 4. バージョン管理とロールバック

### スキルのバージョン固定

特定のバージョンで動作を安定させたい場合、スキルを固定できます。

```
「react-frontend-coder を今のバージョンに固定して」
「全スキルをロックして」
```

固定されたスキルは `pull` で上書きされません。

### スナップショットとロールバック

pull 前に自動保存されたスナップショットを使って、いつでも前の状態に戻せます。

```
「スナップショット一覧を見せて」          → 保存済みスナップショットを表示
「直近のスナップショットに戻して」        → 最新のスナップショットにロールバック
```

スナップショットには以下が保存されます：

```
snapshot-20260227T103000/
├── meta.json           # 作成日時・スキル一覧
├── skill-registry.json # レジストリの完全コピー
└── skills/             # <AGENT_HOME>/skills/ の完全コピー
```

### ローカル差分の追跡

中央リポジトリのスキルをローカルで改善した場合、差分が自動追跡されます。

レジストリの `lineage` フィールドで確認できます：

```json
{
  "lineage": {
    "origin_repo": "team-skills",
    "origin_commit": "a1b2c3d",
    "origin_version": "1.2.0",
    "local_modified": true,
    "diverged_at": "2026-02-20T00:00:00Z",
    "local_changes_summary": "RSC対応を追加"
  }
}
```

## 5. プロファイルによるスキルセットの切り替え

プロジェクトや役割に応じて、利用するスキルのセットを切り替えられます。

### プロファイルの例

```json
{
  "profiles": {
    "default": ["*"],
    "frontend": ["react-*", "ui-designer", "webapp-testing", "tdd-executing"],
    "backend": ["api-designer", "domain-modeler", "dynamodb-designer"],
    "review": ["architecture-reviewer", "code-reviewer", "code-simplifier", "design-reviewer", "document-reviewer", "security-reviewer", "test-reviewer"],
    "gitlab": ["gitlab-idd"],
    "patent": ["patent-coach", "patent-writer"]
  }
}
```

### プロファイルの切り替え

```
「frontend プロファイルに切り替えて」
「backend プロファイルに切り替えて」
「デフォルトに戻して」
```

### 使い分けのコツ

| シーン | おすすめプロファイル |
|--------|-------------------|
| React の画面開発 | `frontend`（React 系 + UI + テスト + TDD） |
| API サーバー開発 | `backend`（API 設計 + ドメインモデル） |
| コードレビュー | `review`（全レビュー系スキル） |
| GitLab イシュー駆動開発 | `gitlab`（gitlab-idd） |
| 特許・知財 | `patent`（patent-coach + patent-writer） |
| 何でもやる | `default`（全スキル有効） |

## 6. 既存コードベースからスキルを自動生成する

`skill-creator`（モードB: コードベース変換）は、既存のプロジェクトを分析して **そのプロジェクト専用のスキル** を自動生成します。

### 使い方

```
「このプロジェクトのアーキテクチャをスキル化して」
「コードの規約をスキルにまとめて」
「このリポジトリをスキルにして」
```

### 何が生成されるか

- プロジェクトのコーディング規約
- アーキテクチャパターン
- 開発ワークフロー（ビルド・テスト・デプロイ手順）

これにより、新メンバーが参加したときに **エージェントがプロジェクトの慣習を理解した状態で** 作業を支援できます。

## 7. git-skill-manager の操作一覧

| 操作 | フレーズ例 | 説明 |
|------|-----------|------|
| リポジトリ登録 | 「リポジトリを登録して」 | スキルの取得元を追加 |
| スキル取得 | 「スキルをpullして」 | 最新スキルをダウンロード |
| スキル共有 | 「スキルをpushして」 | ローカルスキルを中央に共有 |
| 検索 | 「スキルを探して」 | キーワードでスキルを検索 |
| 有効化/無効化 | 「スキルを有効化/無効化して」 | スキルの利用を切り替え |
| バージョン固定 | 「スキルを固定して」 | pull による上書きを防止 |
| 昇格 | 「スキルを昇格して」 | ワークスペース → ユーザー領域 |
| フィードバック | 「〇〇のフィードバックを記録して」 | ok / needs-improvement / broken |
| プロファイル切替 | 「〇〇プロファイルに切り替えて」 | スキルセットを変更 |
| 変更履歴生成 | 「CHANGELOGを作って」 | スキルの変更履歴を生成 |

## 8. 実践的なワークフロー例

### 例1：新しいプロジェクトを立ち上げる

```
1. 「このリポジトリをスキルにして」
     → skill-creator（モードB）がプロジェクト固有スキルを生成

2. 「React + TypeScript で管理画面を作って。テストも書いて。デプロイ設定もして」
     → scrum-master が以下を自動実行：
       Sprint 1: コンポーネント設計 + API 接続
       Sprint 2: テスト作成 + CI/CD 設定

3. 各スプリント終了後にフィードバックが自動収集される
```

### 例2：チームにスキルを展開する

```
1. プロジェクト固有のスキルを .github/skills/ に作成
     → 「レビュー観点をまとめたスキルを作って」

2. チームメンバーが pull で取得
     → 「スキルをpullして」

3. 各自がローカルで改善・フィードバック記録

4. 良い改善は中央に貢献
     → 「このスキルを中央リポジトリに共有して」
```

### 例3：スキルの品質管理サイクル

```
1. スキルを使う → フィードバック記録（ok / needs-improvement）
2. フィードバック蓄積 → skill-evaluator が改善提案
3. 改善実施 → skill-creator で SKILL.md 更新
4. ok_rate が向上 → 昇格ポリシーが自動通知
5. 中央リポジトリにマージ → チーム全体に反映
```

## 9. ディレクトリ構成リファレンス

```
<AGENT_HOME>/
├── skills/              ← インストール済みスキル（ユーザー領域）
│   ├── scrum-master/
│   │   ├── SKILL.md
│   │   ├── scripts/
│   │   └── references/
│   ├── git-skill-manager/
│   └── ...
├── cache/               ← リポジトリキャッシュ（永続）
├── snapshots/           ← pull 前の自動バックアップ
│   └── snapshot-{timestamp}/
└── skill-registry.json  ← レジストリ（全設定の中心）
```

```
プロジェクト/
├── .github/
│   ├── skills/          ← ワークスペーススキル（プロジェクト固有）
│   └── copilot-instructions.md  ← エージェントへの共通指示
└── ...
```
