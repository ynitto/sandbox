# Agent Dashboard: 定常業務・実行状況の段階表示UI設計

## 背景と目的

定常業務画面は、登録済みの業務をすべて大きなカードで並べるため、件数が少ないと間延びし、
件数や説明が増えると一覧性が下がる。実行状況画面は、一覧、工程、イベント、ログなどが同時に
見えるため、利用者が現在の状態と次に行う操作を判断しにくい。

両画面を「対象を選ぶ」「概要を確認する」「必要な詳細だけ開く」の順に統一する。
定常業務画面では選択した業務だけを下部に表示し、実行状況画面では選択した定常業務に属する
実行だけを扱う。データ量や文字量が変わっても主要領域と操作位置が変わらない構成にする。

## 検討した案

| アプローチ | 見やすさ | 情報量への耐性 | 画面の安定性 | 実装コスト | 推奨度 |
|---|---|---|---|---|---|
| 上部セレクター＋下部詳細 | 高 | 高 | 高 | 中 | ★★★ |
| 左一覧＋右詳細 | 高 | 中 | 高 | 中 | ★★☆ |
| 全件カード＋折りたたみ | 中 | 低 | 中 | 低 | ★☆☆ |

## 採用設計

### 共通の情報階層

両画面を次の3段階で構成する。

1. **対象の選択** — 名前、状態、時刻など、選択に必要な情報だけを表示する
2. **現在の概要** — 状態、直近結果、次回予定、主要操作を常時表示する
3. **技術的な詳細** — 履歴、工程、イベント、ログ、設定は利用者が要求したときだけ表示する

選択領域と概要領域の高さ・配置はデータによって変えない。長い名前と説明は省略し、全文は
ツールチップまたは詳細領域で確認できるようにする。履歴やログは画面全体を押し広げず、
専用領域の内部でスクロールさせる。

### 定常業務画面

画面上部に、登録済み定常業務を選択するコンパクトなセレクターを置く。各項目に常時表示するのは
次の情報だけとする。

- 業務名
- 稼働状態
- 最終実行結果
- 次回実行予定

初期選択は、現在選択中の業務を維持し、無い場合は対応が必要な業務、直近更新された業務の順に
決定する。セレクターが横幅に収まらない場合は専用領域内でスクロールし、下部詳細の幅や位置を
変えない。

選択中の業務だけを下部へ表示し、次の順に構成する。

1. **現在の状態**
   - 稼働状態
   - 最終実行日時と結果
   - 次回予定
   - 主操作の「実行」ボタン
2. **基本情報**
   - 対象リポジトリ
   - 実行方式
   - スケジュール
   - 実行元
3. **最新結果**
   - 成功・失敗の短い要約
   - ログの短いプレビュー
   - 「実行状況を見る」操作
4. **詳細情報**
   - 実行履歴
   - ログ全文
   - 設定編集
   - 削除

履歴、ログ全文、設定は初期状態で展開しない。編集と削除は主要操作から分離し、詳細メニューへ
配置する。実行中は「実行」ボタンを状態表示へ置き換え、同じ位置を維持する。

### 実行状況画面

画面上部で定常業務を選択し、その業務に属する実行だけを下部へ表示する。全業務の実行を横断して
確認する必要がある場合は、明示的な「すべて」選択肢を用意するが、初期表示にはしない。

選択後の領域を次の順に構成する。

1. **実行の選択**
   - 実行日時
   - 状態
   - 所要時間
   - 再実行や派生関係の短い表示
2. **実行概要**
   - 現在の状態
   - 開始時刻と経過時間
   - 進捗
   - 成功・失敗・実行中の工程数
   - キャンセルなど、その状態で有効な主要操作
3. **工程タイムライン**
   - 工程名
   - 状態
   - 所要時間
   - 短い結果要約
4. **工程詳細**
   - 選択した工程の入出力
   - エラー
   - イベント
   - ログ

工程タイムラインは全工程を同じ密度で表示するが、技術情報は選択した1工程だけについて開く。
ログとイベントは折りたたみまたは詳細パネルで遅延表示する。失敗時は失敗工程を初期選択するが、
詳細領域を画面上部へ割り込ませない。

## 状態管理とデータ取得

### 基本方針

一覧・概要データはまとめて取得し、画面内の選択やフィルター変更では再取得しない。サイズの大きい
履歴、実行詳細、ログだけを必要時に取得する。

- 定常業務概要: プロジェクト単位で取得し、メモリ上のスナップショットを再利用する
- 実行一覧: 定常業務ID単位で絞り込み可能な概要データとして保持する
- 実行詳細: `project + runId` 単位で遅延取得し、キャッシュする
- 定常業務履歴: `project + routineId` 単位で初回表示時に取得し、キャッシュする
- ログ: 初期表示には含めず、ログ表示時に取得する

既存の実行概要一覧を全体取得している場合は、業務選択による絞り込みをクライアント側で行う。
概要一覧が大きくなり、取得量または更新頻度が問題になった時点で、定常業務IDによるサーバー側
絞り込みへ移行できる境界を維持する。

### キャッシュの無効化

時間切れだけに依存せず、状態を変更する操作に合わせて該当範囲を無効化する。

- 手動更新: 選択中プロジェクトの概要と表示中詳細を再取得する
- 実行開始: 対象業務の概要、実行一覧、最新実行詳細を無効化する
- 編集: 対象業務の概要と履歴を無効化する
- 削除: 対象業務に属するすべてのキャッシュを削除し、安全な次項目へ選択を移す
- キャンセル・再実行: 対象実行、実行一覧、対象業務の概要を無効化する
- プロジェクト変更: 表示選択を切り替え、別プロジェクトのキャッシュと混在させない
- 一覧の `status` または `updatedAt` が変化: 対応する実行詳細を無効化する

キャッシュには保持上限を設け、最近使用されていない詳細から解放する。取得中の同一キーに対する
要求は1つへまとめ、連続選択による重複リクエストを防ぐ。選択変更中に古い応答が返っても、現在の
選択先へ表示しない。

## レイアウト安定性

- 上部セレクター、概要、詳細の領域を明確に分ける
- Flex/Gridの子要素に `min-width: 0` と `min-height: 0` を設定し、長文によるはみ出しを防ぐ
- 名前と要約には行数上限を設ける
- ログ、イベント、工程詳細は専用スクロール領域にする
- 読み込み中も概要の枠を維持し、スケルトンまたはプレースホルダーを表示する
- 空状態、エラー状態でも主要操作と見出しの位置を維持する
- 狭い画面では概要の列だけを1列化し、情報の順序は変えない

## アクセシビリティ

- セレクターには選択状態を示す `aria-selected` または `aria-pressed` を付与する
- 左右キーまたは上下キーで項目を移動できるようにする
- 選択変更後もフォーカスを失わず、詳細見出しとの関係を明示する
- フォーカス表示を常に視認できるようにする
- 状態は色だけで表現せず、状態名とアイコンを併記する
- エラーは `role="alert"` または `aria-live` で通知する
- 主要なクリック領域は44px相当を確保する

## エラー・空状態

- 定常業務が無い場合は、登録操作を1つだけ主操作として案内する
- 選択した業務に実行が無い場合は、未実行であることと「実行」操作を表示する
- 詳細取得に失敗しても概要と選択状態は維持し、詳細領域だけで再試行できるようにする
- 古いキャッシュを表示する場合は更新日時を示し、バックグラウンド再取得に失敗しても画面全体を
  空にしない
- 権限やデータ欠損によるエラーは利用者向け要約と技術詳細を分離する

## テスト方針

### 表示と操作

- 定常業務を選択すると、下部にその業務だけが表示されること
- 初期表示で現在状態、最終結果、次回予定、実行ボタンが確認できること
- 履歴、ログ、設定が初期状態で展開されないこと
- 実行状況で選択した定常業務に属する実行だけが表示されること
- 工程選択前に技術詳細が過剰表示されず、選択後は1工程分だけ表示されること
- 実行失敗時に失敗工程が判別でき、詳細を開けること
- 長い名称、ログ、大量の業務・工程でも主要領域が押し出されないこと

### データ取得

- 業務選択だけでは定常業務概要APIを再呼び出ししないこと
- 同じ実行を再選択した場合は有効な詳細キャッシュを利用すること
- 同一詳細への並行リクエストが1回に集約されること
- 実行、編集、削除、キャンセル、再実行後に該当キャッシュだけが無効化されること
- 古い非同期応答が新しい選択内容を上書きしないこと
- キャッシュ保持数が上限を超えないこと

### アクセシビリティと画面幅

- キーボードだけで業務、実行、工程を選択できること
- 選択状態、エラー、実行状態を支援技術が認識できること
- 375px、768px、1024px、1440px相当で横方向にはみ出さないこと
- 表示内容が増減しても主要操作の位置と情報順序が変わらないこと

## 既存スキルとの関係

- `ui-designer`: 段階的開示、マスター・詳細構造、フォーカス、オーバーフロー対策に活用する
- `webapp-testing`: 実装後の選択導線、画面幅別表示、キーボード操作を確認する
- `performance-profiler`: 現時点では使用せず、API呼び出し回数をテストで計測して不足時に導入する
- `react-frontend-coder`: 対象はプレーンElectronのため使用しない
