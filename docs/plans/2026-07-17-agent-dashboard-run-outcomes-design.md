# Agent Dashboard: 実行結果・タスク状態・成果表示の設計

## 背景

実行タブでは agent-flow の run が `done` になると「完了」と表示される。一方、agent-project の
タスクは外側の検証、検収、承認を終えるまで完了ではない。この二つを同じ「完了」で表すため、
利用者にはタスクまで完了したように見える。また、完了runから成果、コード差分、作業ブランチや
ターゲットブランチへ直接到達できない。

既存の成果ダイアログは検収時の差分表示を備えるが、要確認項目を入口としており、完了runからは
開けない。左右比較の内側にあるDiff2Htmlの要素がコンテナ幅を押し広げ、コード差分がダイアログ
外へはみ出す場合もある。

## 目的

- runの終了とタスクの完了を明確に区別する。
- 完了runから成果へ迷わず到達できるようにする。
- 実装の所在を理解するためのブランチ情報を表示する。
- 既存の検収ダイアログを、回答操作を伴わない成果閲覧にも再利用する。
- コード差分のスクロールをダイアログ内部に閉じ込める。

## 過去の設計判断との関係

既存の「概要から詳細へ」「一覧から一件を選択」「日常情報と技術情報を段階的に開示する」という
方針を維持する。成果は日常利用で必要な情報なので実行概要から直接開けるようにし、run IDや生ログ
のような内部情報は引き続き技術情報へ置く。ブランチ情報は成果の所在を説明する情報として成果表示
に含める。

## 検討した案

| アプローチ | 実装コスト | リスク | 保守性 | 拡張性 | 学習コスト | 推奨度 |
|---|---|---|---|---|---|---|
| A. 状態二段表示＋成果ダイアログ再利用 | 中 | 低 | 高 | 高 | 低 | ★★★ |
| B. 実行概要へ成果をすべて展開 | 低 | 中 | 低 | 中 | 低 | ★★☆ |
| C. 実行詳細へ成果サブタブを追加 | 高 | 中 | 高 | 高 | 中 | ★★☆ |

案Aを採用する。概要の情報量を増やしすぎず、既存の差分閲覧・MR・ファイル選択を一つの成果表示へ
集約できるためである。

## 状態表示

実行詳細の先頭で、状態を二段に分ける。

- **実行**: agent-flow runの状態。`done` は「実行完了」とする。
- **タスク**: 関連するbacklog/archiveタスクの状態。「検収待ち」「要対応」「実行中」「タスク完了」
  などを表示する。

runが完了してもタスクがarchiveされていない場合は、警告調ではなく判断を促す情報として
「実行は完了しましたが、タスクはまだ完了していません」を表示し、必要なら要対応タブへの導線を
出す。archive済みの場合だけ「タスク完了」「納品済み」と表示する。工程集計のラベルはタスク完了と
混同しないよう「完了」から「工程完了」へ変更する。

関連タスクを解決できないagent-flow単体runでは、タスク状態を推測せず「タスクとの関連なし」とする。

## 成果への導線

`done` runの実行概要に、主要操作「成果を見る」を表示する。run選択時にダイアログを自動表示せず、
利用者の操作で開く。失敗runや進行中runではこの主要操作を表示しない。

成果モデルは次の優先順で組み立てる。

1. 関連する要確認項目の構造化delivery情報とMR
2. 関連タスク・納品記録から取得できる成果参照
3. runのworkspace、references、final、工程のdata/output

上位の情報がない場合も、下位の情報で成果要約と実装情報を表示する。コード差分を取得できないこと
自体はダイアログ全体のエラーにせず、「差分を表示できません」と局所表示する。

## 成果ダイアログ

既存の検収ダイアログを、次の二つのモードで共有する。

- **検収モード**: 現行どおり回答欄、承認、差し戻し、AI補助を表示する。
- **成果閲覧モード**: runから開き、回答操作を表示しない。タスク状態、成果要約、MR、変更ファイル、
  差分だけを表示する。

成果閲覧モードではリポジトリごとに次を表示する。

- リポジトリ名・URL・ローカル所在
- 作業ブランチ（workspace `branch`）
- ターゲットブランチ（workspace `target`、未指定なら `base`）
- ベースブランチ（workspace `base`）
- MRへのリンク
- 変更ファイルとコード差分

複数リポジトリの場合も既存の左側リポジトリ／ファイル一覧を使う。参照専用リポジトリは成果差分の
対象外であることを明示する。

## レイアウトとoverflow

ダイアログ本体はビューポート幅を超えない。grid/flex配下の成果ペインとDiff2Htmlラッパーへ
`min-width: 0` と `max-width: 100%` を伝搬させ、長いコードとside-by-sideテーブルの横スクロールは
`.delivery-diff-view` の内部だけで扱う。ファイル名、ブランチ名、パスは折り返しまたは省略表示する。

狭い画面ではリポジトリ／ファイル一覧と差分を一列へ切り替える。左右比較が可読幅を確保できない
場合は行単位表示へ縮退し、ページ全体に水平スクロールを発生させない。キーボードフォーカスと
ダイアログの見出し構造は維持する。

## データフロー

1. run選択時に既存のflow詳細と関連タスクを解決する。
2. 表示用のrun状態とタスク状態を別々に導出する。
3. 「成果を見る」でrun IDを入口に成果モデルを構築する。
4. 関連needがあれば既存deliveryを再利用し、なければworkspace等から読み取り専用モデルを合成する。
5. 差分要求は既存の限定IPCを使い、取得結果をダイアログ内部へ描画する。

## エラーハンドリング

- 関連タスク不明: run状態だけ表示し、タスク状態は「関連なし」とする。
- 成果情報なし: 空のダイアログにせず、runのfinalまたは「記録された成果情報はありません」を表示する。
- リポジトリ/ref解決失敗: 対象リポジトリ内へエラーを表示し、他の成果は閲覧可能にする。
- 差分が長い: 既存の文字数上限を維持し、打ち切りを明示する。
- 古いrun: 読み取り専用の記録として表示し、現在タスクの操作を誤って出さない。

## テスト

- `done` run＋review/blockedタスクは「実行完了」「タスク未完了」を併記する。
- `done` run＋archiveタスクだけが「タスク完了」「納品済み」になる。
- agent-flow単体runはタスク状態を推測しない。
- 完了runに「成果を見る」が表示され、進行中runには表示されない。
- 成果モデルがneed、task/delivery、workspaceの順でフォールバックする。
- 成果閲覧モードでは検収操作を表示しない。
- 作業／ターゲット／ベースブランチを区別して表示する。
- Diff2Htmlの長い行・長いファイル名でもダイアログ外へはみ出さない。
- 375px、768px、1024px、1440pxで水平スクロールが発生しない。
- 既存のdelivery-review、flow、task-completion-hintテストが通る。

## 既存スキルとの関係

- `ui-designer`: 状態階層、主要操作、レスポンシブな差分閲覧、アクセシビリティに使用する。
- `systematic-debugging`: Diff2Htmlのどの内側要素が幅を押し広げるかを仮説検証して修正する。
- `webapp-testing`: 実画面でダイアログの横幅、スクロール領域、成果導線を検証する。
