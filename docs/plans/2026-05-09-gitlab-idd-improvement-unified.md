# gitlab-idd 改善案 — 統合版（2026-05-09）

スキーマ固定案（A〜E）とブレスト案（I-1〜V-3）を、案同士のつながりに基づいて再グルーピングした統合ドキュメント。

---

## 根本ボトルネック

すべての改善案は以下6つの根本課題のいずれかに対応している。

| # | 根本課題 | 現れ方 |
|---|---|---|
| **R-1** | **仕様品質が低い** | AC が曖昧・非検証可能 → needs-clarification 多発、レビュー差し戻し頻発 |
| **R-2** | **完成基準が分断されている** | イシュー AC ≠ プロジェクト横断 DoD → テスト・セキュリティ・ドキュメントが漏れる |
| **R-3** | **検証に実装者バイアスが入る** | agent-reviewer が汎用 perspective を使う、ワーカー自身の目線でレビュー → 盲点が残る |
| **R-4** | **エージェント間通信が一方通行** | コメントが非対話的 → 意図の確認・方向性合意に往復コストがかかる |
| **R-5** | **フロー停滞が手動解消依存** | 依存解除・タスク割当・スコープ超過がすべて人手を要する |
| **R-6** | **知識が次の実行に引き継がれない** | スキル選択・設計判断・失敗パターンが毎回ゼロリセット |

## グルーピング

24案を「何を変えようとしているか」という軸で7グループに整理する。

### Group 1 — 「着手前の情報品質」を上げる

**対応課題**: R-1  
**共通の問い**: *「実装に入る前に、全員が同じ理解を持てているか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **A** | `issue-quality-gate` スキル | 中 | スキーマ不変 |
| **I-1** | 仕様詰め対話フェーズ | 低 | スキーマ不変 |
| **III-4** | AC 自動テストスケルトン変換 | 中 | 新スキル |
| **II-2** | 偵察ワーカー | 中 | 新ロール |

**内部のつながり**:

```
I-1（対話で AC を確定）
  └→ A（確定した AC を品質ゲートで検証）
      └→ III-4（合格した AC をテストスケルトンに変換）
            └→ II-2（偵察ワーカーが影響範囲マップを添えて渡す）
```

上流から順に積み重ねると「ワーカーが着手するときには AC・テスト骨格・影響範囲の三点が揃っている」状態になる。

**現行 gitlab-idd との整合**:  
`needs-clarification` フロー（ワーカー → リクエスター → ワーカー）をこの Group で事前に潰す。うまく機能すると `needs-clarification` ラベル自体が不要になる。

---

### Group 2 — 「実装中の逸脱」を防ぐ

**対応課題**: R-2  
**共通の問い**: *「合意した仕様から実装がずれていないか、常に確認できるか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **B** | project-dod リファレンス | 低 | スキーマ不変 |
| **III-1** | ライブ仕様書としてのイシュー | 低 | スキーマ不変 |
| **IV-1** | タイムボックス駆動 | 中 | 新フロー |
| **IV-2** | リスク先行実装 | 中 | 新フロー |

**内部のつながり**:

```
B（何が "完成" かをプロジェクト横断で定義）
  + III-1（実装中にイシュー本文が変わっても仕様が追従する）
    + IV-2（リスクの高い部分を最初に実装して不確実性を早期に潰す）
      + IV-1（超過したらスコープを自動縮小・イシューを分割する）
```

B が「完成の外縁」、III-1 が「仕様のドリフト防止」、IV-2 が「リスク優先順序」、IV-1 が「スコープ暴走への安全弁」として機能する。

**現行 gitlab-idd との整合**:  
B は agent-reviewer と worker-role Phase 5 への参照追加のみ。III-1 は Phase 4 ループにイシュー本文更新ステップを追加するだけで実現できる。

---

### Group 3 — 「検証の深度と独立性」を高める

**対応課題**: R-2, R-3  
**共通の問い**: *「コードが AC を本当に満たしているか、バイアスなく確認できるか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **C** | AC 特化レビュー perspective | 中 | agent-reviewer 拡張 |
| **II-3** | 独立検証エージェント | 中〜高 | 新ロール + 実行環境 |
| **III-2** | 変異テスト自動実行 | 高 | 実行環境必須 |
| **III-3** | コントラファクチュアル差し戻し | 低〜中 | スキーマ不変 |

**内部のつながり**:

```
C（AC ↔ diff の対応を reviewer 視点で確認）   ← ホワイトボックス
II-3（AC だけを見てブラックボックスで動作確認）  ← ブラックボックス
  └→ III-2（テストスイートが変異を検出できるか検証）
III-3（差し戻し時に「どう壊れるか」を具体提示）  ← 差し戻し品質向上
```

C と II-3 は相補的。C が「コードが AC の意図通りか」を問い、II-3 が「ACの条件が実際に動くか」を問う。

**Group 1 との接続**:  
III-4（AC → テストスケルトン）があると II-3 が使える検証手段を最初から持てる。Group 1 が充実するほど Group 3 の精度が上がる。

---

### Group 4 — 「エージェント間対話」を双方向にする

**対応課題**: R-4  
**共通の問い**: *「一方的なコメント交換を、意味のある対話に変えられるか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **I-2** | ワーカーを提案者に | 低 | スキーマ不変 |
| **I-3** | ペアレビュー対話 | 低 | スキーマ不変 |
| **II-1** | ペアプログラミングモード | 中 | 新ロール |

**内部のつながり**:

```
I-2（着手前: ワーカーが複数アプローチ案を提示 → リクエスターが選択）
  → I-3（レビュー中: reviewer が質問 → worker が返答 → 対話で解決）
    → II-1（実装中: 2ワーカーが同時進行で相互レビュー）
```

I-2 が「方向性の合意」、I-3 が「レビューの深化」、II-1 が「実装精度の最大化」として時系列順に機能する。

**現行 gitlab-idd との整合**:  
I-2 と I-3 は SKILL.md の行動指針への追記と worker-role.md のフロー拡張だけで実現できる最低コストの改善。「最小往復」原則と一見矛盾するが、往復を「量を減らす」のではなく「質を高める」方向にシフトさせる。

---

### Group 5 — 「フローの詰まり」を自動解消する

**対応課題**: R-5  
**共通の問い**: *「人が介在しなくてもパイプラインが滑らかに流れるか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **D** | 依存解除の自動化 | 低 | スキーマ不変 |
| **II-4** | イシューのオークション制 | 高 | 大幅な新ロジック |
| **IV-3** | 並列ドラフト | 高 | 並列管理が複雑 |

**内部のつながり**:

```
D（依存が解消したら status:blocked を自動で status:open に戻す）
  ← 最小コストで最大の詰まり解消
II-4（複数ワーカーが入札し最適なワーカーを選定）
  ← Dの上位版：誰が引き受けるかも最適化
IV-3（同一イシューを並列で複数ワーカーが試作、最良を選択）
  ← 探索的問題向け、コスト大
```

D は現行フローへの小さな追加で大きな効果を得られる。II-4 と IV-3 は将来的な高度化。

---

### Group 6 — 「知識の蓄積と学習」でシステムを改善し続ける

**対応課題**: R-6  
**共通の問い**: *「今回の実行を、次の実行に役立てられているか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **E** | LTM フィードバックループ | 低 | スキーマ不変 |
| **IV-4** | プロジェクトメモリ | 中 | ltm-use 連携 |
| **IV-5** | 自動レトロスペクティブ | 中 | sprint-reviewer 連携 |

**内部のつながり**:

```
E（完了ごとに有効なスキル組み合わせを ltm-use に保存）
  ↓ skill-selector の精度が向上
IV-4（設計判断・失敗パターンを ltm-use に保存）
  ↓ 将来のワーカーが同じ過ちを繰り返さない
IV-5（定期的に蓄積データを sprint-reviewer で分析）
  ↓ プロセス設定（閾値・ラベル・フロー）を自動提案
```

**他グループへの逆流**:  
Group 6 が充実するほど Group 1〜3 の精度が自然に上がる。E の蓄積が issue-quality-gate（案 A）の判定基準を改善し、IV-4 が偵察ワーカー（案 II-2）に渡す影響範囲マップを豊かにする。

---

### Group 7 — アーキテクチャの根本を問い直す

**対応課題**: R-1〜R-6 すべて  
**共通の問い**: *「現在の構造的前提を変えたら何が変わるか？」*

| 案ID | 案名 | コスト | 制約 |
|------|------|--------|------|
| **V-1** | イシューレスモード | 高 | 大規模再設計 |
| **V-2** | 継続実行エージェント | 高 | CI/CD 連携 |
| **V-3** | ロールの流動化 | 高 | 状態管理の再設計 |

**内部のつながり**:

```
V-1（イシューを廃止 → タスクグラフ直接実行、イシューは事後記録）
V-2（プロンプトトリガーを廃止 → CI コミットが自動でフロー起動）
V-3（固定ロールを廃止 → 状態を見て自己決定）

3案は独立しているが組み合わせると「全自動・アダプティブ開発基盤」になる
```

Group 7 は現時点では実装コストと安全弁設計が課題。ただし Group 1〜6 を着実に積み上げると V-2（継続実行）への移行がより安全になる。

| G3 × G4 | ブラックボックス検証（II-3）＋ペアレビュー対話（I-3）の組み合わせが最も深い品質保証を実現 |
| G6 → G1 | LTM 蓄積が issue-quality-gate の判定精度と偵察ワーカーの情報品質を継続的に改善する |
| G2 × G3 | project-dod（B）＋ AC perspective（C）= 「完成の二重保証」（グローバル DoD + イシュー固有 AC） |
| G1〜6 → G7 | Group 1〜6 の成熟が継続実行エージェント（V-2）への移行を安全にする |

## 実装ロードマップ

### Phase 1 — フロー追記だけで実現（低コスト・高効果）

目標: 手を入れやすいフロー記述の変更で、最大のボトルネック（R-1・R-4・R-5・R-6）を削る。

| 案 | 変更場所 |
|---|---|
| **I-1** 仕様詰め対話フェーズ | worker-role.md に「着手前対話ステップ」を追加 |
| **I-2** ワーカーを提案者に | worker-role.md Phase 2 にアプローチ提案ステップを追加 |
| **B** project-dod リファレンス | `references/project-dod.md` を新規作成、worker-role.md Phase 5 から参照 |
| **III-1** ライブ仕様書 | worker-role.md Phase 4 に「変更時はイシュー本文も更新」を追記 |
| **III-3** コントラファクチュアル差し戻し | requester-review.md ステップ 4b のテンプレートを拡充 |
| **I-3** ペアレビュー対話 | non-requester-review.md と requester-review.md にコメント往復規約を追加 |
| **D** 依存解除の自動化 | worker-role.md Phase 5 に依存解除チェックステップを追加 |
| **E** LTM フィードバック | worker-role.md Phase 5 に `ltm-use` 保存ステップを追加 |

### Phase 2 — 新スキル・新 perspective の導入

目標: Group 1・3 の精度を新スキルで底上げし、「AC 品質 → 検証深度」の連鎖を実現する。

- **A** `issue-quality-gate` スキル（新規 SKILL.md）
- **C** AC 特化 perspective（`agent-reviewer` に `acceptance-criteria` perspective を追加）
- **III-4** AC 自動テストスケルトン変換（新規スキル or agent-reviewer の前処理）
- **II-2** 偵察ワーカー（worker-role.md に新サブロール定義）
- **II-3** 独立検証エージェント（新ロール定義・実行環境整備）
- **IV-4** プロジェクトメモリ（`ltm-use` 連携の強化）

### Phase 3 — プロセス最適化

目標: 実装順序・スコープ管理・レビュー対話の高度化。

- **IV-2** リスク先行実装
- **IV-1** タイムボックス駆動
- **II-1** ペアプログラミング（設計フェーズ限定から）
- **IV-5** 自動レトロスペクティブ

### Phase 4 — 根本的な再設計（長期・選択的）

目標: Phase 1〜3 の成熟を前提に、アーキテクチャの仮定を変える。

- **V-2** 継続実行エージェント
- **V-3** ロールの流動化
- **V-1** イシューレスモード
- **II-4** オークション制・**III-2** 変異テスト・**IV-3** 並列ドラフト
