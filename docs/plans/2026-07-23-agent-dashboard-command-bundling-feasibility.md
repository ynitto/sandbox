# Agent Dashboard: エージェントコマンド一括投入の現実性調査

> 日付: 2026-07-23
> 目的: agent-dashboard で設定したエージェントコマンドが tmux / CLI セッションへ正しく送られないことがある問題に対し、システムプロンプトのように「実行するコマンド群」をまとめて CLI へ渡し、エージェント側で実行させる方式が現実的かを調査する。
> 関連: [`2026-07-19-agent-dashboard-global-instructions-design.md`](2026-07-19-agent-dashboard-global-instructions-design.md) / [`2026-07-20-agent-dashboard-session-commands-design.md`](2026-07-20-agent-dashboard-session-commands-design.md) / [`../../schemas/agent-instructions.schema.json`](../../schemas/agent-instructions.schema.json) / [`../../schemas/agent-session-commands.schema.json`](../../schemas/agent-session-commands.schema.json)

## 結論

**現実味はあるが、既存の `agent-session-commands` を単に `chat` モードで増強するだけでは不足する。**

一番堅い方向性は、`agent-instructions` と同じ「有界テキストブロックとしてプロンプトへ前置する」方式を流用し、そこへ **エージェント側で実行すべき初期アクション束**を宣言することである。ただし、これは「ホスト側で必ず完了させる前準備」ではなく、**AI CLI が入力を受け付けた後に、エージェントへ実行依頼する ToDo / 指示**として扱う必要がある。

したがって、用途を次の 2 つへ分離するのが現実的である。

1. **必ず開始前に終わるべき処理**: 既存設計どおり `agent-session-commands` の `process` モードでホスト側が実行する。
2. **CLI 内部コマンド・スキル起動・作業前の読み込み指示**: 新しい「起動アクション束」をプロンプトブロックとしてまとめて送る。エージェントが実行できるが、成功保証はログ / 自己申告 / 後続検証で補う。

## 現状の契約から分かること

### agent-instructions は「テキストを確実に届ける」用途に向く

`agent-instructions` は dashboard が編集し、各エンジンが JSON を有界テキストブロックへ描画して実行エージェントのプロンプトへ前置する契約である。スキルは名前参照のみで、実体は運ばない設計になっている。

この性質は「コマンドを実行してほしい」という**依頼文**を届ける用途には向いている。既に `skills` と `tools` は強制ではなく、助言テキストとして描画される前提であるため、起動アクションも同じ思想で足せる。

### agent-session-commands は「ホストが実行するコマンド」と「チャット初手」を扱う

`agent-session-commands` はセッション開始時に 1 回だけ実行するコマンド列で、`process` はホストのシェルで完了待ち、`chat` はセッションへ最初のプロンプトとして送る契約である。設計書でも、CLI 内部のスラッシュコマンドやスキル起動はプロセス起動引数から撃てず、セッションへ送るしかないと整理されている。

一方で、`chat` は tmux / paste / 起動待ちの不安定さに影響されるため、今回の「正しく送られないことがある」問題の根本対策としては弱い。単発系ではセッション概念が無いため `chat` はスキップされる点も制約になる。

## なぜ「コマンド束を system prompt 風に送る」が有効そうか

1. **paste の回数を減らせる**  
   複数のコマンドを個別送信すると、tmux の起動待ち・バッファ貼り付け・Enter 送信の race に複数回さらされる。1 つの有界ブロックにまとめれば、送信点は 1 回になる。

2. **通常プロンプト組立経路に乗せられる**  
   `agent-instructions` と同じ前置方式に寄せると、常駐系だけでなく agent-flow のような単発系にも適用しやすい。単発系では「最初のチャット」ではなく、ワーカーのプロンプト本文に含めればよい。

3. **スキル参照との相性が良い**  
   `agent-instructions` は既に `skills` の名前参照を持つ。起動アクション束でも「このセッション開始時に skill X を読み、次のコマンドを実行する」といった宣言を自然に表現できる。

4. **安全境界を明確にできる**  
   ホスト実行ではなくエージェントへの依頼にすれば、任意シェルコマンドを GitBus や meta.json で配るリスクを避けやすい。ただし agent-flow へ伝播する場合は、実行指示の内容がリモートノードで解釈されるため、後述の scope 制御が必要になる。

## 限界とリスク

### 実行完了の保証はない

プロンプトに書いたコマンドは、ホスト側の `subprocess.run` と違って終了コードを管理面が直接取得できない。エージェントが無視する、途中で失敗する、別の解釈をする可能性がある。よって `docker compose up -d` や認証更新のような**実行完了が前提条件になる処理**には向かない。

### `system prompt` そのものを全 CLI に注入できるわけではない

現行設計では、各 CLI 経路に `--append-system-prompt` 相当や CLAUDE.md の動的読込が共通に存在しない前提で、プロンプト文字列への連結で実現している。したがって「本物の system prompt」ではなく、実装上は**通常入力の先頭に強い見出し付きブロックを前置する**形になる。

### スラッシュコマンドは CLI ごとに挙動が違う

`/skill` や `/compact` のような CLI 内部コマンドは、通常のテキストプロンプト内に混ぜるだけでは実行されない可能性がある。特に複数行ブロックの中に置いた場合、CLI が「内部コマンド」として解釈するか「ユーザー本文」として解釈するかは CLI 実装依存である。

そのため、v1 ではスラッシュコマンドを魔法のように扱わず、**エージェントに実行してほしい作業手順**として表現するのが安全である。どうしても内部コマンドを撃つ場合は、既存の `chat` モードを「1 paste 1 command」で慎重に使い、再送・確認を別途設ける。

### スキル実体は運べない

`agent-instructions` の `skills` は名前参照であり、スキル実体は node federation が配布する設計である。起動アクション束にスキルを含めても、リモートノードに SKILL.md が存在しない場合は効かない。UI は「ローカルに存在する場合のみ適用」と明示し、必要なら配布状態を別途可視化する必要がある。

## 推奨設計: `startup_actions` をテキストブロックとして描画する

### 位置づけ

`agent-instructions` に近いが、意味論は分ける。

- `agent-instructions.text`: 常時効く行動規範・共通指示。
- `startup_actions`: セッション / run の開始時に 1 回だけ、エージェントへ実行を依頼するアクション束。
- `agent-session-commands.process`: ホスト側で必ず実行し、終了コードを見たい前準備。

`agent-session-commands.chat` を置き換えるというより、**不安定な複数 chat 送信を 1 つの起動アクションブロックへ畳む**位置づけにする。

### データ例

```json
{
  "version": 1,
  "revision": 4,
  "enabled": true,
  "actions": [
    {
      "id": "load-project-rules",
      "kind": "instruction",
      "text": "作業前に README.md と docs/guides/ を確認し、プロジェクト固有の注意点を短く要約してから着手する。"
    },
    {
      "id": "use-skill",
      "kind": "skill",
      "skill": "skill-selector",
      "text": "複合タスクではこのスキルを使って必要スキルを選ぶ。"
    },
    {
      "id": "verify-command",
      "kind": "agent_shell",
      "text": "変更後は可能なら npm test を実行し、失敗時は原因を分類する。"
    }
  ],
  "max_chars": 3000,
  "scope": "local-session",
  "updated_at": "2026-07-23T00:00:00Z",
  "updated_by": "dashboard"
}
```

`agent_shell` はホストが実行するシェルではなく、**エージェントが必要に応じてツールで実行する作業指示**という名前にする。`run: "rm -rf ..."` のような生コマンド列をそのまま自動実行する印象を避けるため、UI でも「エージェントへの依頼」と表記する。

### 描画例

```markdown
<!-- agent-startup-actions rev:4 -->
## セッション開始時アクション（agent-dashboard 管理）
このブロックは開始時に 1 回だけ確認し、可能なものを実行してください。実行できない場合は理由を短く報告してください。

1. [load-project-rules] 作業前に README.md と docs/guides/ を確認し、プロジェクト固有の注意点を短く要約してから着手する。
2. [use-skill] 推奨スキル: skill-selector — 複合タスクではこのスキルを使って必要スキルを選ぶ。
3. [verify-command] 変更後は可能なら npm test を実行し、失敗時は原因を分類する。
```

## エンジン別の現実性

| エンジン | 現実性 | 推奨注入点 | 注意点 |
|---|---:|---|---|
| kiro-loop / agent-loop | 高 | ペイン作成後の最初の業務プロンプトへ前置。既存の instructions revision 管理と同型にする | 既存ペインへは遡及しない。ペイン再起動または次回プロンプトで反映 |
| agent-dashboard cowork | 高 | `withGlobalInstructions` と同じ位置で起動入力へ前置 | Windows/WSL 差異を避けるため、ホスト実行はしない |
| agent-flow | 中〜高 | run 作成時に meta.json へスナップショットし、ワーカープロンプトへ前置 | リモートノードへ伝播する場合、任意コマンド指示の安全性を scope で制御する |
| agent-project / agent-amigos | 中 | 直接注入せず、実作業を agent-flow へ渡す経路で効かせる | planner / adjudicate 等のメタ LLM には混ぜない |

## 実装するなら先に決めるべきこと

1. **伝播範囲**  
   既定は `local-session` にし、agent-flow の GitBus へ載せる場合は `propagate: true` を明示するのが安全である。`agent-instructions` は伝播が価値だが、起動アクションは実行指示に近いため、同じ既定にしない方がよい。

2. **生コマンドの扱い**  
   v1 ではホスト自動実行をしない。生の shell command は `process` モードへ誘導し、起動アクション側では自然言語の実行依頼に寄せる。

3. **成功確認の UI**  
   エージェント側実行は終了コードを取れないため、「反映済み revision」と「実行成功」は分けて表示する。必要なら、エージェントに `startup_actions` の実行結果を冒頭で箇条書き報告させる。

4. **スキル配布状態**  
   `skills` は名前参照のみなので、dashboard は inventory で見つかったものを候補表示し、見つからないスキルは警告する。

## 推奨する段階導入

1. **調査結果を設計へ反映**  
   `agent-session-commands` の `chat` 連打ではなく、`startup_actions` テキストブロック方式を設計として追加する。

2. **常駐系から実装**  
   kiro-loop / agent-loop / dashboard cowork で、最初の業務プロンプトに 1 回だけ前置する。まずは任意シェル実行なし。

3. **agent-flow へ限定伝播**  
   `scope` / `propagate` を明示した場合だけ meta.json スナップショットする。既定はローカル。

4. **実行結果報告テンプレート**  
   ブロック内に「実行できた / できなかった / スキップした」を冒頭で短く報告する指示を入れ、dashboard 側はログから拾うだけに留める。

## 判定

- **やる価値**: 高い。複数 paste の不安定さを 1 回のプロンプト前置に減らせる。
- **実装難度**: 中。`agent-instructions` のレンダラ・revision・status の流儀を再利用できる。
- **安全性**: 中。ホスト自動実行と混同しなければ許容可能。任意コマンドのリモート伝播は既定オフにすべき。
- **制約**: 終了コードの取得、CLI 内部スラッシュコマンドの確実な実行、スキル実体配布は別問題として残る。

結論として、**「システムプロンプトのようにコマンド束とスキル参照をまとめて送る」は現実的**である。ただし、それは「確実に実行されるホストコマンド」ではなく、**開始時にエージェントが読む実行依頼ブロック**として実装するのが安全で、既存の `process` 型セッションコマンドと役割分担させるのがよい。

## 追記: コマンドごとの送信戦略を選べる UI / 実装方針

前述の調査だけでは「従来の 1 コマンド 1 ペースト式を完全に捨てる」ように読めるが、実運用では両方を残すべきである。理由は、スラッシュコマンドのように **1 行ずつ CLI に打鍵した方が解釈されやすい操作**と、作業手順のように **まとめてエージェントへ依頼した方が race に強い操作**が混在するためである。

### 方針

`mode: "chat"` のコマンドごとに `strategy` を追加する。

| `strategy` | 意味 | 向いている用途 |
|---|---|---|
| `paste` | 従来どおり 1 コマンド 1 ペーストで送る | CLI 内部コマンド、1 行単位で解釈してほしい短い指示 |
| `bundle` | `bundle` 指定された chat 行を 1 つの起動アクションブロックへまとめる | スキル利用指示、作業前確認、複数ステップの自然言語 ToDo |

`process` はホスト実行であり、送信戦略の対象外である。`strategy` が無い既存設定は `paste` として扱い、後方互換を保つ。

### dashboard UI

agent-dashboard のセッション開始コマンド行では、実行方法が「エージェントに送る」のときだけ **「まとめて依頼」チェックボックス**を表示する。

- チェックなし: 従来の 1 コマンド 1 ペースト式。
- チェックあり: 同じセッション開始時にチェックされた行を 1 つの `agent-startup-actions` ブロックへ畳む。
- プレビューでは、通常送信は「エージェントに送る」、まとめ送信は「まとめて依頼」と表示し、まとめブロックに含まれる元 ID を併記する。

### 実行計画

保存される JSON は次のようになる。

```json
{
  "commands": [
    { "id": "slash-skill", "mode": "chat", "strategy": "paste", "run": "/skill skill-selector" },
    { "id": "read-docs", "mode": "chat", "strategy": "bundle", "run": "作業前に docs/ を読み、前提を要約する" },
    { "id": "use-skill", "mode": "chat", "strategy": "bundle", "run": "必要なら skill-selector を使って実行計画を立てる" }
  ]
}
```

計画時には `paste` の行は従来どおり個別 entry として残り、`bundle` の行は次のような 1 entry へ集約される。

```markdown
<!-- agent-session-command-bundle rev:4 -->
## セッション開始時アクション（agent-dashboard 管理）
次の項目は個別ペーストではなく、まとめて依頼された起動アクションです。可能なものを順に実行し、実行できない項目は理由を短く報告してください。

1. [read-docs] 作業前に docs/ を読み、前提を要約する
2. [use-skill] 必要なら skill-selector を使って実行計画を立てる
```

このため、同じ画面で「CLI に 1 行ずつ送るべきもの」と「エージェントにまとめて任せるべきもの」をコマンド単位に混在させられる。
