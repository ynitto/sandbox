# gitlab-idd 自律実行の質を上げる改善ブレスト（2026-05-19）

## 背景・課題

gitlab-idd での作業の「思慮が浅い」。エージェント（kiro-cli 上で動作）に次の問題がある。

- `ltm-use` / `wiki-use` のクエリ頻度が低い
- ユーザー指示に迎合しやすい
- 実施方法の振り返り・立ち止まりがなく、その結果やり直しが起きない
- これによりユーザーによる修正・手戻りが発生している

gitlab-idd の価値は **自律実行** と **動作環境固有の知見・経験の活用** を分散して実行することにある。
本ドキュメントはその価値を高めるための改善策のブレスト。

---

## 診断 — なぜ「思慮が浅い」状態になるか

問題は「エージェントが悪い」ことではなく、**スキルの記述構造**にある。具体的な穴が 3 つ。

### 1. LTM が「書くだけで読まない」非対称

`worker-role.md:733-752`（ステップ 5-5a）で実行知見を `gitlab-idd-execution` カテゴリに **save** しているのに、
Phase 2〜4 に **recall ステップが一つも無い**。`SKILL.md:278` の行動原則 12 も
「`ltm-use` が利用可能な場合は…参照する」という努力目標で、コマンドすら書かれていない。
→ エージェントは飛ばす。蓄積した知見が死蔵されている。

### 2. wiki-use は gitlab-idd から一度も呼ばれない

`worker-role.md` 全体に `wiki` の文字が無い。kiro-cli / winauto のような環境固有の落とし穴は
wiki に蓄積され得るのに、ワーカーは参照経路を持たない。

### 3. 立ち止まりの関門が「任意」

アプローチ提案（`worker-role.md:277-324` ステップ 2-8）は「**任意・判断による**」。
実装ループは 5 回まで惰性で回り、未解決でも提出される（`worker-role.md:569-573`）。
レトロスペクティブは 10 件ごと（ステップ 5-5b）と粒度が粗い。
異議を唱える経路・やり直す経路が制度化されていない。

### 結論

迎合・手戻りは「努力目標であって必須手順でない」構造の帰結。
改善の核は **任意 → 必須ステップ + 機械可読な痕跡 + eval.json での担保** に格上げすること。

- 最も効くのは **A（recall を必須化し save と対称にする）** と **C（approach gate と早期中断）**。

# 第2観点 — 構造から見た隠れた要因（2026-05-19 追記）

第1観点は人間が観測した症状（クエリ頻度・迎合・振り返り欠如）からの分析。
本セクションは症状を一旦カッコに入れ、設計構造そのものから隠れた要因を洗い出す。

## 症状の再解釈

観測された症状は「エージェントの弱さ」ではなく **設計の帰結**。

- **「迎合しやすい」** → 迎合ではなく **仕様リテラリズム**。行動原則 3「受け入れ条件最優先」、
  ステップ 4-0「worker は推薦結果を再解釈しない」が「考えるな、従え」を明示している。
  さらに `worker-role.md:528`「ライブ仕様書として AC を維持」と `worker-role.md:470`
  「`## 受け入れ条件` の内容は変更しない」は **正面から矛盾**しており、矛盾時の安全手は
  「触らない＝リテラルに従う」。
- **「振り返りがない」** → 振り返りが **構造的に不可能**。セッションはワンショット、
  テンポラリクローンはステップ 5-6 で削除、永続状態はイシューコメントのみ。そのコメントは
  「リクエスター向けの清書レポート」(行動原則 13) であり、正直なスクラッチパッドがない。
  比較対象の過去状態が毎回破棄されるので、振り返る材料そのものが無い。
- **「クエリ頻度が低い」** → 合理的な学習結果の可能性。ステップ 5-5a の save テンプレートは
  「どのスキルを使ったか」というプロセスメタデータしか記録せず、「何が難しかったか」は
  `{problems_if_any}`（任意）に埋もれる。中身が低信号なら参照しないのが正しい。
  頻度だけ上げると雑音が増える。

→ 「思慮が浅い」は相当部分が **仕様で指定されている**。

## 隠れた根本原因と対策

### RC1. メモリはあるが「学習ループ」がない

記録は溜まる（イシューコメント・LTM save）が、**結果が生産者に返らない**。save はするが
rate は gitlab-idd から一度も呼ばれない。差し戻し（needs-rework）は別ノード・別セッションが
拾うため、雑な仕事をしたワーカーに手戻りの結果が届かない。レトロは 10 件ごとのバッチで
個別失敗から切り離されている。accountability がノード間に拡散し、誰も自分のミスの帰結を負わない。

**対策**
- 差し戻し時に「何が誤りだったか」を元の approach / スキルにキー付けして記録し、recall した
  記憶を `rate --correction` で減点する。save→recall→rate の三角形を閉じる。
- イシューコメントを 2 層に分ける: リクエスター向け清書レポートと、
  `<!-- gitlab-idd:worker-journal -->`（次の自分 / ワーカー向けの正直な疑念ログ）。
- レトロの出力を「ドキュメント」でなく **行動変更** に縛る。

### RC2. 判断が「情報の乏しい場所」に置かれ「豊富な場所」から奪われている

最重要決定（どのアプローチ／スキル）を、最も文脈の乏しい skill-selector が下す
（イシュータイトル＋AC＋スカウトのみ・コード未読）。一方これからコードを読む最も文脈の
豊富なワーカーは「再解釈するな」(4-0 ルール 1) と禁じられる。AC もコードを見る前に
requester が凍結する。検証する agent-reviewer も同一モデル族＝相関した盲点を持つ。

**対策**
- アプローチ確定をスカウト（コード読解）の後に移す。skill-selector は候補出しに留め、
  ワーカーに「approach 確認ゲート」を与えて改訂を許可かつ義務化する。
- AC 矛盾を解消: requester は意図を所有、AC は共同ドラフト。ワーカーは根拠付きで AC 改訂を
  提案でき、差分が可視化される、と定義し直す。
- 独立検証の主役を「同一モデルのレビュー」から **実行（テストを書いて走らせる・ビルドを通す）**
  に移す。実行が現状ステップ 5-0「任意」なのを必須化する。

### RC3. 「自律」が "不確実性の解消" でなく "不確実性の迂回" として実装されている

clarification・approach 提案の「24h 無回答 → 推測で着手」は、「人が見ていない」を
「人が承認した」へ系統的に変換する。autonomy の仕組み（ブロックしない）が、そのまま
手戻りの製造ラインになっている。5 回ループ後に未解決のまま提出するステップ 4-4 も同型。

**対策**
- 自律の定義を変える: 不確実性への第一手は自分で解消する（コードを追う・実験する・調べる）。
  人への質問は最後の手段。
- 沈黙＝同意を廃止。タイムアウトは「着手」でなく「成果物を provisional とマークし
  エスカレーション」。MR に `⚠️ 未検証の仮定` フラグを立てる。
- 非収束（5 回回っても直らない）は締切でなく信号。提出でなく停止・エスカレーションが正解。

### RC4. スループット最適化で品質の対抗圧力がなく、不確実性モデルもない

全イシューが同一パイプラインを通る。requester の分割は L/XL の「サイズ」であって
「難易度・不確実性」ではない。システムは自分がどこで自信がないかを知らないので難所に
思考を多く配分できない。worker-role.md の約 4 割が協調プロトコルで、エージェントの注意と
プロンプト予算が協調税に食われる。

**対策**
- スカウト後にワーカーが自己確信度スコアを付ける。低確信 → レビュー観点増・人間
  チェックポイント必須・スコープ縮小。高確信 → 高速パス。思考を難易度で配分する。
- ノード単位で見える品質対抗指標（差し戻し率・マージ後欠陥率）を持つ。
- 協調税の削減: defer / lock 判定を `gl.py` 側に寄せ、スクリプトが「即着手可能なイシュー」を
  ワーカーに手渡す。

## 統合的な洞察

4 つの根本原因は 1 つの設計思想に収束する — **gitlab-idd は「速い従順な実行者の艦隊」として
設計されており、「考える主体」としては設計されていない**。記憶はあるが学習はなく、判断は
情報の乏しい場所に固定され、自律は迂回として実装され、品質より closure が報われる。
「思慮の浅さ」は能力の問題ではなく **役割定義の問題**。

最も効くのは RC3（沈黙＝同意の廃止＋実行を必須検証に）と RC2（判断をコード読解後に移す）。
この 2 つは現在の最優先原則「一気通貫（行動原則 2: 途中で止めない）」と正面衝突するため、
**「一気通貫」を最上位の価値から降ろす意思決定** が前提になる。これが最大の隠れた要因 —
立ち止まりが構造的に禁止されている。

# 第3観点 — ゲートをすり抜ける失敗モード（レッドチーム視点）

第1観点は「何が観測されたか」、第2観点は「構造がどう仕向けているか」。
第3観点は攻撃者の頭で考える — 「このシステムは、全ゲートを緑にしたまま、
どうやって間違ったものを出荷するか」。

## 前提の問い直し: ゲートの数 ≠ 品質

gitlab-idd には多くの品質ゲートがある（明確性チェック・agent-reviewer・project-dod・
AC チェック・レトロ）。だがゲートが価値を持つのは **(a) 実際に落ちうる** かつ
**(b) 落ちることにコストがある** 場合だけ。gitlab-idd の多くのゲートは落ちえない／
黙って no-op する。すり抜け可能なゲートは「レビュー済み・DoD 確認済み・AC 充足」という
表示で人間の警戒を下げる分、ゲートが無いより有害 — これが false assurance。

## 失敗モード

### FM1. スカウトマップ汚染（単一障害点）

ステップ 3-5 は既存の `<!-- gitlab-idd:scout-map -->` があれば再利用してスキップする。
最初のワーカーの誤ったスカウトマップが、以降の全ワーカー（差し戻し再作業者を含む）を
黙って誤誘導する。スカウトマップは一度も検証されない。分散実行が最初の一手の誤りを増幅する。

**対策**: スカウトマップに作成者と信頼度を記録。差し戻し時は破棄して再スカウト。
2 人目以降は「検証モード」で既存マップを疑う。

### FM2. `priority:high` が全安全装置のマスターキー

self-defer・assigned-lock・self-review-lock・review-defer のチェックを `priority:high` が
一括バイパスする（worker-role 2-2/2-3、SKILL.md 行動原則 6/7）。プレッシャー下の requester は
全部 high にする → 安全装置が常時オフ。high 乱用を検知・抑制する仕組みがない。

**対策**: high のバイパスは「待ちロック（defer/lock）」に限定し、レビュー・明確性チェックは
バイパスさせない。ノード単位で high 付与率を監視する。

### FM3. 自己採点される受け入れ条件

ステップ 5-3 で `- [x] {条件} → {対応}` のチェックボックスをワーカー自身が書く。
AC 充足の最終証跡が自己申告。検証する agent-reviewer も同一モデル＝相関盲点。

**対策**: チェックを実行可能な検証コマンド（テスト名・コマンド・期待結果）に紐付ける。
チェックボックスでなく検証ログ。

### FM4. 推測が世代を経て事実に洗浄される（assumption laundering）

ステップ 2-7 でワーカーが推測解釈をコメント → 24h で着手 → ステップ 3-6 で design doc に
`⚠️ 推測:` 付きで書き込む → 次のワーカーが design doc を読み、`⚠️` マーカーを見落として
確定事項として扱う。推測がイシュー本文に昇格し、世代交代で事実に化ける。

**対策**: 推測は design doc 本文に混ぜず「未検証の仮定」セクションに隔離。
解消されるまで MR に provisional フラグを残す（RC3 と接続）。

### FM5. 学習ループの心臓が黙って no-op する

ステップ 5-5a の LTM save は `SKILLS_DIR=$(...) 2>/dev/null || true` でパス解決し、
存在しなければ黙ってスキップ。5-5b のレトロ起動は `python -c "json..." || echo 0` で
どんなエラーでも 0 になり、レトロが永久に発火しないことがありうる。
学習機構の心臓部が、誰にも気づかれず停止する。

**対策**: save / retro を必須化し、失敗を黙殺せずイシューコメントに「LTM 保存失敗」と
残して観測可能にする。

### FM6. 5 回ループの上限が「逃走線」になる

ステップ 4-4「5 回を超えたら未解決のまま Phase 5 へ」。ループ後半のワーカーは、本質的修正より
agent-reviewer を黙らせる表面的変更に傾く（上限到達で解放されるため）。
上限が品質の天井でなく逃げ道になる。

**対策**: 非収束は「提出」でなく「停止・エスカレーション」。未解決指摘数を MR の可視
メトリクスにし、ゼロでなければ人間レビュー必須。

### FM7. レビュアーの「助言コメント」が宙に浮く

非リクエスターレビュアーは助言コメントを投稿するだけ（行動原則 17/18）。マージもラベル変更も
しない。その助言が読まれた・反映された保証がゼロ。`<!-- non-requester-reviewed -->` タグは
「レビュー済み」を意味するが「反映済み」は意味しない。レビューが儀式化する。

**対策**: 助言コメントに「要対応／参考」の区別を持たせ、要対応はリクエスターレビューの
チェック項目に流し込む。

### FM8. 並列ワーカーの「意味的」コンフリクト

「1 イシュー = 1 ブランチ = 1 MR」でテキスト衝突は分離されるが、複数ブランチが同じ統合
ブランチを狙うとき、片方が関数シグネチャを変え他方がその関数を呼ぶ、といった意味的衝突は
検出されない。各 MR は単独では緑、統合ブランチで壊れる。

**対策**: 統合ブランチ向け MR は「統合ブランチ最新 + 他のオープン MR」を合成した状態で
テストする。または統合段階の最終 MR で結合テストを必須化する。

## 統合的な洞察

gitlab-idd の品質保証は「儀式」になっており、分散実行がそれを増幅する。ゲートは数だけあって
落ちず、安全装置にはマスターキー（`priority:high`）があり、学習機構は黙って停止し、推測は
事実に洗浄される。分散（匿名の複数ノード・スカウトマップ共有・統合ブランチ）が、検証なき
1 点の誤りを艦隊全体に伝播させる。

第3観点の処方は **「すり抜け不能なゲートを少数だけ持つ」** — 実行（テスト）と人間の taste
チェックという、本当に落ちうる 2 つに集約し、形だけのゲートは捨てる。分散には「継ぎ目での
検証」が要る。

# 具体提案の深掘り（2026-05-19 追記）

第1〜3観点で挙げた課題に対する、3 つの具体提案の詳細。

## P1. ワーカー・レビュアーの複数ターン義務化（ペアプロ的アプローチ）

**接続する課題**: FM3（自己採点）/ FM7（レビューの儀式化）/ RC2（独立検証の欠如）/
第1観点 B-4（迎合メトリクス）

**現状**: 実装は Phase 4 でワーカー単独。レビューは review-ready 後に 1 パス。
worker↔agent-reviewer の 5 ループは同一モデルの自己対話に近い。レビューはリレー
（完成品の手渡し）であってペアプロ（並走）ではない。

**提案**

- レビュー関与点を「結果レビュー 1 回」から **最低 2 回（設計時 + 結果時）** に増やし義務化する。
- **設計レビュー（早期・必須）**: Phase 3 の design doc 作成後、Phase 4 実装前に新フェーズ
  「Phase 3.5 設計レビュー」を挿入。ワーカー（driver）が approach を提示し、
  レビュアー（navigator）が具体的・反証可能な懸念を最低 1 つ返す。ワーカーは各懸念を
  解消してから実装に入る。
- **複数ターンの実質性ゲート**: 1 往復「LGTM」を禁止。navigator のターンは「X は Y のため
  壊れる」のような具体的予測か、なければ「何を確認し、なぜ問題ないか」を明示する。
  空の同意を構造的に許さない。
- **driver と navigator は別ノードを推奨**: 真の独立視点 + 環境多様性。ノード A のワーカー、
  ノード B のナビゲータ。両ノード ID を記録。これが「多視点」と「分散」「環境固有知見」を
  同時に満たし、本スキルの価値命題に直結する。
- **確信度ゲートで深さを変える**: 高確信・軽微なイシュー → 非同期 1 ターンの軽量レビュー。
  低確信・高リスク → 別ノードとのフル複数ターンペア。RC4 の自己確信度スコアと連動。

**設計の勘所・トレードオフ**

- 早期ほど安い: 設計段階の指摘修正はコード書き直しより遥かに安い。だから設計レビューを
  特に必須化する。
- 儀式化リスク: 「N コメント投稿」でなく「実質的な反証提示」を要件にする。count でなく
  substance。FM7 と同じ轍を踏まない。
- コスト: ターン増 = レイテンシ・トークン増。確信度ゲートで全件には課さない。
- council-system との棲み分け: ペアプロは 2 者の軽量版、council は多者の重量版。
  デフォルトをペアプロにする。

**実装イメージ**: 中間ラベル `status:design-review` を新設、または design doc コメントに
`<!-- gitlab-idd:design-review-requested -->` / `<!-- gitlab-idd:design-challenge:{node} -->`
のマーカー対で往復を強制する。

## P2. DesignDoc 改訂は取り消し線で証跡を残す

**接続する課題**: FM4（assumption laundering）/ RC1（学習ループ欠如）/
第2観点「振り返りが構造的に不可能」

**現状**: ステップ 3-6 / 4-1 で design doc を `update-issue --body-file` で上書きする。
GitLab の編集履歴は残るが UI に埋もれ、エージェントの読み取りパス
（`get-issue --get description`）には現れない。改訂の意図（どの記述がどの訂正か）も失われる。

**提案**

- design doc / AC を改訂するとき既存記述を削除せず `~~取り消し線~~` で残し、直後に改訂後を
  追記する。改訂理由を 1 行添える。
  - 例: `~~⚠️ 推測: X モジュールへの変更~~ → **改訂(コード確認)**: 実体は Y モジュール。`
- doc が肥大化したら、古い取り消し線部分を末尾の `<details>改訂履歴</details>` に
  折りたたみ、ライブ部分の可読性を保つ。履歴は保全、表示は折りたたみ。
- 「未検証の仮定」セクション（FM4 対策）と接続: 仮定が検証／反証されたら取り消し線にして
  「確定／却下」へ移す。仮定の解消過程が見える。

**なぜ価値があるか**

- **振り返りの材料を生む**: 次のワーカーが「X と仮定 → Y に訂正」という軌跡を読める。
  これが第2観点で「構造的に不可能」とした reflection の材料そのもの。
- **assumption laundering を阻止**: 訂正された `⚠️ 推測` が `~~推測~~ → 確定` として
  可視化され、推測が事実に化けない（FM4 直撃）。
- **レビュアーへの焦点信号**: 「この design は実装中 4 回変わった」= 高 churn =
  高不確実性 → レビュアーが重点的に見るべき箇所が分かる。
- **高信号な LTM へ**: クローズ時、取り消し線の軌跡を 5-5a の LTM save に流す。
  「スキル X を使った」より「X と仮定して外し Y が正解だった」の方が遥かに高信号。
  RC1 / 第1観点の低信号 LTM 問題を直接改善する。

**なぜ GitLab 編集履歴では不十分か**: (a) 編集履歴はエージェントの既定読み取りパスに
出ない、(b) どの文がどの訂正かという意味的意図を持たない。インライン取り消し線は
読み取りパス上にあり意図を運ぶ。

**トレードオフ・実装**: 規律が要る（削除でなく取り消し線を忘れない）。`gl.py` に
「取り消し線追記」変換ヘルパーを置き、上書きを構造的に防ぐ。または lint / eval.json で
チェックする。

## P3. タスク分解の促進（エージェント作業の単純化）

**接続する課題**: RC4（不確実性モデルの欠如）/ FM8（意味的コンフリクト）/
第1観点 D（環境固有知見）

**現状**: 行動原則 11 — requester が質問なしに自動分解、L/XL をファイル数・行数で見積もり、
L 以上（6-10 ファイル / 201-500 行）を再分割する。

**提案の核**: これは上記提案群が是正的なのに対し予防的な戦略。各タスクが小さく一意なら、
浅い思慮が損害を生む余地そのものが減る。ただし分解にはコストがあり、最適点がある。

**深掘り**

- **分解軸を「サイズ」から「不確実性・凝集性」へ**: 行数は粗い proxy。分解は「単一の明確な
  AC を持つ／一つの凝集した変更／単独でテスト可能／低不確実性」になるまで進める。
- **垂直スライス**: レイヤー（モデル群 → コントローラ群…）でなく、各タスクが単独で
  end-to-end テスト可能な垂直スライスに切る。水平分割は隠れ依存を生み FM8（意味的
  コンフリクト）の温床。垂直スライスはタスク間結合を下げる。
- **不確実性軸での分解 — spike イシュー**: リスクの高い領域は「調査タスク（spike）+
  実装タスク」に割る。spike の成果物は知識（推奨・wiki ページ・LTM エントリ）でありコード
  ではない。spike が不確実性を安く解消し、後続の実装タスクは高確信になる。これは本スキルの
  価値命題「環境固有知見の活用」に直結 — spike が後続タスクの消費する知見を生産する。
- **決定面の削減**: コード量でなく「タスクあたりの判断点」を減らす。approach の分岐が残る
  タスクは、その分岐を切り出す（spike 化）かイシュー内で解消する。判断点ゼロのタスクが理想。
- **ワーカーにも分割権限を**: 分解は requester 専権（原則 11）だが、コードを最も読むのは
  ワーカー（RC2）。ワーカーが「まだ大きい／隠れたサブタスクがある」と判断したら分割・
  派生イシュー化できる「split 要求」経路を設ける。
- **自己完結性**: 各イシューは兄弟イシューを読まずに完遂できる文脈を持つ。親 epic イシュー
  （`sub_issue_write` の sub-issue 機構が利用可能）が共有文脈を保持し、サブイシューがリンクする。

**トレードオフ（過剰分解の害）**

- **協調税の爆発**: イシュー増 = 依存エッジ増・統合ブランチマージ増・defer/lock オーバー
  ヘッド増・意味的コンフリクト面（FM8）増。分解は per-task 単純さと inter-task 協調
  複雑さのトレードオフ。
- **文脈喪失**: 小さすぎるタスクは「なぜ」を失う。親 epic で補う。
- **ハンドオフ増 = 情報損失増**（lossy telephone）。

**最適点**: 「独立して意味を持ちテスト可能な範囲で可能な限り小さく」。それを超えると
協調コストが支配する。不確実な領域は細かく（spike 化）、確実な領域は粗く分解する。
