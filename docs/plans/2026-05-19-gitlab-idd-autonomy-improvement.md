# gitlab-idd 自律実行の質を上げる改善ブレスト（2026-05-19）

## 背景・課題

gitlab-idd での作業の「思慮が浅い」。エージェント（kiro-cli 上で動作）に次の問題がある。

- `ltm-use` / `wiki-use` のクエリ頻度が低い
- ユーザー指示に迎合しやすい
- 実施方法の振り返り・立ち止まりがなく、その結果やり直しが起きない
- これによりユーザーによる修正・手戻りが発生している

gitlab-idd の価値は **自律実行** と **動作環境固有の知見・経験の活用** を分散して実行することにある。
本ドキュメントはその価値を高めるための改善策のブレスト。

---

## 診断 — なぜ「思慮が浅い」状態になるか

問題は「エージェントが悪い」ことではなく、**スキルの記述構造**にある。具体的な穴が 3 つ。

### 1. LTM が「書くだけで読まない」非対称

`worker-role.md:733-752`（ステップ 5-5a）で実行知見を `gitlab-idd-execution` カテゴリに **save** しているのに、
Phase 2〜4 に **recall ステップが一つも無い**。`SKILL.md:278` の行動原則 12 も
「`ltm-use` が利用可能な場合は…参照する」という努力目標で、コマンドすら書かれていない。
→ エージェントは飛ばす。蓄積した知見が死蔵されている。

### 2. wiki-use は gitlab-idd から一度も呼ばれない

`worker-role.md` 全体に `wiki` の文字が無い。kiro-cli / winauto のような環境固有の落とし穴は
wiki に蓄積され得るのに、ワーカーは参照経路を持たない。

### 3. 立ち止まりの関門が「任意」

アプローチ提案（`worker-role.md:277-324` ステップ 2-8）は「**任意・判断による**」。
実装ループは 5 回まで惰性で回り、未解決でも提出される（`worker-role.md:569-573`）。
レトロスペクティブは 10 件ごと（ステップ 5-5b）と粒度が粗い。
異議を唱える経路・やり直す経路が制度化されていない。

### 結論

迎合・手戻りは「努力目標であって必須手順でない」構造の帰結。
改善の核は **任意 → 必須ステップ + 機械可読な痕跡 + eval.json での担保** に格上げすること。

---

## 改善案

### A. ltm-use / wiki-use のクエリ頻度を上げる（参照の構造化）

| # | 施策 | 内容 | コスト |
|---|------|------|--------|
| A-1 | 着手前 recall を必須ステップ化 | Phase 2 に「ステップ 2-6.5: 過去知見の想起」を新設。`recall_memory.py "{技術スタック} {キーワード}" --context "gitlab-idd: {イシュータイトル}"` を明示コマンドで置き、save と対称にする | 低 |
| A-2 | スカウトに wiki query を組み込む | ステップ 3-5 で変更対象ファイル・使用技術・対象ツール（kiro-cli / winauto 等）を `wiki_query.py search` で検索し、環境固有の注意点をスカウトマップの「リスク箇所」に転記する | 低 |
| A-3 | 差し戻し時は必ず recall | `status:needs-rework` 再着手時に「同種の差し戻しが過去にあったか」を recall し、同じ轍を踏まない | 低 |
| A-4 | skill-selector への入力を構造化 | 行動原則 12 の曖昧な「参照する」をやめ、recall 結果を skill-selector の入力フィールドとして渡す | 中 |
| A-5 | rate のループを閉じる | レビュアーが approve / reject した時、ワーカーが recall した `mem-id` に `rate_memory.py --good / --correction` を自動適用する。現状は save するだけで「役立ったか」が記憶に返らない | 中 |

### B. ユーザー指示への迎合を減らす

| # | 施策 | 内容 | コスト |
|---|------|------|--------|
| B-1 | 着手前「要件の妥当性チェック」を独立ステップ化 | 受け入れ条件が技術的に矛盾 / 不可能 / オーバースペックでないかを評価し、問題があれば `needs-clarification` ではなく「異議コメント + 代替案」を投稿する | 中 |
| B-2 | アプローチ提案を「任意」→「判定基準で必須」に | LLM の主観判定をやめチェックリスト化（非互換変更／新規依存追加／スキーマ変更／セキュリティ境界に触れる、のいずれか true なら提案必須） | 低 |
| B-3 | agent-reviewer に「指示そのものへの批判」観点を追加 | 現状は実装が AC を満たすかしか見ない。AC 自体の妥当性を見る perspective（`requirement-critique`）を追加する | 中 |
| B-4 | 迎合メトリクス | 「反論ゼロのまま approve」が続くノードを sprint-reviewer が検出して警告する | 中 |

### C. 立ち止まり・振り返り（やり直しの制度化）

| # | 施策 | 内容 | コスト |
|---|------|------|--------|
| C-1 | 実装前の軽量プレモータム | `worker-role.md:513` は `council_hint` がある時だけ合議。これを「初回は必ず軽量 self-critique（この approach で AC を満たせるか・既知の落とし穴）」に変更する | 低 |
| C-2 | 実装ループの早期中断条件 | 5 回惰性で回す代わりに、2 回目で「同じ指摘の反復」「diff の発散」を検出したら修正続行を禁止しアプローチ変更を強制する | 中 |
| C-3 | レトロを問題直後に | 10 件ごとに加え「差し戻し 2 回以上」「`needs-clarification` 経由」のイシュー完了時にも sprint-reviewer を起動する | 低 |
| C-4 | 検討の痕跡を残す | `<!-- gitlab-idd:reflection -->` コメントに「却下した別案と理由」を記録し、次のワーカー / レビュアーが再利用できるようにする | 低 |

### D. 環境固有知見の分散活用（gitlab-idd の価値の核）

| # | 施策 | 内容 | コスト |
|---|------|------|--------|
| D-1 | ltm-use に node / environment スコープを追加 | 現状 `home`（個人横断）と `shared`（チーム）のみ。「このノードは WSL+winauto 対応」「kiro-cli の `--trust-all-tools` の癖」など環境単位の知見を貯める層が無い | 高 |
| D-2 | 実行知見に環境タグを付与 | `gitlab-idd-execution` の記憶に `node_id` / OS / 利用可能ツールを付け、recall 時に「同一環境で成功した知見」を優先ランキングする | 中 |
| D-3 | Kiro steering との双方向化 | `sync_kiro_memory.py` は import 一方向。gitlab-idd 実行で得た環境知見を `~/.kiro/steering/` に書き戻し、次回 kiro-cli セッションが常時参照できるようにする | 中 |
| D-4 | イシューに「実行環境要件」セクション | winauto 必須イシューは対応ノードだけが拾えるよう、`assignee:any` の取得フィルタに環境ケイパビリティを足し、分散実行の精度を上げる | 中 |

### E. 仕組みとして担保する（観測・強制）

| # | 施策 | 内容 | コスト |
|---|------|------|--------|
| E-1 | knowledge 使用ログを機械可読に | スカウトマップに `<!-- gitlab-idd:knowledge-used: recall=N wiki=N ids=mem-... -->` を埋め、レビュアー / sprint-reviewer が「recall していない」を指摘可能にする | 低 |
| E-2 | eval.json にチェック項目追加 | 「着手前 recall を実行したか」「wiki query を実行したか」をスキルテストに入れて頻度を担保する | 低 |
| E-3 | sprint-reviewer に KPI | 差し戻し率・clarification 率・recall 実行率・アプローチ変更回数を集計し、低 recall ノードを警告する | 中 |

## 優先度の所感

- 最も効くのは **A（recall を必須化し save と対称にする）** と **C（approach gate と早期中断）**。
  前者は「死蔵された知見」を即解消でき、後者は手戻りの根本原因に効く。いずれも実装コストは低い。
- **D** は中長期的に gitlab-idd 固有の価値（環境固有知見の分散活用）になるが、
  ltm-use 側のスコープ拡張を伴うため工数が大きい。
- **E** は A〜D の効果を観測・強制するための土台。A・C と同時に最小限を入れると定着しやすい。
