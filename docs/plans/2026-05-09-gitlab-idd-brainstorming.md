# gitlab-idd 改善ブレスト（2026-05-09）

制約なし（スキーマ変更可・新スキル可・大規模再設計可）でのアイデア出し。

---

## I. ワークフローの設計思想を問い直す

### I-1. イシューを「仕様書」ではなく「会話の起点」にする

**問題**: 今の gitlab-idd はイシューに最初から受け入れ条件を書かせる前提。実際には要求が曖昧なまま動き始めることが多い。

**アイデア**: ワーカーが着手する前に「仕様を詰める対話フェーズ」を設ける。リクエスターとワーカーがイシューコメントで対話し、双方が合意した AC が確定してから実装に入る。

```
Requester → 粗いイシューを投稿（"ざっくりした依頼" でOK）
Worker    → イシューを読んで仕様詰め質問をコメント投稿
Requester → 回答して AC を確定
Worker    → 確定 AC で実装開始
```

**効果**: `needs-clarification` の発生を0に近づける。往復コストを構造化して最小化。

**実装コスト**: 低（フロー記述の追記のみ）

---

### I-2. ワーカーを「提案者」にする

**問題**: 今のワーカーは「イシューを読んで実装する」受け身なロール。方向性の不一致が手戻りの根本原因になっている。

**アイデア**: ワーカーが実装前に「複数の実装アプローチ案（トレードオフ付き）」をコメントに投稿し、リクエスターが選んでから着手する。

```
Worker → Approach A: シンプル実装（速い・拡張性低）
         Approach B: 抽象レイヤー導入（遅い・拡張性高）
         推奨: B（長期コスト考慮）
Requester → A で行ってください
Worker    → 実装開始
```

**効果**: 着手前に方向性を合意。リクエスターが実装方針を意識的に選択できる。

**実装コスト**: 低（worker-role.md にアプローチ提案ステップを追加）

---

### I-3. レビューを「非同期評価」から「ペアレビュー対話」に変える

**問題**: 今のレビューはコメントの一方通行。ワーカーはレビュー結果を受け取って修正するだけ。

**アイデア**: レビュアーとワーカーが同一イシューのコメント上で対話ラウンドを回す。

```
Reviewer → "この箇所はなぜこうした？" コメント
Worker   → "○○の制約があったから。代替案は△△だが XX が問題" 返信
Reviewer → "なるほど、△△ + YY でどうか"
Worker   → "試した。LGTM" → status:review-ready に戻す
```

**効果**: レビューの深度が上がり、形式的な差し戻し→修正ループが減る。

**実装コスト**: 低（コメント規約の追加）

---

## II. マルチエージェントの新しい使い方

### II-1. ワーカーの「ペアプログラミングモード」

**アイデア**: 1つのイシューを2つのワーカーエージェントが同時に担当。一方が実装し、もう一方がリアルタイムでレビュー・提案する。最終的に合意した実装だけをコミット。

**スモールスタート案**: 実装フェーズ全体ではなく「設計フェーズだけペア」という形。

**効果**: 実装精度が上がる。ただし競合管理が複雑になる。

**実装コスト**: 中（競合管理ロジックが必要）

---

### II-2. 「偵察ワーカー」の導入

**アイデア**: 実装に着手する前に、軽量な調査専用エージェントがリポジトリを探索して「影響範囲マップ」を作りイシューコメントに貼る。本ワーカーはそれを読んで実装に集中する。

```
Scout Worker → コードベース探索 → 影響ファイル一覧・依存グラフ・リスク箇所をコメント投稿
Main Worker  → Scout レポートを読んで実装
```

**効果**: ワーカーの探索フェーズを高速化・標準化。複数ワーカーが同じ調査を重複してやることを防ぐ。

**実装コスト**: 中（新ロール定義。SKILL.md への追記で対応可能）

---

### II-3. 「独立検証エージェント」の導入

**アイデア**: ワーカーが実装したコードを、ワーカーとは独立した別エージェントが**受け入れ条件だけを見てブラックボックステスト**で検証する。実装の詳細を知らない状態で「仕様通りに動くか」だけを確認する。

```
Worker   → 実装・PR 作成
Verifier（独立）→ AC だけを読んで動作確認コマンドを作成・実行
Verifier → 全条件 PASS → status:review-ready 更新
Verifier → 一部 FAIL  → 差し戻しコメント
```

**効果**: ホワイトボックスレビューとブラックボックス検証を分離。「実装者のバイアス入りレビュー」を構造的に排除できる。

**実装コスト**: 中〜高（検証インフラ・実行環境が必要）

---

### II-4. イシューの「オークション制」

**アイデア**: 複数ワーカーが同一イシューに入札（コメントで着手宣言）し、スキルセット・負荷・過去実績を見て最適なワーカーが自動選定される。

**効果**: 現在の「先着順アサイン」より品質と効率のバランスが良くなる。

**実装コスト**: 高（調停ロジックが複雑）

---

## III. 品質保証の新しいアプローチ

### III-1. 「ライブ仕様書」としてのイシュー

**アイデア**: イシュー本文を「仕様書」として扱い、ワーカーがコードを変更するたびにイシュー本文も更新する。イシューが常に最新の設計意図を反映した状態を保つ。

**効果**: ドリフト防止。ドキュメントとコードの乖離が自然に解消される。

**実装コスト**: 低（worker-role.md に「実装変更時にイシュー本文を更新する」ステップを追加）

---

### III-2. 「変異テスト」を自動生成・実行する

**アイデア**: ワーカーの実装に対して、自動的に変異テスト（コードに意図的なバグを入れて既存テストが落ちることを確認）を実行。テストスイートの有効性を検証してからレビューに回す。

**効果**: 「テストが存在する」だけでなく「テストが機能している」を保証できる。

**実装コスト**: 高（実行環境・変異生成ロジックが必要）

---

### III-3. 「コントラファクチュアル差し戻し」

**アイデア**: レビュー時に「このコードが仕様を満たさない場合のシナリオ」をシミュレートし、どのケースで壊れるかを具体的に提示してから差し戻す。「なぜダメか」ではなく「どういうときにどう壊れるか」を伝える。

**効果**: 差し戻しの情報量が上がり、ワーカーの修正精度が向上する。

**実装コスト**: 低〜中（agent-reviewer の差し戻しコメントテンプレートに追記）

---

### III-4. 受け入れ条件の自動テストスケルトン変換

**アイデア**: リクエスターが書いた AC（日本語の箇条書き）を、ワーカー着手前に自動的に実行可能なテストスケルトンに変換する。

```
AC: "ユーザーが無効なメールアドレスを入力した場合に400エラーが返る"
  ↓ 自動変換
def test_invalid_email_returns_400():
    response = client.post("/users", json={"email": "invalid"})
    assert response.status_code == 400
```

**効果**: AC の曖昧さが即座に顕在化する。「テスト化できない AC は AC じゃない」を強制できる。テストスケルトンがワーカーの実装ガイドにもなる。

**実装コスト**: 中（LLM の変換精度に依存。新スキルとして実装）

---

## IV. プロセス全体の再設計

### IV-1. 「タイムボックス駆動」の導入

**アイデア**: 各イシューに実装時間の上限（タイムボックス）を設定。超過したら自動的にスコープを縮小して「done so far」としてコミットし、残りを新イシューに分割する。

**効果**: XL イシューの「永遠に終わらない実装」問題を防ぐ。

**実装コスト**: 中（タイムボックス管理ロジックとイシュー分割の自動化）

---

### IV-2. 「リスク先行実装」

**アイデア**: 依存関係とは別に「技術的リスク」でイシューを並び替える。最もリスクの高い実装を最初に試み、失敗したらプロジェクト全体の計画を早期に変える。

**効果**: 後半での大規模な手戻りを防ぐ。`risk-driven-development` スキルとの連携が有効。

**実装コスト**: 中（リスク評価ロジックを requester-post に追加）

---

### IV-3. 「並列ドラフト」

**アイデア**: 同一イシューに対して複数のワーカーが独立して実装ドラフトを作り、レビュアーが最良のものを選ぶか合成する。

**効果**: 探索的な問題（最良解が不明瞭）に対して有効。

**実装コスト**: 高（コスト・並列管理が重い）

---

### IV-4. 「プロジェクトメモリ」としての活用

**アイデア**: 完了したイシューからコードベースの「暗黙知」（なぜこの設計にしたか、何を試みて失敗したか）を抽出し、以降のイシューに自動的に渡す。

```
過去イシュー #10: "JWT は HS256 ではなく RS256 を選んだ（マルチサービス対応のため）"
  ↓
将来のワーカーがトークン関連のイシューを拾うと、このコンテキストが自動的に注入される
```

**効果**: 組織の意思決定履歴がコード実装に生きる。同じ過ちを繰り返さない。

**実装コスト**: 中（`ltm-use` との連携で実現可能）

---

### IV-5. スプリント単位の自動レトロスペクティブ

**アイデア**: 一定期間（例: 1週間）または一定数（例: 10イシュー）完了したタイミングで、`sprint-reviewer` を自動起動してプロセス改善点を抽出し、gitlab-idd の設定（閾値・ラベル・フロー）を自動提案する。

**効果**: プロセス自体が継続的に改善される。

**実装コスト**: 中（トリガー条件の設定と sprint-reviewer との連携）

---

## V. 根本的な再設計案（大胆）

### V-1. 「イシューレス」モード

**アイデア**: イシューを経由せず、自然言語の依頼を直接タスクグラフに変換してエージェントに配布する。GitLab イシューは成果の記録（事後生成）として使う。

**効果**: セットアップコストがゼロになる。ただし追跡可能性が低下する。

**実装コスト**: 高（大規模再設計）

---

### V-2. 「継続実行エージェント」

**アイデア**: プロンプトトリガーではなく、CI/CD パイプラインに組み込んでコードのコミットをトリガーに自動でイシュー生成・実行・レビューを回す。

**効果**: 完全自動化。ただし暴走リスクと人間の介在ポイントの設計が重要。

**実装コスト**: 高（CI/CD 連携・安全弁の設計が必要）

---

### V-3. 「ロールの流動化」

**アイデア**: Requester/Worker/Reviewer の固定ロールをなくし、各エージェントがイシューの状態と自分のコンテキストを見て「今どのロールで動くか」を自己決定する。

**効果**: 人数・状況に応じた柔軟な対応が可能。ただし責任の所在が曖昧になる。

**実装コスト**: 高（ロール推定ロジックと状態管理の再設計）

## P1 案の概要まとめ

最もインパクトが高く実現可能性も高い3案：

1. **I-1 仕様詰め対話フェーズ** — `needs-clarification` の根本的撲滅。フロー追記のみで実現。
2. **I-2 ワーカーを提案者に** — 方向性不一致による手戻りの構造的解消。フロー追記のみで実現。
3. **II-3 独立検証エージェント** — 実装者バイアスを排除したブラックボックス検証。新ロール追加が必要だが効果は大きい。
