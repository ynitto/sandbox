# Agent Dashboard タスク失敗診断エージェント設計

## 背景

要対応タブでは、検証コマンド、終了コード、生ログを確認できるが、ユーザー自身が原因を読み解き、
成果物・検査設定・実行環境のどこを直すべきか判断する必要がある。既知パターンは機械的に要約
できる一方、複数工程や環境依存の失敗は自由文だけでは十分に診断できない。

既存のAI相談は、現在画面の構造化コンテキストを読み取り専用エージェントへ渡し、追加質問できる。
この契約を維持したまま、失敗タスクに特化した入口とプロンプトを追加する。

## 目的

- 失敗した要対応項目から1操作で診断を開始できるようにする。
- 原因候補だけでなく、対処対象、確認手順、修正候補、再実行方法を提示する。
- 診断後に同じ画面で追加質問できるようにする。
- 診断エージェントは読み取り専用とし、設定変更や再実行を自動で行わない。

## 検討した案

## 採用設計

### UI

失敗情報を持つ未回答の要対応カードに「AIで失敗を診断」ボタンを表示する。押下すると既存Doctor
ダイアログを失敗診断モードで開き、確認操作を挟まず自動的に診断を開始する。

ダイアログのタイトルと状態表示は通常相談と区別し、「タスク失敗を診断」とする。初回診断後は
既存の補足入力欄を使って追加質問できる。通常の共通ヘッダーから開くAI相談は従来どおり自動実行
せず、用途を混同しない。

### コンテキスト

失敗診断では、選択中の要対応項目について次を構造化して渡す。

- タスクID、タイトル、状態、失敗工程
- 実行コマンド、作業ディレクトリ、終了コード
- 機械診断の分類、要約、対処案
- 要対応票の原文
- 関連runのstdout、stderr、failureReason、各工程の出力
- 検証設定、変更ファイル、MR・検収情報

既存の `buildDoctorContext` と関連run読込を再利用し、`mode: failure-diagnosis` を追加する。IPCは
既存の読み取り専用Doctor経路を維持する。

### 回答契約

失敗診断モードのプロンプトは、次の順序で回答するよう要求する。

1. 結論
2. 根本原因候補と確度
3. 対処対象（成果物／検査設定／実行環境／情報不足）
4. 確認手順
5. 修正候補
6. 再実行方法
7. 判断に不足している情報

ログに根拠がない断定は禁止し、推測には確度を付ける。破壊的操作、設定変更、再実行は提案に
留める。

### 状態とエラー処理

- 診断開始時はボタンと入力を無効化し、二重実行を防ぐ。
- 関連runが無い場合も、要対応票と機械診断だけで診断を続行する。
- CLI未導入、認証切れ、タイムアウトは既存Doctorと同様にダイアログ内へ表示する。
- 失敗時は診断対象と追加入力を保持し、再試行できるようにする。
- 対象タスクを切り替えた場合、古い診断結果を新しいタスクへ引き継がない。

## テスト

- 失敗情報のある要対応カードだけに診断ボタンが表示される。
- 押下すると失敗診断モードでDoctorが自動実行される。
- 要対応票、機械診断、関連run、生ログがコンテキストへ含まれる。
- 通常相談と失敗診断でプロンプト契約が分離される。
- 診断後の追加質問でも同じタスクコンテキストが維持される。
- 読み取り専用CLI設定が維持される。
- 関連runなし、CLI失敗、連打、タスク切替を安全に扱う。

## 既存スキルとの関係

- `systematic-debugging`: 回答契約の原因候補、根拠、確認手順の構造に活用する。
- `agent-assist` 相当の既存実装: Doctorの読み取り専用IPC、CLI選択、エラー処理を再利用する。
- `webapp-testing`: 自動開始、追加質問、失敗時の状態保持をUI統合テストで確認できる。
