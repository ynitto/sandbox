# 実行手法スキル拡張案（2026-04-04）

## 背景

既存スキル群には、`test-driven-development`、`scrum-master`、`skill-mentor`、`systematic-debugging`、`self-checking` のように、
**ドメイン知識そのものではなく、タスクの進め方・判断ゲート・反復手順を提供するスキル** が存在する。

一方で、この系統のスキルはまだ数が限られており、特に以下の領域に空白がある。

- プロジェクト全体のテスト戦略
- 本番障害対応の進行手順
- 段階的な技術移行・破壊的変更の管理
- 大規模リファクタリングの進め方
- 異常系・回復戦略を起点とした設計

本ドキュメントでは、これまでの調査と検討で出た案を **実行手法スキル** として整理し、優先順位付きでまとめる。

---

## 1. 実行手法スキルとは何か

本ドキュメントでいう実行手法スキルは、以下の性質を持つものを指す。

1. 駆動源が明確である
   - 例: テスト、リスク、契約、障害、仮説、ロールバック
2. 手順がフェーズまたはループとして定義できる
3. 各フェーズにゲート条件または終了条件がある
4. 失敗時のフォールバックや相談条件を定義できる
5. 既存の実装スキル・レビュースキルに委譲しやすい

この観点では、既存スキルの中核は以下の 5 つである。

| スキル | 役割 | 実行手法としての特徴 |
|---|---|---|
| `test-driven-development` | Red-Green-Refactor の実行 | 反復ループ、ゲート、カバレッジ分析を持つ |
| `scrum-master` | 複数スプリントのオーケストレーション | 7 フェーズ、承認ゲート、委譲ルールを持つ |
| `skill-mentor` | 単発タスクの委譲実行 | 振り分け、確認、計画、実行、レビューを持つ |
| `systematic-debugging` | 根本原因特定型デバッグ | 仮説駆動、計装、検証、修正の順序を持つ |
| `self-checking` | 成果物の反復評価 | ルーブリック評価と改善ループを持つ |

## 2. 明確なギャップ

### 2.1 テスト戦略の上位レイヤーがない

`test-driven-development` は個別機能の実装サイクルには強いが、
プロジェクト全体で Unit / Integration / E2E / 非機能テストをどう配分するかを決める手法がない。

### 2.2 本番障害対応の手順がない

`systematic-debugging` は開発中の問題解決には有効だが、
重大度判定、暫定復旧、ステークホルダー連絡、ポストモーテムまで含む本番運用フローは未整備。

### 2.3 大規模変更の段階導入を扱う手法が弱い

技術移行、破壊的変更、段階的リファクタリングは、
単体の実装スキルではなく順序・互換性・ロールバック計画が本質になるが、その層のスキルが不足している。

### 2.4 異常系・回復戦略を先に詰める手法がない

正常系中心で設計が進みやすく、失敗モード・検知方法・回復経路を先に定義するワークフローが不足している。

### 2.5 ドメイン知識を構造化する前段手法が薄い

`domain-modeler` は設計段階で強いが、曖昧な業務知識やユーザーの頭の中にある判断ルールを整理してモデルに落とす前段の手法が弱い。

## 3. 優先度高の新規スキル案

### 3.1 `test-strategy-planner`

**目的:**
プロジェクト全体のテスト戦略を、テストピラミッド、品質目標、CI 統合まで含めて設計する。

**既存との差分:**
- `test-driven-development` は 1 機能単位の実装サイクル
- `test-reviewer` はテストコードの品質評価
- 本スキルは、どこにどの種類のテストをどの濃度で置くかを決める上位レイヤー

**想定フェーズ:**
1. 対象システムと品質要求の確認
2. テスト対象の分類
3. Unit / Integration / E2E / 非機能の配分決定
4. フレームワークと実行タイミングの選定
5. CI への統合設計
6. カバレッジ目標と運用ルールの出力

**想定トリガー:**
- 「テスト戦略を設計して」
- 「テストピラミッドを整理して」
- 「どのレイヤーに何をテストすべき？」

**実装難度:** 中

---

### 3.2 `incident-responder`

**目的:**
本番障害発生時の初動、切り分け、暫定復旧、根本原因分析、ポストモーテムまでを体系化する。

**既存との差分:**
- `systematic-debugging` は主に開発中のデバッグ
- 本スキルは本番影響、重大度、関係者通知、復旧優先の意思決定を含む

**想定フェーズ:**
1. インシデント宣言と重大度判定
2. 影響範囲の特定
3. 暫定復旧の選択
4. 根本原因調査
5. 恒久対応方針の整理
6. ポストモーテム作成

**想定トリガー:**
- 「障害対応して」
- 「インシデントを整理して」
- 「ポストモーテムを書いて」

**実装難度:** 高

---

### 3.3 `migration-planner`

**目的:**
フレームワーク、言語、DB、アーキテクチャの移行を、段階導入前提で計画する。

**既存との差分:**
- 実装スキルは個別変更の実施に強い
- 本スキルは現状分析、戦略選択、フェーズ分割、ロールバックの順序設計を担う

**想定フェーズ:**
1. 現状資産と依存関係の棚卸し
2. 移行リスクの評価
3. 移行戦略の選択
4. フェーズ分割と完了条件定義
5. 並行運用・ロールバック設計
6. 切替計画の出力

**想定トリガー:**
- 「段階移行したい」
- 「レガシーを置き換えたい」
- 「Vue 2 から Vue 3 に移したい」

**実装難度:** 中

---

### 3.4 `refactoring-roadmap-builder`

**目的:**
技術的負債を棚卸しし、局所修正ではなく段階的なリファクタリング計画へ落とす。

**既存との差分:**
- `code-simplifier` は問題を直接修正する
- 本スキルは複数モジュールをまたぐ順序設計、依存解消、回帰防止計画を担う

**想定フェーズ:**
1. 負債候補の列挙
2. 依存関係と影響範囲の整理
3. 優先順位付け
4. 小さなリファクタ単位への分解
5. テスト前提と実施順の確定
6. ロードマップ出力

**想定トリガー:**
- 「リファクタリング計画を立てて」
- 「負債を整理して」
- 「どこから直すべきか決めて」

**実装難度:** 中

---

### 3.5 `error-recovery-strategist`

**目的:**
異常系、フォールバック、再試行、劣化運転、障害検知の戦略を先に設計する。

**既存との差分:**
- `incident-responder` は障害発生後の対応
- 本スキルは障害発生前に回復戦略を定義する設計系の実行手法

**想定フェーズ:**
1. 失敗モード列挙
2. 検知方法の定義
3. フォールバック方針の選択
4. ユーザー影響と許容劣化の定義
5. テスト観点への変換
6. 回復戦略シート出力

**想定トリガー:**
- 「エラーハンドリング戦略を決めて」
- 「フォールバックを設計して」
- 「異常系を整理して」

**実装難度:** 中

---

### 3.6 `breaking-change-coordinator`

**目的:**
破壊的変更を互換性管理、非推奨期間、段階導入込みで安全に進める。

**既存との差分:**
- `migration-planner` より狭く、API・ライブラリ・内部基盤の breaking change 管理に特化
- スキル自体が「壊し方を管理するプロセス」を提供する

**想定フェーズ:**
1. 破壊的変更点の特定
2. 利用者影響の整理
3. 互換レイヤまたは移行期間の設計
4. deprecation ポリシー定義
5. 切替条件とロールバック条件定義
6. 移行告知・実施計画の出力

**想定トリガー:**
- 「互換性を保って廃止したい」
- 「破壊的変更を安全に入れたい」
- 「段階的に API を切り替えたい」

**実装難度:** 中

## 4. 優先度中の新規スキル案

### 4.1 `domain-knowledge-structurer`

**目的:**
曖昧な業務知識、判断ルール、用語の揺れを、ユビキタス言語と構造化モデルに落とす。

**差分:**
`domain-modeler` よりも前段の、知識整理と概念境界確定に特化する。

---

### 4.2 `design-system-builder`

**目的:**
既存 UI からトークン、共通部品、命名規則、運用ガイドを抽出し、デザインシステムを構築する。

**差分:**
`ui-designer` は画面設計や方向性提示が中心。本スキルは構築・棚卸し・文書化のワークフローを持つ。

---

### 4.3 `infra-doc-lifecycle`

**目的:**
インフラ構成図、運用手順、変更追跡を一体で維持する。

**差分:**
`aws-architecture-diagram` は図生成のみ。本スキルは図の更新運用とドキュメント同期まで扱う。

## 5. x-driven development 系の案

以下は、`test-driven-development` に続く「X を駆動源にした実行手法スキル」として有力な候補である。

### 5.1 採用候補（強い案）

| 名称 | 駆動源 | 何を先に固定するか | 向いている場面 |
|---|---|---|---|
| `contract-driven-development` | 契約 | API 契約、I/O 契約、境界条件 | API 開発、外部連携、モジュール境界 |
| `risk-driven-development` | リスク | 最大リスク、検証優先順位 | 複雑案件、影響範囲が大きい変更 |
| `failure-driven-development` | 失敗モード | 異常系、検知、回復手段 | 可用性重視、運用を見据えた設計 |
| `rollback-driven-development` | ロールバック可能性 | 戻し方、段階投入条件 | 本番影響が大きい変更、移行 |
| `hypothesis-driven-development` | 仮説 | 仮説、計測方法、判定条件 | 最適化、不確実性の高い改善 |

### 5.2 補助候補（状況次第で有効）

| 名称 | 駆動源 | 注意点 |
|---|---|---|
| `evidence-driven-development` | 計測・観測証拠 | `hypothesis-driven-development` と近いため差別化が必要 |
| `constraint-driven-development` | 制約 | 範囲が広く、設計一般論に拡散しやすい |
| `traceability-driven-development` | 要件追跡性 | `requirements-definer` と重なりやすい |
| `incident-driven-development` | 過去障害 | `incident-responder` と連携前提でないと弱い |
| `migration-driven-development` | 移行そのもの | `migration-planner` と統合設計した方が自然 |

### 5.3 非推奨寄りの案

以下は名前としては成立するが、スキル化すると守備範囲が広すぎるか、既存スキルと衝突しやすい。

- `quality-driven-development`
- `user-driven-development`
- `feature-driven-development`

特に `feature-driven-development` は一般名称として既存の意味づけが強く、
本リポジトリで独自の手法スキルとして追加すると混乱しやすい。

## 6. 最初に作るべき x-driven スキル

### 推奨順位

1. `contract-driven-development`
2. `risk-driven-development`
3. `failure-driven-development`
4. `rollback-driven-development`
5. `hypothesis-driven-development`

### `skill-selector` の補助スキル代替として最初に作るなら

**第一候補は `risk-driven-development`** とするのが良い。

#### 理由

`test-driven-development` が適さないケースは、主に以下である。

- UI コンポーネント中心
- プロトタイプ・探索的コーディング
- インフラ・設定ファイル
- 既存コードのリファクタリング
- テスト環境未整備
- スコープが小さく TDD のオーバーヘッドが見合わない

これらに共通するのは、**テストファーストでは進めにくいが、無秩序に実装すると危険** という点である。
`risk-driven-development` は、テスト先行を前提とせずに、
「何が一番危険か」「何を先に小さく確認すべきか」「どこで止まるべきか」を与えられるため、
`test-driven-development` の代替補助スキルとして最も守備範囲が広い。

#### 補助スキルとしての役割

`risk-driven-development` はプライマリスキルにはせず、
**実装系・変更系のプライマリスキルに付ける補助スキル** として使う。

例:

- `react-frontend-coder` + `risk-driven-development`
- `ci-cd-configurator` + `risk-driven-development`
- `code-simplifier` + `risk-driven-development`

#### 想定フェーズ

1. 変更対象のリスク棚卸し
2. 最大リスクの特定
3. 最初に検証すべき最小変更の決定
4. 検証方法の定義（手動確認、静的チェック、限定実行など）
5. 段階適用と停止条件の定義
6. 実施順と完了条件の出力

#### `skill-selector` への組み込みルール案

`test-driven-development` の判定で「付加しない」となった場合でも、次の条件に当てはまるなら `risk-driven-development` を付加候補にする。

- 実装または変更が発生する
- 仕様不確実性または変更リスクがある
- 手動確認や限定実行での検証は可能
- 変更順序や停止条件を明示した方が安全

逆に、以下では付加しなくてよい。

- 純粋な調査のみ
- 単純な文書更新のみ
- 影響範囲が極小で、順序設計やリスク管理が不要

#### `skill-selector` での優先順位

`test-driven-development` が不適なときの代替補助スキル候補は、以下の順で判定するとよい。

1. **`risk-driven-development`**
   - 汎用の第一候補。UI、インフラ、探索実装、軽量リファクタに広く対応
2. **`contract-driven-development`**
   - API やモジュール境界など、契約を先に固められる場合
3. **`failure-driven-development`**
   - 異常系、回復、可用性が主論点の場合
4. **補助スキルなし**
   - 変更が小さく、追加の手法スキルを載せるほどではない場合

### 理由

#### `contract-driven-development`

最も既存資産と接続しやすい。

- `requirements-definer` と連携できる
- `api-designer` と接続しやすい
- `test-reviewer` や `test-driven-development` に自然につなげられる
- モジュール境界や API 境界がある案件で汎用性が高い

#### `risk-driven-development`

複雑タスクの優先順位付けと相性が良い。

- `scrum-master` の前段または内部補助として有効
- 「何から潰すべきか」を明示できる

#### `failure-driven-development`

可用性や運用性を強化しやすい。

- `error-recovery-strategist`
- `incident-responder`
- `systematic-debugging`

との相互補完が明確。

## 7. スキル設計テンプレート案

今後の実行手法スキルは、以下の骨格を共通化すると品質が揃う。

### 必須セクション

1. **駆動源の定義**
   - 何を起点に意思決定するか
2. **Step 0: スコーピング**
   - このスキルが本当に適用対象か判定する
3. **Phase / Loop**
   - 順序立てた手順または反復サイクル
4. **ゲート条件**
   - 次へ進んでよい条件
5. **エラーリカバリー**
   - 失敗時にどう戻るか、いつユーザーに相談するか
6. **完了レポート**
   - 出力形式、最終判断、残課題

### 理想形

`test-driven-development` を基準にすると、以下を備えているスキルが強い。

- フェーズ定義
- 反復上限
- ツール確認の初期ゲート
- 途中レポート
- 完了レポート
- ユーザー相談条件

## 8. 推奨着手順

### 第1段

1. `test-strategy-planner`
2. `contract-driven-development`

**理由:**
既存スキル群との接続が最も自然で、再利用頻度も高い。

### 第2段

3. `incident-responder`
4. `failure-driven-development`
5. `error-recovery-strategist`

**理由:**
異常系と運用系の厚みが増し、実運用への対応力が上がる。

### 第3段

6. `migration-planner`
7. `rollback-driven-development`
8. `breaking-change-coordinator`
9. `refactoring-roadmap-builder`

**理由:**
大規模変更を安全に進める層を整備できる。

## 9. 結論

`test-driven-development` 型のスキルを増やすなら、単に新しいドメインを増やすのではなく、
**何を駆動源にするかを明示したワークフロー型スキル** を増やすのが筋が良い。

現時点で最も有望なのは以下。

- `test-strategy-planner`
- `contract-driven-development`
- `incident-responder`
- `failure-driven-development`
- `migration-planner`

この順に整備すると、既存の `test-driven-development`、`systematic-debugging`、`scrum-master`、`skill-mentor` と重複せず、
上位レイヤーの実行手法を拡張できる。
