# Kiro Projects Viewer: Overview-first UI Design

## 背景

現行 UI は詳細情報が概要、バックログ、要対応、フロー、履歴に分散し、さらに「かんたん／メンテナンス」の表示モードで情報構造そのものが切り替わる。利用者は最初に見る場所と、モード切替が及ぼす影響を推測しなければならず、実際の機能以上に難しい印象を受ける。

## 目的

- 最初の画面だけで、プロジェクト全体の状態と人間の次の行動を把握できるようにする。
- 詳細情報は、概要内の対象から一段ずつ深掘りできるようにする。
- 操作ラベルを短くし、操作結果の説明はラベルではなく補助テキストや `title` に分離する。
- 既存のデータ、操作、Electron 構成、配色を維持する。

## 検討した案

| アプローチ | 実装コスト | リスク | 保守性 | 拡張性 | 学習コスト | 推奨度 |

## 採用設計

### 情報構造

表示モードを廃止し、常に概要を入口とする。詳細タブは隠さず、概要からも対象タブへ移動できるようにする。

概要は次の順序で表示する。

1. **現在の状態** — 稼働状態、現在の作業、全体進捗を一文と少数の数値で示す。
2. **あなたの対応** — 判断が必要な項目がある場合だけ強調し、「対応する」で要対応へ移動する。
3. **進捗** — 完了、作業中、これからの件数を示し、「タスクを見る」「実行を見る」で詳細へ進む。
4. **成果** — 直近の成果を簡潔に示し、「成果を見る」で履歴へ進む。

プロジェクト定義の編集、運用ルール、リポジトリ設定、リセットは概要から外し、独立した「プロジェクト設定」ダイアログへまとめる。

### ナビゲーションと文言

- 「かんたん表示へ切り替え（概要と要対応だけに絞る）」のような説明口調のモード操作は削除する。
- 主要操作は「対応する」「タスクを見る」「実行を見る」「成果を見る」「プロジェクト設定」のような短い動詞にする。
- 補足説明はボタンの近くの短文またはツールチップに置き、ボタン自体には含めない。
- 詳細タブから概要へ戻るタブ導線は常時見える状態を保つ。

### コンポーネント

- `summary-hero`: 状態文、目標、稼働操作。
- `action-card`: 人間の対応件数と主要 CTA。対応なしの場合は静かな成功状態。
- `progress-card`: 3区分の件数と、バックログ／フローへの導線。
- `deliveries-card`: 直近成果と履歴への導線。
- `project-settings` dialog: 既存の定義編集・危険な操作への入口。

既存のバックログ、要対応、フロー、履歴のレンダリングとデータフローは変更しない。概要からの遷移には既存の `switchTab` を利用する。

### アクセシビリティとレスポンシブ

- アイコンのみのボタンには可視ラベルまたは `aria-label` を付与する。
- `:focus-visible` でキーボードフォーカスを明示する。
- 見出し階層を概要内で順序立てる。
- カード群は狭い幅で一列に落ち、横スクロールを発生させない。
- 色だけで状態を伝えず、状態文と数値を併記する。

### エラーハンドリング

データ未取得、目標未設定、成果なし、要対応なしは、それぞれ空状態の短文を表示する。既存 API 操作の失敗通知と確認ダイアログは維持する。

### テスト

- モード切替 UI と `simpleMode` 分岐がなくなったこと。
- 概要カードが状態に応じて正しい文言・件数・導線を表示すること。
- 各 CTA が正しいタブへ移動すること。
- プロジェクト設定から既存編集・リセット操作へ到達できること。
- 既存ユニットテスト、ビルドが成功すること。
- Electron 画面を 375px、768px、1024px、1440px 相当で確認すること。

## 既存スキルとの関係

- `ui-designer`: 段階的開示、短い CTA、見出し階層、フォーカス可視化を設計へ反映する。
- `react-frontend-coder`: 対象が React ではなくプレーン Electron のため使用しない。
- `webapp-testing`: 実装後の画面操作確認に利用できる。

## Decision Record

| 項目 | 内容 |
|---|---|
| 決定日 | 2026-07-14 |
| 決定者 | チーム |
| 採用案 | 概要ハブ＋詳細タブ |
| 却下案 | すべて折りたたみ（同一画面内の情報量が残る）、ウィザード形式（自由な横断閲覧を妨げる） |
| 主な理由 | 人間の「概要で把握し、関心のある詳細を深掘りする」読み方に沿い、既存機能を壊さず迷いを減らせるため |
| トレードオフ | 詳細タブは残るため、情報量自体は減らない。代わりに入口と見る順序を明確にする |
| 再評価条件 | 概要から詳細への遷移が多すぎる、または設定項目が増えて設定ダイアログが過密になった場合 |
