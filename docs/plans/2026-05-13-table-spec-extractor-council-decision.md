# table-spec-extractor 改良方針 — 合議結論と実測検証計画

**合議日**: 2026-05-13  
**合議方式**: council-system（MELCHIOR / BALTHASAR / CASPER の三賢者合議）

---

## 背景

table-spec-extractor を「仕様書A → 仕様書B への設定値落とし込み」の**検証ツール（短期）**および**自動化基盤（長期）**として使いたい。

現状の取り組みと問題:
- 仕様書A（メーカー仕様等）と仕様書B（自社設定定義書等）を別々の Neo4j DB に登録済み
- キーワード（製品名）で両 DB を検索して比較 → ヒット件数が少なく比較作業として不十分

---

## 合議の核心的発見

> **「全員が改良策を議論していたが、そもそもなぜヒットしていないかの診断がゼロだった」**

「ヒット件数が少ない」という現象には少なくとも3つの原因仮説がある:

| 仮説 | 内容 | 対処 |
|---|---|---|
| A. 語彙ギャップ | 仕様書Aが "Maximum Transmission Unit"、仕様書Bが "MTU最大値" のように表記揺れがある | ベクトル検索（フレームA）が有効 |
| B. 構造解析エラー | Excel結合セルの誤分割・PDF OCR 品質不良でテキストがグラフに正しく入っていない | データ品質修正が先 |
| C. スコープ差 | 仕様書Aと仕様書Bで比較できる項目がそもそも半分程度しか重なっていない | マッピング設計の見直しが必要 |

**原因を診断しないまま改良策を実装すると、どのフレームで進めても効果が出ない可能性がある。**

---

## 統合判断

**診断ファースト・構造設計セカンド・技術実装サードの順序で段階的に進める**

### フェーズ概要

```
Phase 1: 診断       → なぜヒットしないかを実測で特定
Phase 2: 構造設計   → 比較モードとクロスDB対応関係を設計
Phase 3: 技術実装   → 診断結果に基づきベクトル検索等を追加
```

### 設計方針の柱（3者収束点）

1. **クロスDB対応関係の明示化**: 仕様書 A の Cell と仕様書 B の Cell が「同一パラメータ」を指しているという知識をグラフに構造化する（`MAPS_TO` エッジ相当の機能）
2. **確信度スコアの付与**: 自動生成したマッピング候補には必ず確信度スコアを付け、低確信度には人間レビューゲートを設ける
3. **段階的信頼の積み上げ**: 不完全でも「確信度付き比較モード」を早期にデリバリーし、現場担当者のフィードバックを設計に還元する

## Phase 1: 実測検証計画

診断フェーズの合格基準: **50件以上のサンプルで原因を分類し、主因を特定する**

### Step 1-1: 失敗クエリの収集（語彙ギャップの実測）

```bash
# 手作業で落とし込んだペアを5〜10件取り出し、両DBに対してそれぞれ検索する
# 例: 仕様書Aに "MTU" が含まれる場合、仕様書BのDBに "MTU" で検索してヒットするか

python scripts/run.py search "MTU" --profile hw-spec --json
python scripts/run.py search "MTU" --profile sw-spec --json

# ヒット0件になるクエリを集め、その理由を記録する
```

収集観点:
- [ ] ヒットするキーワード vs ヒットしないキーワードの比率
- [ ] ヒットしない場合、検索対象 DB にそのテキストが存在するか（`--dry-run` で確認）

### Step 1-2: Cell.path 構造の対応可能性を実測

```bash
# 両仕様書から各 DB の Cell ノードを 50 件ずつサンプル抽出する Cypher クエリ
MATCH (d:Document)-[:HAS_SECTION]->(s:Section)-[:CONTAINS]->(t:Table)-[:HAS_ROW]->(r:Row)-[:HAS_CELL]->(c:Cell)
WHERE c.is_header = false
RETURN c.path, c.text LIMIT 50
```

確認観点:
- [ ] 仕様書Aと仕様書Bで path の階層数は揃っているか（例: Aが4階層 vs Bが2階層）
- [ ] path の命名規則が対応可能な粒度か（翻訳・略称の揺れを含む）
- [ ] path が空または `null` になっているセルの割合（構造解析エラーの指標）

### Step 1-3: 製品名エンティティの揺れを確認

手作業で使っている検索キーワード（製品名・型番・通称）をリストアップし、実際に DB に格納されているテキストと照合する:

- [ ] 正式型番 / 短縮型番 / シリーズ名 / 略称が混在していないか
- [ ] 日本語表記と英語表記が混在していないか

### Step 1-4: スコープ差の確認

- [ ] 手作業での落とし込みリストの総項目数を数える
- [ ] そのうち仕様書Aに対応する記述が存在する項目数を数える
- [ ] 対応なしの項目が一定割合以上あれば「スコープ差」が主因の一部

### 診断まとめシート（記入例）

| 観点 | 結果 | 主因への寄与 |
|---|---|---|
| ヒットしないキーワードの例 | （記入） | 語彙ギャップ: 高/中/低 |
| Cell.path の階層差 | Aは○階層 / Bは○階層 | 構造差: 高/中/低 |
| path=null の割合 | ○% | 解析エラー: 高/中/低 |
| 製品名の表記揺れ | ○件 | 正規化不足: 高/中/低 |
| スコープ差（比較不可項目）| ○% | スコープ差: 高/中/低 |

## Phase 2: 構造設計（診断完了後に着手）

Phase 1 の診断結果をもとに以下を設計する:

### 2-1: 比較モードの設計

`search` コマンドに `--compare-with <profile>` オプションを追加し、2 DB の結果を並べて表示する:

```bash
# イメージ
python scripts/run.py search "MTU" --profile hw-spec --compare-with sw-spec
```

出力イメージ:
```
[hw-spec] Network > MTU設定 > 最大値: 1500
  → [sw-spec] ネットワーク > MTU上限: 1500  (確信度: 高)
  → [sw-spec] フレーム設定 > 最大フレーム長: 9000  (確信度: 低 ⚠️ 要確認)
```

### 2-2: 製品名正規化辞書の定義

仕様書担当者と協力して、エンティティ辞書を定義する（**フェーズ3開始の前提条件**）:

```json
{
  "entities": [
    { "canonical": "製品A-X100", "aliases": ["X100", "X-100", "X100型"] },
    { "canonical": "製品B-SW200", "aliases": ["SW200", "SW-200"] }
  ]
}
```

## Phase 3: 技術実装（診断・設計完了後に着手）

診断で「語彙ギャップ」が主因と確認された場合のみ実施:

### 3-1: ベクトル検索の追加

```bash
# requirements-vector.txt を追加
sentence-transformers>=2.x

# Neo4j vector index 作成（Neo4j 5.x 以上が必要）
CREATE VECTOR INDEX cell_vector FOR (n:Cell) ON (n.embedding)
OPTIONS {indexConfig: {`vector.dimensions`: 384, `vector.similarity_function`: 'cosine'}}
```

注意事項:
- Neo4j のバージョン確認が必要（5.x 以上でないと vector index は使えない）
- 既存の全 Cell ノードへの埋め込み生成バックフィルが必要（大容量の場合は時間がかかる）
- 日本語仕様書に適したモデルを選定すること（例: `paraphrase-multilingual-MiniLM-L12-v2`）

### 3-2: 確信度スコアと人間レビューゲート

マッピング候補に確信度スコアを付与し、3段階に分類する:

| 確信度 | 閾値（例） | 扱い |
|---|---|---|
| 高 | cosine ≥ 0.90 | 自動受理候補として表示 |
| 中 | 0.75 ≤ cosine < 0.90 | 要確認として表示 |
| 低 | cosine < 0.75 | 要人間判断として警告表示 |

## リスクと対策

| リスク | 対策 |
|---|---|
| 診断が形式的に終わる | 合格基準（50件以上・原因分類完了）をフェーズ移行の条件として明示する |
| 担当者を巻き込めず正規化辞書が作れない | 製品名辞書レビュー完了を Phase 3 開始のゲート条件にする |
| フレームAが独り歩きして確信度設計が後回しになる | 確信度スコアと人間レビューゲートを比較モード MVP に含める（後付け厳禁） |
| 誤マッピングがグラフ全体に伝播する | 対応候補は確信度スコア付きで「提案」扱いにし、人間が承認するまでグラフに固定しない |

## 未解決の問いとユーザーへの確認事項

以下の点が判明すると方針の精度が上がる:

1. **仕様書の実サンプル**: 両仕様書の Cell.path はどのような構造になっているか（階層数・命名規則）
2. **製品名の揺れ**: 検索時に使っているキーワードと DB に格納されているテキストの間に表記の差はあるか
3. **比較対象のスコープ**: 手作業での落とし込みリストの項目数と、仕様書Aに対応記述が存在する項目の割合
4. **Neo4j バージョン**: 現在使用している Neo4j のバージョン（vector index の利用可否を確認するため）
