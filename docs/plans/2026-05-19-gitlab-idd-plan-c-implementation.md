| 決定 | 本計画への制約 |
|------|--------------|
| ①中間（フェーズ境界で中断可能・人介在は低頻度・判断ロジック熟慮必須） | チェックポイントは「常に止まる」でなく「確信度が閾値未満のときだけ停止」。navigator は人でなく別ノードのエージェント。判断ロジックを §5 で明文化する |
| ②B（環境ファセット推奨） | 案C は環境スコープに直接依存しない。確信度スコアの環境タグ付けで将来連携 |

| ④B（品質優先・分散多視点が前提） | II-1 の navigator は「別ノード」を必須要素とする。同一ノード内レビューはフォールバック扱い |

## 3. 新しい worker-role フロー

```
現状:  Phase 1 環境確認 → 2 取得 → 3 着手 → 4 実装ループ(最大5) → 5 提出

案C後: Phase 1 環境確認
       → 2 取得（末尾で確信度スコア初期見積もり）
       → 3 着手（design doc 作成）
       → 3.5 設計レビュー関門 ★新設（II-1）── フェーズ境界の中断点
       → 4 実装ループ（非収束は停止・エスカレーション ★II-2）
       → 5 提出（実行検証ゲート ★II-3 / provisional フラグ ★III-1）
```

1 イシューが Phase 3.5 を境に**複数回の起動にまたがる**。これが決定①「中断可能」の具体形。

## 4. 施策別の実装詳細

### 4-0. 確信度スコア（V-1 から先取り）

- ワーカーが Phase 3（design doc 作成後）に自己評価する。4 軸を各 0–2 で採点:
  要件の明確さ / 影響範囲の把握度 / 既知の落とし穴の有無 / アプローチの一意性。
- 合計を `gl.py` に記録し、イシューコメントに `<!-- gitlab-idd:confidence:{score} -->` を付与。
- スコアで Phase 3.5 の深さを分岐（§4-1）。
- **較正リスク**（自己過信）への対策: 差し戻し発生時に当時の確信度を突合し、V-1 の
  メトリクスで「高確信なのに差し戻された」率を観測する（案A/V-1 と接続）。

### 4-1. II-1 — Phase 3.5 設計レビュー関門

- 新ラベル `status:design-review`（`status:in-progress` と `status:review-ready` の間）。
- **フロー**:
  1. driver（ワーカー）が design doc と approach を提示し、`status:design-review` に更新、
     `<!-- gitlab-idd:design-review-requested:{driver-node} -->` を付けて**ターン終了**。
  2. navigator（**別ノード**推奨）が別起動で design を読み、具体的・反証可能な懸念を
     最低 1 つ投稿（`<!-- gitlab-idd:design-challenge:{navigator-node} -->`）。
  3. driver が次起動で各懸念に応答・design 改訂。全懸念解消で
     `<!-- gitlab-idd:design-review-cleared -->` を付け Phase 4 へ。
- **複数ターンの実質性ゲート**: 1 往復「LGTM」を禁止。navigator のターンは
  「X は Y のため壊れる」型の具体的予測か、なければ「何を確認し、なぜ問題ないか」を明示。
- **確信度ゲート**: 高確信 → 非同期 1 ターンの軽量レビューで可（実質ほぼノンブロッキング）。
  低確信 → 別ノード navigator とのフル複数ターン。
- **navigator 不在時のフォールバック**（単一ノード環境）: `agent-reviewer` の
  design-critique perspective + 人間チェックポイント要求。④の前提が崩れるため §8 の残課題。

### 4-2. II-2 — 非収束の停止・エスカレーション

- 現状 `worker-role.md:569-573`: 5 回超で未解決のまま Phase 5 へ。
- **変更**: 実装ループ 2 回目以降で「同じ指摘の反復」または「diff の発散」を検出したら、
  提出せず `status:design-review` に戻して approach を再検討する。
- 5 回上限到達も「提出」でなく**人間エスカレーション**。
- MR 本文に未解決指摘数を可視メトリクス化（`未解決指摘: N 件`）。N>0 は人間レビュー必須フラグ。

### 4-3. II-3 — 検証の主役を「実行」に

- Phase 5-0 の project-dod チェックを「任意」→「**必須**」に。
- `project-dod.md` 不在のリポジトリ向けに**最小 DoD（ビルド通過・テスト実行）**を
  デフォルト適用する。
- AC を検証コマンドに紐付け: requester-post で AC 各項目に「検証方法（テスト名/コマンド/
  期待結果）」を付与。ワーカーは 5-3 で自己チェックボックスでなく**実行ログ**を貼る。
- テスト不在・実行不能のリポジトリ: ごまかさず「実行検証不能」を明示し人間レビュー必須に倒す。

### 4-4. III-1 — 仕様の共同ドラフト化（+ III-2 同梱）

- **AC 矛盾の解消**: `worker-role.md:470`（変更不可）と `:528`（ライブ仕様書）の矛盾を、
  単一規定に統合 — 「ワーカーは AC 改訂を**提案**できる。単独適用は不可。requester が
  レビュー時に承認する」。
- **推測の隔離**: design doc に「## 未検証の仮定」セクションを新設。`⚠️ 推測` を本文に
  混ぜず隔離。検証/反証されたら取り消し線で「確定/却下」へ遷移させる。
- **provisional フラグ**: タイムアウト着手・未検証の仮定が残る成果物は MR に
  `⚠️ 未検証の仮定あり` フラグを立てる。「沈黙＝同意」を廃止。
- **III-2 取り消し線証跡**: design doc / AC / 未検証の仮定の改訂は削除せず `~~取り消し線~~`
  で残す。古い証跡は末尾の `<details>改訂履歴</details>` に集約し肥大化を防ぐ（決定③）。
- `gl.py` に「取り消し線追記」ヘルパーを追加し、上書きを構造的に防ぐ。

## 5. チェックポイント判断ロジック（決定①「熟慮必須」への回答）

決定①が明示的に求めた「判断ロジックの熟慮」を以下に明文化する。

| 論点 | 規定 |
|------|------|
| **いつ止まるか** | Phase 3.5 は必ず通る。ただし高確信なら軽量 1 ターン非同期（実質ノンブロッキング）、低確信ならフル複数ターン |
| **誰を待つか** | 既定の待ち先は別ノードの navigator（エージェント）。**人を待つのは例外** |
| **人間を待つ条件** | (a) navigator が不在、(b) 確信度が最低閾値未満、(c) 未解決指摘 N>0 のまま提出、(d) 実行検証不能 — のみ |
| **タイムアウト挙動** | navigator 無応答は既存の self-defer/lock 枠組みを流用。タイムアウトは「提出」でなく人間エスカレーション or agent-reviewer フォールバック。**沈黙＝同意は廃止** |

この設計により「人の介在は低頻度」（決定①）を満たす — 通常は別ノード間で完結し、
人間は確信度が低い・検証不能などの例外時のみ関与する。

## 8. 残課題・検証

- **navigator 不在の単一ノード環境**: 決定④（分散多視点で品質担保）の前提が崩れる。
  フォールバック（agent-reviewer + 人間）の品質が十分かを実測する必要がある。
- **確信度スコアの較正**: エージェントの自己評価が過信に偏らないか。V-1 のメトリクスで
  「高確信なのに差し戻された」率を観測して閾値を調整する。
- **レイテンシ実測**: 1 イシューが複数起動にまたがることで、完了までの実時間がどれだけ
  延びるか。決定④で許容済みだが定量把握は必要。
- 検証: 案C 適用前後で差し戻し率・clarification 率・未解決指摘数を比較する。
