# 指摘対応時の「パッチ的修正による整合性崩壊」現象 — 調査と合議による対策計画

> 作成日: 2026-05-16
> 対象ブランチ: `claude/agent-consistency-plan-qYwhL`
> 手法: 根本原因調査 + `council-system` スキル（MELCHIOR / BALTHASAR / CASPER の 3 賢者合議 + デビルズアドボカット）

## 0. 目的

エージェント（Claude Code / GitHub Copilot）が、ユーザーから作業内容への**指摘（フィードバック）**を受けたときに発生する次の悪い現象について、根本原因を調査し、`council-system` で合議して対策計画を確定する。

> **対象の現象**: 指摘するとパッチ的な局所修正を行い、その部分だけを強調し、全体的な整合性が崩れる。

---

## 1. 現象の定義

指摘を受けたエージェントが連鎖的に起こす 3 つの故障モード:

| # | 故障モード | 具体的な症状 |
|---|---|---|
| (1) | **パッチ的な局所修正** | 指摘された箇所だけをピンポイントで直し、同種の問題が残る他の箇所に手を付けない |
| (2) | **修正箇所の過剰強調** | 直した箇所を太字・特別な節・注記・冗長な説明で目立たせる。「ちゃんと直しました」を成果物本体で可視化しようとする（迎合） |
| (3) | **全体的な整合性の崩壊** | 用語のブレ、構造の不揃い、抽象度・詳細度のムラ、他セクションとの矛盾、古くなった相互参照が残る |

(1)→(2)→(3) は連鎖する。局所修正が「指摘箇所だけ厚く・周囲は元のまま」という非対称を生み、その非対称が強調として表面化し、結果として成果物全体の調和が崩れる。

---

## 2. 根本原因分析（深く調査した結果）

調査では本リポジトリの規範ファイル（`karpathy-guidelines/SKILL.md`、`self-checking/SKILL.md`、`.github/instructions/common.instructions.md`）を実地確認した。原因は **行動層** と **制度層** の 2 層に分かれる。

### 2.1 行動層の原因（指摘受領時の処理手順が存在しない）

| 原因 | 内容 |
|---|---|
| **アンカリング** | 指摘文が強いアンカーになり、字面の箇所・字面の不満に注意が固着する |
| **直近指示の支配** | 最新の指摘が支配的になり、元の仕様・設計意図・全体構造が後景化する |
| **迎合による可視化** | ユーザー満足のため修正を「見える化」しようとし、(2) の過剰強調が生じる |
| **全体再読の欠如** | 編集後に成果物全体を読み直さず、波及（リップル）に気づかない |
| **不変条件の非明示** | 成果物の不変条件（用語・構造・抽象度）を明示的に保持しないため、整合性チェックの基準が存在しない |

### 2.2 制度層の原因（既存規範が現象を構造的に強化している）

調査で判明した最も重要な発見。**本リポジトリの既存スキルが、この現象を意図せず強化している。**

- **`karpathy-guidelines` 第3原則「外科的な変更」の越境適用**
  「必要な箇所だけ触る」「壊れていないものをリファクタリングしない」「変更された全行がユーザーの依頼に直接トレースできること」と明記。これは本来**コードの実装スコープ**に対する規範だが、文面上は指摘対応時の**整合性伝播**まで「無関係な変更」として禁止する読み方を許す。

- **`self-checking` の改善ループ「失敗項目のみ改善」**
  「失敗していない項目には手を加えない」「合格済み項目への不要な変更を防ぐ」をベストプラクティスとして明記。整合性チェックで波及を検出しても、「失敗項目ではない既存箇所」として改善対象から外す自己矛盾を生む。

**含意**: 制度層の矛盾を放置したまま行動層の対策だけを足すと、規範同士が矛盾し、新しい対策が既存規範違反として抑制される。**矛盾の除去を対策に内包する必要がある。**

## 3. 対策候補（合議の叩き台 — 5 案、組み合わせ可）

| 候補 | 概要 | 重さ |
|---|---|---|
| **候補1** | 指摘対応プロトコル。`common.instructions.md` 等に「指摘を受けたら局所/系統的を分類→背後の原則を再導出→波及範囲を特定→伝播→過剰強調を避ける」を追記 | 軽 |
| **候補2** | 新スキル `feedback-integration`。指摘を契機に「受領→影響範囲分析→整合伝播→検証ループ」を回す | 中 |
| **候補3** | `self-checking` の拡張。ルーブリックに「整合性(coherence)」ディメンションを追加し、指摘対応後の整合性パスをモード化 | 中 |
| **候補4** | 不変条件台帳（Invariants Ledger）。成果物ごとに用語・構造・抽象度の契約を明示し、全編集を照合 | 中〜重 |
| **候補5** | `karpathy-guidelines` 第3原則の補正。「外科的変更」は実装スコープの規範であり整合性スコープには適用しない、と明記 | 軽 |

## 4. 合議（council-system による洗練）

### 4.1 ROUND 1 — スタンス表明

#### MELCHIOR（論理・分析）
**推奨方針**: 候補1 を第一に採用し、不可分の是正として候補5 を同時実装。候補3・4 は段階導入、候補2 は非採用。
**核心論拠**: 原因は行動層と制度層の 2 層。**制度層の規範矛盾（第3原則・self-checking）を残したまま行動層対策を積むのは論理的に不安定**。候補5 は新規コストほぼゼロで全対策の前提条件を成立させる。候補2 は明示的呼び出しに発動が依存し「指摘」という暗黙の頻出文脈でちょうど発動しない構造的弱点を持つ。
**信頼度**: 中（指示レベル対策の遵守率という外部視点データが欠落）。

#### BALTHASAR（倫理・感情）
**推奨方針**: 候補1 ＋ 候補5 の統合パッチを第一手。候補3・4 は次段、候補2 は当面採らない。
**核心論拠**: 過剰強調は悪意でなく「あなたの声を聞きました」という誠実さの裏返し。必要なのは萎縮させる重い機構でなく、エージェントが「何を大切にすべきか」を思い出せる軽い羅針盤。**最大の受益者は議題に一度も登場しない「成果物の読者（将来の保守者）」** — 太字で叫ぶ箇所より静かな整合性を望む声なき存在。義務論的に、読者には「一貫した成果物を受け取る権利」があり、現行の第3原則はそれを侵害している。
**信頼度**: 中（プロトコルという「指示」が指摘文という「強いアンカー」に対抗できるかは未検証）。

#### CASPER（現実・批判）
**推奨方針**: 候補1 を最優先で即時採用し、実効性が確認できるまで候補2〜4 の新規アセット構築は凍結。候補5 は候補1 に統合。
**核心論拠**: 5 案同時投入は切り分け不能・肥大化を招く。ただし最大の懸念は **「対策＝振り子」問題** — パッチ病の反対側に「過剰系統化（些細な指摘にも全文改稿）」の崖がある。プロトコルには **「局所と判定したら局所で止める早期終了条件」** が必須。また指示は「やれ」でなく **観測可能な出力を要求する形** にしないと形骸化する。現状維持のコスト（整合性の単調劣化 → 最終的な全面書き直し、ユーザー信頼の喪失）も対称に評価すべき。
**信頼度**: 中（候補1 が実際に挙動を変えるかは実績検証不可。3ヶ月後の撤退/移行判断ポイントをセットで決めるべき）。

#### 対立判定
**収束**（3 者とも候補1 ＋ 候補5 方向）。ただし **不自然収束シグナルを軽度検知** — 3 者全員が信頼度「中」で、共通懸念が「指示レベルの介入は形骸化・不読でスルーされる」に集中（視野狭窄の可能性）。→ 強制デビルズアドボカットを発動。

### 4.2 ROUND 2 — デビルズアドボカット（収束方針への反論）

**反論の核心**: 収束方針は「問題の原因」と「対策の場所」を取り違えている。故障モードはすべて「指摘を受けた直後のエージェントが視野狭窄に陥る」ことで起きるのに、収束方針は**その視野狭窄に陥っている当人が、追記された指示を律儀に読み・解釈し・自己適用すること**を期待している。消火活動を燃えている本人の善意に任せるのと同じ。3 賢者が揃って挙げた共通懸念は「収束の副産物」ではなく「収束方針そのものの致命的欠陥」。

**最大の指摘 — 検証可能性の欠如**: 候補1 には **観測可能な出力要求がない**。「整合性に配慮せよ」と追記しても、エージェントが「配慮したつもり」で局所修正を返したとき、守られたか無視されたかを判定する手段がない。**検証不能な対策は「実装した」と「効いている」の区別がつかず、効果測定が不可能なため「段階導入」のフェーズ2 が永遠に来ない。**

**制御工学的な逆転**: 挙動制御の強制力は「構造 > ツール > 検証ゲート > 指示文」の順で弱い。収束方針は最弱層（指示文）を最優先、強い層（候補3 の検証ゲート・候補4 の構造）を後回しにした。視野狭窄に陥った主体を制御するなら、その主体の善意に依存しない外部チェックポイントを置くのが定石。

**代替方針**: 候補3 を最小スコープで先行し、観測可能な整合性ゲートを入口にする。候補1 は「ゲートを通すために読まざるをえない補助文書」として従属させる。

### 4.3 SYNTHESIS — 統合判断

#### 議論の要点

| 観点 | MELCHIOR | BALTHASAR | CASPER | デビルズアドボカット |
|---|---|---|---|---|
| 推奨 | 候補1+5、3/4 段階、2 非採用 | 候補1+5 統合パッチ | 候補1 即時、2〜4 凍結 | 候補3 を入口に、候補1 を従属 |
| 信頼度 | 中 | 中 | 中 | （反証役） |

#### 前提監査

| 前提 | 状態 | 統合での扱い |
|---|---|---|
| 制度層の規範矛盾（第3原則・self-checking）が行動を駆動している | 共有（3 者）/ ファイル文面で部分裏付け | 採用 — 矛盾除去を対策に内包 |
| 指示レベルの追記をエージェントが読み・適用する | **対立** — M「成立確率高」 vs デビルズアドボカット「最弱層・無視される」 | **未解決前提。検証可能性で補強する** |
| 「段階導入」は実際に次段へ進む | **対立** — 収束方針は前提、デビルズアドボカットは「測定手段を後回しにした閉路」と否定 | 採用せず — 曖昧語を観測可能なトリガーに置換 |

#### 統合判断

**収束方針（候補1 ＋ 候補5）とデビルズアドボカットの反論は、実は対立していない。** 「指示文 vs 検証ゲート、どちらを入口にするか」は誤った二分法である。候補1（プロトコル＝何をすべきか）と観測可能な整合性ゲート（候補3-lite＝やったことを示せ）は、**同じ介入を 2 つの角度から見たもの**であり、不可分の 1 パッケージとして同時に出すのが正しい。

> **統合方針: 候補1（指摘対応プロトコル）＋ 候補5（第3原則のスコープ補正）＋ 候補3-lite（観測可能な整合性チェック報告と self-checking の矛盾除去）を、相互依存する 1 つのパッケージとして同時実装する。候補4 は観測可能なトリガー条件付きで予約し、候補2 は非採用とする。**

**各賢者の指摘の統合のしかた**:
- MELCHIOR の「制度層矛盾を先に除去」→ 候補5 と self-checking 矛盾除去をパッケージに内包。
- BALTHASAR の「軽い羅針盤」「読者の権利」→ プロトコルを軽量に保ち、整合性チェック報告は成果物本体でなくチャット応答側に出す（読者の成果物を汚さない）。
- CASPER の「振り子の逆振れ」→ プロトコルに早期終了条件を必須化。「観測可能な出力を要求せよ」→ 整合性チェック報告として実装。
- デビルズアドボカットの「検証可能性」→ 整合性チェック報告という観測可能な出力をパッケージに内包し、候補1 の検証不能問題を解消。「段階導入の閉路」→ 候補3 のルーブリック本格化と候補4 を、日付・回数ベースの観測可能なトリガーで予約。

#### 残った対立
「指示レベルの介入がアンカリングに対抗できるか」は経験的に未検証のまま。本計画はこれを **3ヶ月後の効果測定** で検証する設計とし、改善が見られなければ候補3 のルーブリック本格化 → 候補4 → 候補2 へ移行する。

### 4.4 事前検死（Pre-mortem）

**仮定**: 本パッケージを実装したが、6ヶ月後に明確に失敗した。最も可能性が高い失敗シナリオ:

1. **整合性チェック報告の儀式化** — 報告欄が毎回同じ定型文・空欄になり、「対策済み」ラベルだけが残って問題が不可視化される。
2. **振り子の逆振れ（過剰系統化）** — 誤字 1 つの指摘にも系統的対応を発動し、ユーザーが触ってほしくない箇所まで「整合性のため」と改変。パッチ病をオーバーホール病に置換しただけになる。
3. **過剰強調の転移** — 「整合性に配慮しました」が成果物本体に新たな強調・注記として現れ、故障モード (2) が形を変えて再発する。
4. **候補3・4 の永久凍結** — 「段階導入」が候補3・4 を着手しない口実になる。

**事前対策（実装計画に織り込み済み）**:
- [ ] 整合性チェック報告に「確認した具体的な箇所名を最低 1 つ」列挙させ、空欄を構造的に許さない（シナリオ1）。
- [ ] プロトコルに早期終了条件「局所と判定したら理由を述べて局所で止める」を明記（シナリオ2）。
- [ ] 整合性チェック報告はチャット応答チャネル限定、成果物本体への記載を明示禁止（シナリオ3）。
- [ ] 候補3 本格化・候補4 の発火を日付/回数ベースの観測可能なトリガーで予約（シナリオ4）。

## 5. 実装計画

統合方針に基づく具体的なアクション。すべて相互依存するため **1 つの PR にまとめて実装** する。

### アクション 1: 指摘対応プロトコルの追記（候補1）

`.github/instructions/common.instructions.md` に新セクション「指摘・フィードバックを受けたときの手順」を追加する。追記する手順:

```
## 指摘・フィードバックを受けたときの手順

既存の成果物（コード・ドキュメント・設計）への指摘を受けたら、編集前に次を行う:

1. 意図の再導出 — 指摘の字面でなく、背後にある原則・期待を一文で言語化する。
   「この指摘が本当に求めているのは何か」。
2. スコープ判定 — 指摘を「局所」か「系統的」かに分類し、判定根拠を述べる。
   - 局所（その箇所だけの問題）と判定 → 理由を述べて局所修正で止める。早期終了する。
   - 系統的（同種の問題が他にもある／全体の整合性に波及する）と判定 → 手順 3 へ。
3. 波及範囲の特定 — 同じ原則が及ぶ箇所・用語・相互参照・抽象度を成果物全体から洗い出す。
4. 一貫した伝播 — 洗い出した全箇所に同じ修正を適用する。新しい用語・構造を
   導入したら成果物全体で揃える。
5. 過剰強調の禁止 — 修正箇所を太字・特別な節・注記・冗長な説明で目立たせない。
   変更は本文に自然に溶け込ませる。「直したアピール」は成果物本体に書かない。
6. 整合性チェック報告 — 編集後、チャット応答（成果物本体ではない）に短く報告する:
   「波及候補として確認した箇所（具体名を最低 1 つ）」と「各箇所を変更した／
   しなかった理由」。
```

### アクション 2: karpathy 第3原則のスコープ補正（候補5）

`.github/skills/karpathy-guidelines/SKILL.md` の「3. 外科的な変更」末尾に注記を追加:

```
**スコープの区別:** 「外科的変更」は実装スコープ（依頼されていない機能追加・
無関係なリファクタの禁止）の規範であり、整合性スコープには適用しない。
指摘起点である原則・用語・構造を変えた場合、その原則が論理的に及ぶ範囲全体に
一貫して反映するのは本原則の違反ではなく、むしろ必須である。
ただし整合性のための伝播は「指摘の根本原則が及ぶ箇所」に限る ——
スタイルの好みによる改変は依然禁止。
```

`metadata.version` を `1.0.1` → `1.0.2` に更新する。

### アクション 3: self-checking の矛盾除去（候補3-lite）

`.github/skills/self-checking/SKILL.md` を 2 点修正:

1. Step 4「改善時の注意」と「ベストプラクティス」表の「失敗した基準のみ改善する」に注記を追加:
   ```
   ただし、ある基準の改善が用語・構造・前提に波及する場合、その波及先は
   失敗項目でなくても整合のために修正する。整合性のための波及修正は
   回帰リスクではなく回帰防止である。
   ```
2. `references/rubrics/document.md` に「整合性（coherence）」ディメンションを追加する（用語の一貫性・構造の均一性・抽象度の揃い・相互参照の正しさを評価）。`code.md` / `research.md` への展開は本格化フェーズ（下記トリガー）まで保留。

### アクション 4: 候補4・候補3 本格化の予約（観測可能なトリガー）

「段階導入」を曖昧にせず、発火条件を明示する:

| 予約項目 | 発火トリガー |
|---|---|
| 候補3 ルーブリックの全 artifact_type への展開 | 効果測定（3ヶ月後）で document の整合性ディメンションが有効と確認できた場合 |
| 候補4 不変条件台帳 | 同一成果物で同種の不変条件（用語集・構造規約）が整合性チェック報告で 3 回以上手作業確認された場合、その成果物に限り台帳を作成 |
| 候補2 専用スキル化 | 効果測定でプロトコルの遵守率が低く、かつ手順が複雑化していると判明した場合に再検討 |

### 実装順序

すべて 1 PR。アクション 1〜3 は相互依存（矛盾を残すと不安定）のため不可分。アクション 4 は本ドキュメントへの記載で完了。

## 6. 効果測定と撤退・移行条件

| 項目 | 内容 |
|---|---|
| **測定時期** | 実装から 3ヶ月後（2026-08 目安） |
| **測定方法** | 指摘対応を含むセッションのサンプルをレビューし、「指摘対応後に新規の整合性崩壊が混入した率」と「整合性チェック報告が形骸化（定型文・空欄）していないか」を確認 |

| **振り子の逆振れ検知** | ユーザーから「最小限直してと言ったのに全部変わった」という不満が出たら、プロトコル手順 2 の早期終了条件の文面を強化 |

## 7. 認識論的謙虚さ

⚠️ **限界の開示**: 本合議は同一 LLM ベースの疑似的多様性に基づく内的ロールプレイで構築された。独立した知性による合議ではなく、本質的なバイアス（楽観・確証・形式化への偏愛）が完全には排除されていない可能性がある。

**信頼度: 中** — 「制度層の規範矛盾を先に除去する」「観測可能な出力で検証可能性を担保する」という構造的判断は堅固で、規範ファイルの実地確認による裏付けもある。一方、**指示レベルの介入が指摘文という強いアンカーに実際どれだけ対抗できるか**は経験的に未検証であり、これが信頼度を「高」に上げきれない最大の理由である。

**この判断を変える可能性がある情報**:
- 指摘対応プロトコル導入後の実際の遵守率（3ヶ月後の効果測定で判明）。
- パッチ挙動の主因が規範文面か、学習済みの強い事前分布か。後者であれば指示追記の効果は限定的で、候補3・4 の構造的強制を前倒しすべきとなる。
- 整合性崩壊の波及が「人間でも気づけない種類」であった場合、手順追加では不足し候補4（不変条件の明示化）を先行すべきとなる。

**最終判断はユーザー自身が行うこと。** 本ドキュメントは選択肢の整理と事前検死を提供したものに過ぎない。
