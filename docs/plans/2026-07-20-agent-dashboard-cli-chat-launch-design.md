# Agent Dashboard CLIチャット起動ボタン設計

## 目的

選択中プロジェクトの Git 管理ルートを作業ディレクトリとして、全体設定のエージェント
（CLI・モデル）で対話セッションを開けるようにする。セッションは tmux 上で実行し、
操作と閲覧には外部ターミナルを使う。

## 採用設計

ボタンはプロジェクトヘッダーのプロジェクト名右側に、文字付きの
「CLIチャットを開く」として置く。

設定の参照元は全体設定だが、操作対象は選択中プロジェクトである。したがって、
全体設定ページ内ではなく、現在の操作対象を示すプロジェクトヘッダーへ置く。
全体設定、概要、タスクなど表示中のタブにかかわらず同じ場所から起動できる。

### 比較した配置

| 配置 | 利点 | 欠点 | 評価 |
|---|---|---|---|
| プロジェクトヘッダー右側 | 対象プロジェクトと操作の関係が明確で、どのタブからも使える | ヘッダー幅を1ボタン分使う | 採用 |
| 全体設定のAIアシスタント内 | 参照する設定との関係が近い | 設定ページを開かないと日常操作に使えない | 不採用 |
| サイドバー上部 | 常時見える | どのプロジェクトで開くか分かりにくい | 不採用 |

## 動作

- 選択中プロジェクトの Git 管理ルートを cwd にする。
- 保存済みの全体設定 `agent.cli` と `agent.model` を使う。
- 既存のエージェント定義と外部ターミナル起動処理を再利用する。
- プロジェクトごとに安定した tmux セッション名を使う。
- セッションが無ければ CLI chat を起動し、あればそのセッションへ再接続する。
- プロジェクト未選択時はボタンを無効にする。
- CLI が未設定または利用不能なら起動せず、操作箇所の近くで
  「全体設定 > エージェント」を案内する。
- 起動の受付失敗はダッシュボード上で通知し、CLI 起動後のエラーは外部ターミナルに残す。

未保存のフォーム値は使わない。「保存した全体設定で開く」という一貫した意味にし、
起動操作が暗黙に設定を保存する副作用を持たないようにする。

## UI

- ボタンラベルは「CLIチャットを開く」とし、アイコンだけにしない。
- `type="button"` を指定し、既存のフォーカスリングを維持する。
- 操作領域は44px相当を確保する。
- 狭幅ではプロジェクト名の下へ折り返し、横スクロールを発生させない。
- 起動中は二重操作を避けるため一時的に無効化し、結果を `aria-live` でも通知する。

## セッション識別

Git 管理ルートの正規化済みパスから既存方式と同様に短いハッシュを作り、tmux
セッション名へ使う。同じプロジェクトでは再接続でき、異なるプロジェクト同士は衝突しない。
CLI 種別の変更後に古いセッションへ誤接続しないよう、識別子には CLI 名も含める。

## テスト

最小の自動テストで次を確認する。

- ボタンがプロジェクトヘッダー内にあり、文字ラベルとアクセシブル名を持つ。
- 未選択では無効、プロジェクト選択後は有効になる。
- Git 管理ルート、保存済みの CLI・モデルが起動処理へ渡る。
- 同じプロジェクトと CLI は同じ tmux セッション名になり、CLI が違えば分かれる。
- CLI 未設定時は起動せず、全体設定への案内を返す。
- 既存セッションには再接続し、新規セッションを重複作成しない。

## 実装計画

1. プロジェクトヘッダーへボタンを追加し、選択状態と連動させる。
2. 既存のエージェント設定解決と外部ターミナル起動経路を使って、tmux の作成・再接続を行う IPC を追加する。
3. 上記の最小テストを追加し、既存の agent-dashboard テストを実行する。

## 既存スキルとの関係

- `ui-designer`: 操作対象の近くへの配置、文字ラベル、フォーカス、狭幅レイアウトへ反映した。
- `webapp-testing`: 実装後の実画面確認に利用できるが、設計段階では使用しない。
- `react-frontend-coder`: 対象はプレーン Electron のため使用しない。
