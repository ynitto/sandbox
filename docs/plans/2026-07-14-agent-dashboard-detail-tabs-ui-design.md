# Agent Dashboard: Detail Tabs UI Design

> **由来**: 対応する `docs/plans/*kiro-projects-viewer*` を置換せずクローンし、
> `agent-dashboard` / `agent-project` / `agent-flow` 名称へ改称した系統。
> 改称方針: [`docs/designs/agent-tools-rename-design.md`](../designs/agent-tools-rename-design.md)。

## 背景

概要タブは「全体を把握してから詳細へ進む」構造へ変更したが、要対応タブは全カードを縦に展開し、実行タブは run 一覧と概要・グラフ・工程詳細の3段独立スクロールを同時表示している。どちらも詳細情報が最初から露出し、見る順序を利用者に委ねている。

## 目的

- 要対応と実行にも「一覧で全体把握 → 1件を選択 → 必要な詳細だけ深掘り」の階層を導入する。
- 人間の判断や操作が必要な項目を最初に示す。
- 既存の状態判定、操作、GitLab連携、入力保持を維持する。
- 375pxから1440pxまで、現在地と選択対象を見失わない構成にする。

## 検討した案

| アプローチ | 実装コスト | リスク | 保守性 | 拡張性 | 学習コスト | 推奨度 |
|---|---|---|---|---|---|---|
| 一覧＋詳細 | 中 | 低 | 高 | 高 | 低 | ★★★ |
| カード折りたたみ | 低 | 中 | 中 | 中 | 中 | ★★☆ |
| 工程別ウィザード | 高 | 高 | 低 | 低 | 高 | ★☆☆ |

## 採用設計

### 共通構造

要対応・実行ともに、上部へ件数サマリーとフィルターを置き、その下をマスター・ディテール構成にする。左側は選択用の短い一覧、右側は選択した1件だけの詳細とする。選択行は色と枠の両方で示し、キーボード操作でも到達できるボタンとして実装する。

デスクトップでは左右分割、狭い画面では一覧と詳細を一度に並べず、「詳細を見る」「一覧へ戻る」で切り替える。未選択時は、現在のフィルターで先頭の項目を自動選択する。

### 要対応タブ

上部に次の件数を表示する。

- 未対応
- 送信済み
- 回答済み
- GitLab

初期フィルターは未対応とする。左の一覧には種別、リスク、タイトル、判断目的だけを表示する。右の詳細は次の順序にする。

1. 何を判断するか
2. 理由・概況・失敗要因・変更内容
3. 回答欄と主要操作
4. Spec・成果物
5. 折りたたまれた判断材料

GitLabレビューは同じ縦画面の末尾へ常時併載せず、件数付きフィルターで切り替える。ポーリング時には選択中ID、スクロール位置、入力中の回答、判断材料の開閉状態を保持する。

### 実行タブ

上部に実行中、要確認、完了の件数を表示する。左の一覧は依頼内容、進捗、`runAdvice` による「次に起きること」を主表示とし、run IDやリトライ世代は補助情報へ下げる。
（要確認 = 人の判断・検収・再操作が必要な実行。delivery_review で run が done でもタスクが検収待ちならここに載せる）

右の詳細は内部ビューで段階表示する。

- **概要**: advice、依頼内容、進捗、主要操作。run ID・戦略は補助情報として折りたたむ。
- **工程**: タスクグラフと選択工程の詳細。工程未選択時は選び方を表示する。
- **履歴**: アクティビティ、応答時刻、リトライや内部状態などの技術情報。

現在の「概要・タスクグラフ・工程詳細」の3段独立スクロールは廃止する。内部ビューは1つだけを表示し、現在のビューを明確に示す。人の操作が必要な場合だけ主要ボタンを強調する。

### データと状態

新しいUI状態として、要対応フィルター・選択中need ID・モバイル詳細表示、実行詳細ビューを保持する。既存の `state.flowRunId` と `state.flowNodeId` は引き続き選択状態の正典とする。

表示モデルは純関数へ分離し、件数分類・既定選択・フィルター結果をテスト可能にする。ファイル読み書き、IPC、状態判定ロジックは変更しない。

### エラーハンドリング

- フィルター対象が0件の場合は、その状態に対応する空メッセージを表示する。
- 選択中項目が更新で消えた場合は、同じフィルターの先頭へ安全に移動する。
- 入力中は既存どおり再描画を抑止し、回答を失わない。
- runやノードの読み込み失敗は既存トーストを維持し、一覧選択は保持する。

### アクセシビリティ

- 一覧行はbuttonとして操作可能にし、選択状態を `aria-pressed` または `aria-current` で示す。
- 内部ビュー切替はタブとして意味付けし、フォーカス表示を維持する。
- 色だけで状態を伝えず、状態名と件数を併記する。
- モバイルの戻る導線を詳細の先頭に配置する。

### テスト

- needの未対応・送信済み・回答済み分類と既定選択。
- フィルター切替、選択、回答入力の保持。
- GitLabフィルターへの切替。
- runの実行中・要確認・完了分類。検収待ち（delivery_review）の done run も含める。
- 実行の概要・工程・履歴切替とノード選択。
- 既存のneeds、flow、advice、GitLab関連テスト。
- 375px、768px、1024px、1440pxで水平スクロールがないこと。

## 既存スキルとの関係

- `ui-designer`: マスター・ディテール、段階的開示、選択状態、フォーカス可視化を反映する。
- `webapp-testing`: 実装後の画面幅、フィルター、選択、内部ビュー遷移を検証する。
- `react-frontend-coder`: プレーンElectronのため使用しない。
