## 2. 変更リスクの評価軸

各施策のリスクは次の 4 軸で判定した。

| 軸 | 内容 | 高リスク側 |
|----|------|-----------|
| **変更範囲** | doc のみ / スクリプト / データモデル・スキーマ | データモデル変更 |
| **既存原則との衝突** | 行動原則・設計思想と整合するか | 「一気通貫」等の最上位原則に抵触 |
| **振る舞い・コスト影響** | スループット・レイテンシ・トークンコストへの影響 | 全イシューに恒常コスト増 |
| **移行・後方互換** | 既存イシュー・記憶・ラベルへの影響 | 移行作業や非互換を伴う |

### 高リスク施策の詳細

- **II-1（設計レビュー関門）**: 新フェーズ追加で worker-role の構造が変わる。行動原則2
  「一気通貫」と正面衝突（分岐①）。全イシューにレビュー往復のコスト・レイテンシが乗る。
  確信度ゲートで軽減するが、ゲート自体が新規。**最も重い施策。**
- **III-1（仕様の共同ドラフト化）**: requester と worker の role 境界（誰が AC を所有するか）を
  再定義する（分岐③）。provisional フラグは MR の新しい状態。AC の living-spec 矛盾を
  解消する点はプラスだが、複数 reference の同時改訂を伴う。
- **IV-3（環境固有知見スコープ）**: ltm-use のコア（memory-format・save/recall/promote）に
  新スコープを追加するデータモデル変更。既存記憶の移行が必要。ブレスト元でも「高コスト」と
  明記済み。単独で大きなプロジェクトになる。

### 低リスク施策（設計判断・移行不要）

I-1 / I-3 / III-2 と V-1 の一部（eval.json・knowledge 使用ログ・KPI）は、doc・テンプレート・
観測の追加のみで既存原則と衝突せず、いつでも巻き戻せる。

## 3. 計画案（パッケージ）

施策を選択可能なパッケージに束ねた。**施策単位での組み替えも可能。**

### 案A — 即効パッケージ（設計判断不要・低リスク・可逆）

- **含む**: I-1, I-3, III-2, V-1（観測系: eval.json・knowledge 使用ログ・KPI）
- **特徴**: すべて doc / テンプレート / 観測の追加。既存原則と衝突せず、いつでも巻き戻せる。
  分岐点の議論を待たずに先行できる。
- **ねらい**: 知識参照の頻度を上げ、記録を高信号化し、証跡を残す。観測を入れて
  以降の施策の効果測定を可能にする。

### 案B — 学習ループ完成パッケージ

- **含む**: 案A + I-2
- **特徴**: 案A に save→recall→rate のクローズとレトロ即時化を足し、「記憶はあるが学習がない」
  状態を解消する。第1観点の課題①（クエリ頻度）の根治。I-2 は rate 自動適用の誤発火
  （正当な記憶を差し戻しで不当に減点）に注意が要る（中リスク）。
- **ねらい**: 記憶を「貯める」から「学習する」へ。

### 案C — 思慮の構造化パッケージ（中核・要設計判断）

- **含む**: II-1, II-2, II-3, III-1
- **特徴**: 当初の不満（思慮が浅い・迎合・振り返りなし・やり直しなし）の中核に直接効く。
  ただし II-1 / II-2 / III-1 は「一気通貫」を最上位から降ろす設計判断（分岐①）が前提。
  II-3（実行検証）のみ判断不要で先行可能。高効果・高コスト。
- **ねらい**: 立ち止まり・複数視点・実行検証を構造として組み込む。

### 案D — 分散実行健全化パッケージ

- **含む**: IV-1, IV-2, IV-3
- **特徴**: gitlab-idd 固有の価値（分散・環境固有知見）を強化。IV-1（分解）と IV-2（継ぎ目封鎖）は
  中リスクで着手しやすい。IV-3 は ltm-use のデータモデル拡張（分岐②）を伴う重量級で、
  単独プロジェクト化を検討する。
- **ねらい**: 分散が誤りを増幅する構造を、検証と知見共有で健全化する。

> 案A ⊂ 案B。案C・案D は案A/B と独立に選択できる。

| # | 分岐点 | 決定 | 補足 |

| ① | 一気通貫を降ろすか | **中間** — フェーズ内は一気通貫、フェーズ境界で中断可能 | スループットは犠牲可。人の介在は「低頻度なら許容」。チェックポイントの**判断ロジックは熟慮必須**（いつ止まる／誰を待つ／タイムアウト挙動） |

| ④ | throughput vs quality | **B — 品質優先** | 遅さは許容。ただし条件付き — **分散多視点（II-1 の別ノード navigator）がローカル完結より高い品質を実際に担保すること**が前提 |

### 決定の含意

- ①中間 + ④品質優先 → **案C（思慮の構造化）に進む**。II-1 の設計レビュー関門が中核。
- ①「判断ロジック熟慮必須」「人介在は低頻度」→ チェックポイントは「常に人を待つ」ではなく
  「自律で進めるが、確信度が閾値未満のときだけ停止・関与要求」。確信度スコア（V-1）が
  停止判定の心臓部になる。
- ③中間 → III-1 を「提案 → 承認」モデルで実装。`worker-role.md:470` と `:528` の矛盾は
  「ワーカーは提案可・単独適用不可」で解消する。
- ④条件 → II-1 は「別ノード navigator」を必須要素とする。同一ノード内レビューは品質担保に
  ならない。

## 8. 分岐点② B案（ltm-use データモデル拡張）のリスク深掘り

**前提**: B案 = ltm-use に `home` / `shared` と並ぶ「ノード/環境」スコープを追加する。

### B-R1. スコープモデルが線形から非線形になる

現状は `home → shared` の一方向・線形。node を足すと昇格経路がグラフ化する
（node→home? node→shared? 一部の環境知見は汎用化して shared へ、多くは node に留める）。
ltm-use の価値の核は promotion ロジック（v6 設計済み: `2026-05-09-ltm-use-promotion-logic-v6.md`）。
新スコープはこれを再オープンさせる。**最も深い構造リスク。**

### B-R2. recall フォールバック連鎖の複雑化

現状 recall は home → shared の 2 段。node 追加で 3 段になり、スコープ横断ランキング
（強く関連する node 記憶 vs 弱く関連する home 記憶のどちらを上位にするか）が難問化する。
ハイブリッド 4 軸ランキングに「スコープ親和性」軸の追加が必要。recall も遅くなる。

### B-R3. node_id の不安定性 ★ 最重要・要注意

node スコープは node_id をキーにする。だが remote / ephemeral 実行環境では
**コンテナ再生成のたびに node_id が変わりうる**（コンテナはリポジトリを毎回 clone し直す）。
すると node スコープ記憶が毎回孤児化し、「環境知見を永続させる」という目的そのものが破綻する。
→ node_id ではなく**決定的な環境シグネチャ（OS + 利用可能ツール構成のフィンガープリント）**で
キーする設計が要る。これは実装でなく**設計の問題**。

### B-R4. 保存場所と同期のジレンマ

`home` はローカル、`shared` は git 管理。node はどちらか。
- ローカルのみ → マシン消滅で失われ「分散活用」にならない（ユーザーの価値命題と矛盾）。
- git 管理 → 全ノードの環境記憶が全ノードに同期され、各ノードのストアが肥大する。

### B-R5. core ティアへの破壊的変更の波及

スコープ解決は `memory_utils.py` にあり、save / recall / list / rate / promote / cleanup /
consolidate / review / build_index / sync 系など **約 15 スクリプト**に波及する。
ltm-use は `tier: core` で、依存スキル（gitlab-idd の 5-5a 等）にも連鎖する。

### B-R6. promotion / consolidation 意味論の再設計

脳モデル（episodic → consolidate → semantic/procedural → promote）に node がどう挟まるか未定義。
node 上の episodic を distill した汎用知見を shared に上げるべきか等、ライフサイクルの再設計が要る。

### B-R7. 移行とフォーマット後方互換

既存記憶は home/shared に据え置けるが、frontmatter にスコープ/環境キーを足すと旧ファイルへの
デフォルト補完が必要。index は `build_index --force` で再構築可（リスク低）。

### B-R8. スコープ増による運用複雑化（workspace 廃止の逆行）

ltm-use は過去に `workspace` スコープを**廃止**して 2 スコープに簡素化済み
（SKILL.md「`workspace` スコープは廃止。全記憶は home に保存する」）。node 追加はその簡素化の
部分的逆行。再導入は「過去に切ったものを戻す」高いバーを越える必要がある。

### 緩和策と推奨実装形態

**最大の発見**: B案の目的（環境固有知見の分散活用）は、真の第 3 スコープを足さずとも
**`shared` 内の「環境ファセット」**で大半達成できる。

- environment 記憶 = `shared` スコープ + `environment:` frontmatter ファセット + 予約カテゴリ。
- これで B-R1（昇格モデル）・B-R2（フォールバック段数）・B-R8（スコープ増）をほぼ回避。
- recall には「環境アフィニティ」軸を 1 つ足すだけ（B-R2 を最小化）。
- ファセット値は node_id でなく**環境シグネチャ**で決める（B-R3 対策）。
- 段階導入: まず環境タグ + recall アフィニティ → 計測 → 真のスコープが必要と判明したときだけ
  B-R1〜R8 のコストを払う。

→ **「B（拡張する）」を選びつつ、拡張を新スコープでなく既存スコープ内のファセットとして
実装する**のが最小リスク。価値の 8 割をリスクの 2 割で得られる。要確認の論点（§9）。
