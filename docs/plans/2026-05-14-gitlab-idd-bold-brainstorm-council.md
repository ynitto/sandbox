# gitlab-idd ハーネス案 — ブレスト（大胆案）と合議洗練

> 作成日: 2026-05-14
> 対象ブランチ: `claude/plan-agent-workflow-tools-Hks7o`
> 関連: `2026-05-14-gitlab-idd-human-overhead-reduction.md`（堅実案、本ドキュメントの前段）

## 0. 目的

前段ドキュメント（H-1〜H-12）は既存ツールの延長線にある「堅実案」だった。
本ドキュメントでは制約を意図的に外し、**フローそのものを書き換える大胆案**を 5 案出す。
そのうえで各案を `council-system` スキル（MELCHIOR / BALTHASAR / CASPER の 3 賢者合議）で
洗練し、SYNTHESIS（統合結論）と Pre-mortem（事前検死）を併記する。

> 大胆さの判定基準: 「既存スキル本体を変えない」「人間を介在地点として残す」のいずれかを
> *破る* 案であること。前段ドキュメントの設計制約をあえて緩める。

## 1. ブレスト案（X-1〜X-5）

### X-1. ナイト・ブリーフィング — 「朝のメニュー提示」モデル

**着想**: 人間が起きている時間に「依頼を考える → イシュー化する」を全部やめる。
夜間にエージェント群が自律的に **次の依頼候補** を作って朝に提示する。

**動作**:
- 夜 22:00〜翌 6:00 の間、`night-briefing` daemon が以下のソースを並列スキャン:
  - リポジトリ全体の `TODO:` / `FIXME:` / `XXX:` コメント差分
  - 直近 1 週間の **マージ済 MR のレビューコメント**（積み残し検出）
  - 直近 1 週間の **CI 失敗・ログエラーの頻出パターン**
  - 既存イシュー本文の `## リスクと対策` セクション（未解決リスク）
  - （任意）Slack / Teams DM・直近の議事録テキスト
- 各ソースから候補タスクを抽出 → `requirements-definer` → `gitlab-idd` リクエスター下書きまで作る
- 朝 7:00 に「本日のメニュー（5〜10 件）」を Markdown / Slack / メールで配信
- 人間は **数字で選ぶだけ**（「1, 3, 5 を実行」）→ 該当下書きが本投稿される

**ねらう変化**:
- B-1（要件インプット）に費やす人間時間 30 分以上 → 1〜2 分
- 「何をやるか思いつかない朝」のロスを排除
- リポジトリの「健康診断」を毎日強制実施

**前段案との違い**: H-2（roadmap-seeder）が静的ドキュメントしか見ないのに対し、
こちらは **動的シグナル（コメント・ログ・失敗）** を取り込んで、人間より早く「次やるべきこと」を見つける。

**懸念**:
- 候補の質が低いとノイズ通知になる
- 人間が「お任せ感」になりすぎて意図がドリフトする

---

### X-2. デジタル・ツイン承認者 — 自分の代わりに承認する自分

**着想**: SKILL.md 行動指針 #19「マージは必ず人間が行う」を*文字通り*守りつつ、
**「人間」を学習済みデジタル・ツインに置き換える**。

**動作**:
- 過去の自分の意思決定（過去 MR の approve/reject、コメント内容、リオープン理由）を
  `ltm-use` に「judgement memory」として蓄積
- `digital-twin` エージェントがこの記憶から「あなたなら承認したか？」を予測
- 信頼度の高い予測（過去 N 件で 90%+ の一致率）のみで自動マージを実行
- 自動マージのたびに**監査ログ**を残し、人間が後から「これは違う」と覆せる
- 覆されたケースは judgement memory に「反省データ」として保存 → ツインが学習を更新

**ねらう変化**:
- B-2（レビュー・マージ）の純粋作業を 90% 削減
- 「マージは人間」原則は形式的に維持（人間の意思を投影したエージェントが代行）
- 人間は週次の「反省レビュー」で覆すべき判断だけ精査

**前段案との違い**: H-4（approval-digest）は「人間が見やすくする」だが、こちらは
「人間が見なくて済む」を狙う。自動マージ禁止ルールへの正面突破。

**懸念**:
- 自動マージは SKILL.md 設計原則と衝突 → 設計改訂を要する
- 学習データの偏りで「全部承認」になる過信リスク
- 監査責任の所在が曖昧化する

### X-3. 並列ベイクオフ — 3 ワーカーで同じイシューを焼く

**着想**: 1 ワーカーが Phase 4 で最大 5 回ループするより、
**3 ワーカーが 1 ループずつ並列実装** したほうが、多様性が確保され差し戻し率が下がる。

**動作**:
- 曖昧度 / リスクが高いイシュー（`priority:high` か `risk:high` ラベル）に対し、
  ハーネスが 3 つの isolated ブランチ（`feature/issue-{id}-attempt-{1,2,3}`）を切る
- 3 ワーカー daemon が並列で同じイシューを実装（モデルや初期スカウトマップを変える）
- 完了後、`agent-reviewer` + `council-system` で 3 結果を比較合議 → 勝者を `status:review-ready`
- 残り 2 ブランチは **LTM に「不採用設計判断」として保存**（学習データ化）
- 人間は勝者だけ見る

**ねらう変化**:
- 高難度タスクの差し戻し率 50%+ 削減
- 探索的問題（設計の正解が不明）が「お試し → ベストを選ぶ」に置換
- B-3（軌道修正）に至る前に複数案で品質保証

**前段案との違い**: 既存スキーマでは 1 イシュー = 1 ブランチ = 1 MR が原則。
これを意図的に破り、**1 イシュー = N アテンプト = 1 採用** へ拡張する。

**懸念**:
- LLM コスト 3 倍、CI 負荷 3 倍
- 「勝者」選定アルゴリズムが不当だと採用ロジック自体に不信が生まれる
- 落選者の苦労を蓄積するモチベーション設計（エージェントには関係ないが LTM 整理が必要）

### X-4. 反転リクエスター — テスト・スクリーンショットから仕様を生やす

**着想**: 人間が **自然言語で依頼する** をやめる。
代わりに「**失敗するテスト 1 本**」「**理想 UI のスクリーンショット**」「**現在のバグ画面**」を投げる。
リクエスターロールはそこから AC・設計ドキュメント・実装スコープを *逆生成* する。

**動作**:
- 入口を 3 つに固定:
  1. `tests/_inbox/*.test.ts`（失敗するテスト）
  2. `screenshots/_inbox/wanted/*.png` + `screenshots/_inbox/current/*.png`（差分が要件）
  3. `bug-reports/_inbox/*.md`（再現手順だけ）
- 反転リクエスター daemon が入口を監視 → 内容から `requirements-definer` で要件化 →
  `gitlab-idd` リクエストフローへ流す
- AC は元の成果物（テスト・スクリーンショット）への参照を埋め込む
  → ワーカーは「テストを通すこと」「スクショ差分を埋めること」が AC そのものになる

**ねらう変化**:
- 人間の「要件を書く」工数が「テストを書く」「画面を撮る」に変わる
- 仕様の曖昧さが構造的にゼロに近づく（テスト・画像は曖昧でない）
- B-1 と B-3 を同時に潰す（実行中も AC が機械検証可能）

**前段案との違い**: H-1（intent-inbox）が *自然言語の入口を増やす* だったのに対し、
これは *自然言語そのものを廃する*。形式言語（テスト）と非言語（画像）に置換する。

**懸念**:
- すべてのタスクがテスト・画像で表現できるわけではない（戦略・ADR は無理）
- テスト先生成のコストが結局人間にかかる
- スクショ差分の意味推論精度がブロッカー

### X-5. 常設組織モデル — gitlab-idd を「会社」にする

**着想**: 役職を毎セッション切り替える今のモデルをやめ、
`multi-agent-shogun-kiro` のように **永続的な役職別 daemon** を立てる。

**動作**:
- 役職 daemon を 5 つ立てる（各々独立 kiro-loop / 独立 LTM）:
  - **PM**（旧リクエスター）: 要件化・優先順位付け・スプリント管理
  - **Tech Lead**（旧レビュアー）: 設計レビュー・アーキテクチャ判断
  - **Worker** ×N（旧ワーカー）: 実装
  - **QA**（新規）: テスト追加・回帰検査・MR の独立検証
  - **SRE**（新規）: CI / 依存パッケージ / 環境管理
- 役職間通信は GitLab イシューラベル + 専用コメントタグで行う（既存スキーマ拡張）
- 人間の役割は **「CEO」**: 月 1 回の方針決定・北極星更新のみ
- ペルソナごとに固有 LTM を持つ → 過去判断との一貫性が自然と保たれる

**ねらう変化**:
- ロール判定ロジック（SKILL.md 行動指針 #16）が不要になる
  → SKILL.md の単純化につながる（前提 P-2 に整合）
- 役職固定で人格が安定 → 判断のブレが減る
- B-1〜B-3 すべてを「組織内の他部署」に押し込み、人間は経営判断のみ

**前段案との違い**: 前段 H-1〜H-12 はすべて単機能ハーネス。
これは **アーキテクチャ全体の張り替え**（前段 G7「アーキテクチャの根本を問い直す」相当）。

**懸念**:
- 5 daemon を常時動かす計算コスト
- 役職間 deadlock のリスク（PM 待ち / QA 待ちの連鎖）
- 既存 gitlab-idd ユーザーへの後方互換性

### X-1. ナイト・ブリーフィング — 合議結果

#### フレーミング監査
- フレーム A: 「夜間自律ブリーフィングを導入すべきか？」（導入前提に誘導）
- フレーム B: 「人間が文章化に 30 分/日かけるコスト vs 自動下書きの誤誘導コストどちらが大きいか？」
- フレーム C: 「未解決の TODO/CI 失敗の検知漏れを朝までに表面化すべきか？」

#### ROUND 1 スタンス表明

**MELCHIOR**: 「最小実装で導入。候補は 5 件以内・スコア閾値＋理由付与必須、A/B で採択率を測る」 / 30 分→2 分は理論的に 14 倍効率化だが `requirements-definer` の下書き精度が未測定。産業平均の自動レコメンド初期採択率は 20-40%。閾値未達ならノイズが固定費化。ログ駆動で逐次調整可能な設計が合理的。 / 反証: 2 週間運用で「採択率<30%」「人間の確認時間が逆に延びる」なら撤退条件成立。 / **信頼度: 中**

**BALTHASAR**: 「人間の判断主体性を守る安全弁付きで導入」 / 影響: 開発者本人＝負担減で歓迎。見えにくい関係者＝過去イシュー起票者（古い TODO が文脈なく蒸し返される懸念）、レビュー対象者（朝 7 時に 5 件提示で朝会対応強要のプレッシャー）、深夜の電力・CI ランナー枠を奪われる他チーム。 / 3 軸: 功利＝○ / 義務論＝△（同意なき他人 DM 要約化はプライバシー疑義）/ 徳倫理＝△（「お任せ」習慣化で職人の意図形成力が萎縮） / **信頼度: 中**

**CASPER**: 「PoC 限定で導入、ただし "5〜10 件" は多すぎ・"DM/議事録スキャン" は要除外」 / 検死①: 朝 7 時に 10 件並び、人間が即決→3 週間後に方向性ドリフトに気づくが手遅れ。検死②: Slack DM スキャンで社外秘・人事ネタが公開 MR に転写→監査事故。 / リスク vs 現状維持: 採用リスク＝ノイズ蓄積・情報漏洩・CI 夜間枠飽和／不採用＝健康診断が属人化、TODO 負債が静かに腐る。 / **信頼度: 中**

#### 対立判定
**収束（条件付き導入）だが前提衝突あり** — M「精度未知・計測駆動」、C「スコープを絞る（特に DM/議事録は除外）」、B「同意とプライバシー」で前提が衝突。**不自然収束シグナル**: 3 者とも信頼度「中」で共通懸念が「ノイズ」に集中＝視野狭窄リスク。→ ROUND 2 を簡易実施。

#### ROUND 2（核心反論のみ）
- **MELCHIOR → B/C**: 「DM 除外で本当に価値が出るか？情報源を絞るなら ROI を再計算せよ。リポ内信号（TODO/MR/CI/Issue）だけで 30 分削減が達成できるかは未検証」
- **BALTHASAR → M**: 「採択率という単一指標は危険。"採択されなかった 4 件" が本来やるべき課題だった場合の機会損失指標も要る」
- **CASPER → 全員**: 「`kiro-loop`＋`issue-mailbox` が既に動いているなら、新規 daemon ではなく既存ループに "夜間集約レポート" を相乗りすべき。新規プロセス追加は障害点を増やす」

#### SYNTHESIS 統合判断
**統合方針**: **リポ内信号のみを情報源とし `kiro-loop` に相乗りする形で、朝 7 時に最大 5 件・各候補に理由とリンクを添えたブリーフィングを 2 週間 PoC で導入。DM/議事録は次フェーズ判断。**

**収束点**:
- B-1 時短は価値が高く、健康診断の強制実施という二次効果も大きい
- 候補数は 5 件上限、各候補に「なぜ今これか」の理由と根拠リンクを必須
- 既存 `kiro-loop` に相乗りし、新規 daemon を増やさない

**残った対立**: DM/議事録スキャンの可否（C 強反対 vs M 要 ROI 再計算）→ PoC 期間中は除外で開始、別途同意設計を経て再評価

**推奨アクション（ToDo）**:
1. `kiro-loop` 内に夜間集約ジョブ追加。情報源はリポ内のみ（TODO/FIXME diff・MR コメント残・CI 失敗・既存 Issue）
2. 各候補に `score` / `reason` / `source_links` / `estimated_effort` フィールドを必須化、下位は朝配信から除外
3. KPI は採択率 / 人間の確認時間 / 「採択されなかった候補の 30 日後ステータス」の 3 軸
4. 2 週間 PoC 後にレビュー。撤退条件: 採択率<30% or 確認時間>5 分が 3 日連続
5. DM/議事録スキャンは別 RFC に切り出し、参加者の事前同意 UI 設計を経て PoC

**信頼度**: **中** — 効率化の理論値は明確だが `requirements-definer` 下書き精度が未実測

#### Pre-mortem
1. **「お任せドリフト」**: 6 ヶ月後、メニューから選ぶだけになり、人間発案のリファクタや戦略課題が消滅。リポは綺麗だが方向性が失われる
2. **健康診断の儀式化**: 採択率が安定して下がるが惰性で続き、本質的負債（アーキテクチャ・組織）を覆い隠す
3. **夜間スコープ拡張の事故**: PoC 成功後に DM/議事録を後付け追加した瞬間、機密情報が Issue 化されて監査インシデント

**追加すべき安全弁**:
- 月 1 で「メニュー無し日」を設けて人間発案を強制
- 採択率 KPI に加え「人間起案イシュー数」も並列モニタ
- 情報源拡張は新規 RFC＋人間同意なしには不可とする設定フラグ
- 配信前に dry-run 差分を 1 日プレビューする期間を設ける

### X-2. デジタル・ツイン承認者 — 合議結果

#### フレーミング監査
- フレーム A: 「人間の判断を高精度に模倣すれば自動マージしてよいか」
- フレーム B: 「人間の権威・最終責任を撤廃してよいか」

#### ROUND 1 スタンス表明

**MELCHIOR**: **反対** — 完全自動マージは導入せず「予測表示＋人間 1 クリック承認」に留める / 90% 一致率は基準率無視。残り 10% が低リスク MR とは限らず、影響度加重後のリスク期待値は線形ではない。判断メモリは過去分布に過適合し、新規パターン（破壊的変更・依存更新）で外挿失敗する。WYSIATI 警戒 / 反証: 実 MR でシャドウ運転 6 ヶ月、覆され率<2% かつ重大欠陥 0、ならば限定領域で再評価可 / **信頼度: 中**

**BALTHASAR**: **条件付き反対** — 「人間の意思を投影」と言うが、それは本人同意なき代理行使で尊厳と責任の所在を侵食 / 影響: ①承認者本人＝説明責任を負わされる影、②他開発者＝AI に裁かれる感覚で心理的安全性低下、③下流ユーザー＝事故時の救済経路不明瞭、④監査人＝声が聞かれていない / 功利: 効率は上がるが信頼資本は減耗 / 義務論: 行動指針 #19 と Kantian「人を手段化するな」に違反 / 徳倫理: 「人間マージ」は単なる作業でなく組織の誠実さの表明 / **信頼度: 高**

**CASPER**: **反対** — 自動マージは封印、自動化対象は「コメント・ラベル・承認下書き」までに留める / 検死①: 反省ループは事故後にしか回らず、prod 障害 1 件で年次効率改善を吹き飛ばす。検死②: 90% 一致率は「簡単 MR が母集団に多い」だけで、難 MR でも同精度と誤認 → サンプリングバイアス / 現状維持コスト: B-2 純粋作業は確かに痛点。ただし「下書き承認＋人間 1 秒押下」で工数 8 割削減は達成可、自動マージは限界利得 2 割のため不釣り合い / **信頼度: 高**

#### 対立判定
**収束（3 者とも自動マージ反対）**。不自然収束チェック: 前提衝突なし、バイアス方向は「悲観系（損失回避・現状維持）」に偏る可能性あり → 警戒シグナル軽度点灯。ROUND 2 を簡易実施。

#### ROUND 2（簡易・要旨）
- **MELCHIOR**: BALTHASAR の倫理論はやや感情先行だが、行動指針 #19 は権威ある独立規範で論理的にも安全弁。CASPER の検死①に同意 — 反省は事後学習で先行防御にならない
- **BALTHASAR**: CASPER の「下書き承認」案を支持。MELCHIOR の「シャドウ運転」も尊厳侵害なし。ただし学習データ収集自体への本人同意プロセスは必須
- **CASPER**: 現状維持バイアス自己点検済 — 「人間 1 秒押下」案はそれ自体が改善で、現状維持ではない。自動マージは限界利得に対し尾部リスクが非対称

#### SYNTHESIS 統合判断

### 3.4 推奨フェーズロードマップ（修正版）


### 3.5 認識論的謙虚さ

本ドキュメントの合議は同一 LLM ベースの疑似的多様性に基づく内的ロールプレイで構築された。
独立した知性による合議ではないため、本質的なバイアス（楽観・確証・形式化への偏愛）が
完全には排除されていない可能性がある。

**この判断を変える可能性がある情報**:
- `requirements-definer` の実下書き精度の実測値（X-1 の根拠が変わる）
- `shogun-kiro` の運用コスト・deadlock 発生率の実測値（X-5 の根拠が変わる）
- 既存 gitlab-idd 利用者の利用パターン分布（軽量利用層の規模次第で X-5 の評価が変動）

**最終判断はユーザー自身が行うこと**。本ドキュメントは選択肢の整理と各選択肢の事前検死を提供したものに過ぎない。
