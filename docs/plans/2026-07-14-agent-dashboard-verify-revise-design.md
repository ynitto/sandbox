# Agent Dashboard 検証コマンド修正・再実行 UI 設計

> **由来**: 対応する `docs/plans/*kiro-projects-viewer*` を置換せずクローンし、
> `agent-dashboard` / `agent-project` / `agent-flow` 名称へ改称した系統。
> 改称方針: [`docs/designs/agent-tools-rename-design.md`](../designs/agent-tools-rename-design.md)。

## 背景

途中で失敗したタスクについて、要対応タブのコメント欄は次の試行への指示しか送れず、
タスクに保存された `verify` を変更できない。一方、タスク詳細の `revise` は `verify` を変更できるが、
要対応画面から発見しにくく、プロジェクト全体の再計画と混同されやすい。

既存の agent-project には、blocked タスクへ `revise` を適用すると `verify` を置換し、needs を解消して
タスクを新しい試行として積み直す契約がある。この契約を要対応画面から直接利用する。

## 目的

- 失敗内容を確認している場所から、そのタスクの検証コマンドだけを修正できる。
- タスク分解、依存関係、既存成果物を維持する。
- 古い run を変更せず監査履歴として残し、新しい試行を開始する。
- 概要から詳細へ深掘りする既存の情報階層を維持する。

## 採用案

要対応詳細の blocked 項目に、折りたたみ式の「検証コマンドを変更」を追加する。
関連タスクの現在の `verify` を初期値として表示し、変更後は既存の
`kiro:action` / `runAction(action=revise)` へ `fields.verify` と任意の feedback を送る。

### 比較

| アプローチ | 実装コスト | リスク | 保守性 | 拡張性 | 学習コスト | 推奨度 |
|---|---|---|---|---|---|---|
| 要対応詳細から既存 `revise` を呼ぶ | 低 | 低 | 高 | 中 | 低 | ★★★ |
| タスク詳細の修正画面へ遷移する | 低 | 中 | 中 | 低 | 中 | ★★☆ |
| 検証専用の新コマンドを追加する | 高 | 中 | 低 | 高 | 高 | ★☆☆ |

## UI

blocked の要対応詳細では、従来の短い失敗理由、判断材料、全出力への導線を先に表示する。
対応操作の中に「検証コマンドを変更」を置き、開いた場合だけ以下を表示する。

- 現在の検証コマンド
- 新しい検証コマンドの入力欄
- 任意の補足指示
- 「変更して再実行」ボタン

送信前の確認画面には旧コマンドと新コマンドを並べ、次を明示する。

- タスク分解はやり直さない。
- 完了済み成果物と依存関係は維持する。
- 古い run は履歴に残し、新しい試行を開始する。

関連タスクが見つからない要対応、milestone、plan-review ではこの操作を表示しない。

## データフロー

1. needs の `taskId`、無ければ needs ID から backlog のタスクを解決する。
2. UI で現在の `task.verify` を初期表示する。
3. 変更があることを検証する。空文字は検証コマンドの削除という既存 `revise` 契約を維持する。
4. `api.runAction({ action: 'revise', id: task.id, fields: { verify }, feedback })` を呼ぶ。
5. actions 層は既存の commands/CLI 経路で agent-project へ届ける。
6. agent-project が blocked タスクを ready に戻し、needs を解消し、rev 世代を上げて新規 run を作る。
7. Viewer を再読込し、送信済み状態または新しいタスク状態を表示する。

## エラー処理

- 関連タスクを解決できない場合は操作を出さない。
- 新旧コマンドが同一で補足指示も無い場合は送信しない。
- 送信失敗時は入力内容を保持し、再試行できるようにする。
- 連打を防ぐため送信中はボタンを無効化する。
- `revise` 取り込み前は needs が残り得るため、送信済み表示で重複操作を防ぐ。

## テスト

- needs から関連 backlog タスクを解決できること。
- blocked 詳細にだけ検証コマンド変更 UI が出ること。
- `fields.verify` と任意 feedback が `revise` として送られること。
- 同一値・関連タスクなし・二重送信を拒否すること。
- agent-project の既存契約により blocked → ready、needs 解消、rev 世代更新となること。
- デスクトップとモバイルの要対応詳細から操作できること。

## 既存スキルとの関係

- `test-driven-development`: 既存 `revise` 契約を壊さずUI接続するため、ペイロードと表示条件のテストを先に追加する。
- `systematic-debugging`: 実装中に状態遷移や二重送信の不具合が出た場合のみ利用する。
- 既存機能との重複: タスク詳細の `revise` と操作契約を共有し、バックエンド機能は新設しない。
