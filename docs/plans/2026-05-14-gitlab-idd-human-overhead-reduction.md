# gitlab-idd ハーネス案 — 人的工数の削減（要件インプット・レビュー・軌道修正）

> 作成日: 2026-05-14
> 対象ブランチ: `claude/plan-agent-workflow-tools-Hks7o`

## 1. 背景

`.github/skills/gitlab-idd` を中核に、複数ノードが daemon CLI（`tools/kiro-loop` 等）を常時起動して、
GitLab イシューを非同期キューとしたイシュー駆動の自律開発体制が成立しつつある。
ワーカー／レビュアー／リクエスターの各ロールはプロンプトトリガーで起動し、
イシュー作成 → 実装 → MR → レビュー → 承認の往復が分散実行されている。

一方で、フローを通すために **人間が必ず介在する地点** が依然として残っている。
作業内容が「コードを書く」から「イシューを書く・読む・直す」に変わっただけで、
人間 1 人あたりに発生する時間工数は本質的には削減されていない。

本ドキュメントは、現状のリポジトリ資産（`gitlab-idd` スキル本体・既存ツール群・各種スキル）と
`docs/plans/2026-05-09-gitlab-idd-improvement-unified.md` で整理済みの改善ロードマップを
踏まえたうえで、**ハーネス／スクリプト層**（スキル本体には手を入れない外部レイヤー）で
何を作れば人的工数が削減できるかを案出する。

---

## 2. 現状の整理：人間が残っている 3 つの地点

`.github/skills/gitlab-idd/SKILL.md` と各 reference を踏まえ、daemon が回り始めても
人間の時間が圧迫される箇所を 3 つに分類する。

| # | 地点 | 何が起きているか | 根因 |
|---|---|---|---|
| **B-1** | 自律起動前の **要件インプット** | 人間が依頼を文章化 → リクエスター起動 → 分割案を確認 → 承認 → イシュー化 のループに張り付く | 入力チャネルがプロンプトのみ。要件の発想・整理・確認も人間頼み |
| **B-2** | イシュー実行後の **レビュー／マージ** | `status:review-ready` → エージェントが評価 → `status:approved` まで自動だが、**マージは必ず人間** （SKILL.md 行動指針 #19）。承認済 MR の差分確認とマージは人間 | マージ権限の安全弁。承認済の山を「捌く」コストが残る |
| **B-3** | **軌道修正**（設計意図からの逸脱） | ワーカーが Phase 4 ループ内で勝手に AC/スコープを更新（III-1 ライブ仕様書）→ レビューで初めて検知 → 差し戻し or 設計事故 | 実装中の差分を人間が監視するチャネルがない |

> いずれも `gitlab-idd` スキル自体の責務範囲外。スキル側で抱え込むと SKILL.md が肥大化する
> （前提 P-2「単純化とセットでないと拡張できない」に抵触）。
> よって **オーケストレーション外のハーネス層** に切り出すのが妥当。

## 3. 既存資産の棚卸し

新規実装に走る前に、既存ツールで再利用できるものを確認する。

| 資産 | 用途 | 拡張余地 |
|---|---|---|
| `tools/issue-mailbox/issue-mailbox.py` | GitLab イシューの差分を tmux ペインに通知（CLI agent への自動入力にも対応） | ラベル変化・コメント追加・MR 状態のトリガー化、複数チャネル（音声・モバイル）への分岐 |
| `tools/kiro-loop/kiro-loop.py` | tmux 内で daemon CLI を常時起動。定期プロンプト送信 | 「特定イシューの状態」をトリガーにした即時プロンプト送信 |
| `tools/multi-agent-shogun-kiro` | 複数エージェントの役職階層（Shogun / Karo / Ashigaru / Gunshi） | 「軌道修正」専任の Gunshi 役を gitlab-idd に転用 |
| `.github/skills/gitlab-idd/scripts/gl.py` | GitLab API ラッパー（issue, comment, MR, label, pipeline） | webhook / SSE 化、差分検知の API 化 |
| `.github/skills/requirements-definer` | 曖昧依頼の要件化 | 音声・チャットからの直接入力フックの噛ませ先 |
| `.github/skills/agent-reviewer` / `council-system` | 多観点レビュー／合議 | 軌道修正の「第二の目」 |
| `.github/skills/ltm-use` | 長期記憶 | 「過去に同じ逸脱パターンがあった」を検知して警告 |

**観察**: B-1 / B-3 のための入出力チャネル（音声・モバイル・チャット・ファイルドロップ）が現状の
資産には存在しない。`issue-mailbox` は GitLab → 人間／CLI 方向のみで、逆向きが弱い。
ここがハーネス層の中心的なギャップ。

## 4. ハーネス／スクリプト案

各案は次の評価軸で粗付けする。

- **インパクト**: 削減できる人的工数の大きさ（★1〜3）
- **コスト**: 実装＋運用立ち上げの重さ（低 / 中 / 高）
- **依存**: 必要な前提（既存スキル・外部システム・スキーマ変更等）
- **対応ボトルネック**: B-1 / B-2 / B-3 のどれを潰すか

### 4.1 B-1（要件インプット）系

#### H-1. `intent-inbox`（マルチチャネル要件受付ハーネス）

人間が「依頼を整理して文章化する」ステップを丸ごと省くチャネル統合ハーネス。
入口を増やし、すべて `requirements-definer` → gitlab-idd リクエスターへと自動接続する。

- **入口**:
  - Slack / Teams DM・特定チャンネル（既存 `teams-use` スキル流用）
  - Outlook メール件名タグ（既存 `outlook-use` スキル流用）
  - 音声メモ（Whisper / OS 標準音声入力）→ テキスト化
  - `~/intent-inbox/` ディレクトリへの `.md` / `.txt` ドロップ
- **処理**: ハーネスが新着を検知 → kiro-loop の既存セッションに
  「以下の依頼を `requirements-definer` で整理し `gitlab-idd` リクエスターで投稿してください: …」を流す
- **承認の省略**: 信頼ラベル `auto:approve` 付きの入口（自分の DM 等）は中間確認を省き、
  分割数 ≤ N かつ XL タスクなしのときのみ自動投稿、それ以外は承認待ち通知のみ
- **インパクト**: ★★★ / **コスト**: 中 / **依存**: `requirements-definer`, `teams-use`, `outlook-use`
- **対応**: B-1

#### H-2. `roadmap-seeder`（ロードマップ → イシュー自動播種）

リポジトリの `docs/plans/*.md` や `docs/designs/*.md` の「未着手項目」を定期スキャンして、
バックログイシューとして自動播種する。すでに `gitlab-idd-improvement-unified.md` のような
案出ドキュメントが豊富にあるため、ここから直接イシューを生やせるだけで人間の介在が大きく減る。

- **動作**: 週次／日次の cron で kiro-loop に投げる定期プロンプト
- **重複防止**: 過去 GitLab イシュータイトルとの類似度（embedding / 文字列）でフィルタ
- **粒度制御**: `priority:low` で投稿 → ワーカーが空いたときだけ拾う
- **インパクト**: ★★ / **コスト**: 低 / **依存**: なし
- **対応**: B-1

#### H-3. `clarification-autoresponder`（needs-clarification 自動応答）

ワーカーが `status:needs-clarification` に落としたイシューを、リクエスター起動を待たずに
**コードベース + LTM** から推論で自動補完する補助デーモン。

- **動作**: `issue-mailbox` の拡張として、`status:needs-clarification` の出現を webhook 風に検知
- **応答**: `Explore` 相当のサブエージェントを起動 → 関連ファイル走査 → 推測解釈をコメント
- **安全弁**: 自動応答は **コメント追記まで** に留め、`status:open` への戻しは人間 or リクエスター daemon に委ねる
- **インパクト**: ★★ / **コスト**: 中 / **依存**: `ltm-use`（あると精度向上）
- **対応**: B-1（部分的に B-3 にも効く）

### 4.2 B-2（レビュー・マージ）系

#### H-4. `approval-digest`（承認済 MR ダイジェスト）

`status:approved` ラベル付きイシュー（= 人間のマージ待ち）を集約し、
1 日 1〜2 回、人間に「マージしてよい候補リスト」を要約付きで通知する。

- **要約**: MR タイトル / 受け入れ条件 ↔ diff 対応表 / CI 状態 / リスク評価（ファイル数・行数・タッチした領域）
- **配信**: Slack / Teams / メール / モバイル通知（push）
- **アクション**: 通知から 1 タップでマージ（GitLab API の `merge` を叩く Web エンドポイントを薄く用意）
- **注意**: SKILL.md 行動指針 #19「マージは必ず人間が行う」を維持。
  自動マージはしない。あくまで「人間の意思決定をスマホ 1 タップに圧縮」する案
- **インパクト**: ★★★ / **コスト**: 中 / **依存**: 通知配信先（既存の `teams-use` / `outlook-use` で代替可）
- **対応**: B-2

#### H-5. `merge-risk-classifier`（マージ前リスク分類）

`status:approved` の MR ごとに、**人間のレビュー深度を変えるための事前分類**を行う。

- **分類軸**:
  - 変更ファイル数 / 変更行数（既存 S/M/L/XL）
  - 触れた領域（CI 設定、auth、DB、依存パッケージ追加）
  - テストカバレッジの増減
  - past LTM の「同種変更で事故った率」
- **出力**: イシューに `risk:low|mid|high` ラベルを自動付与し H-4 の通知に色付け
- **波及効果**: 人間は `risk:low` を流し読み・`risk:high` を熟読 という時間配分が可能になる
- **インパクト**: ★★ / **コスト**: 中 / **依存**: なし（CI が無くても動く）
- **対応**: B-2

#### H-6. `second-eye-council`（二の目合議）

リクエスター承認の前後に、**別ノード（別モデル）** による独立レビューを必ず噛ませる。
既存 `council-system` スキルを呼び出す薄いラッパー。

- **目的**: 自分発行イシューを自分でレビューする（self-review）バイアスの低減
- **動作**: `status:review-ready` 検知 → `council-system` で 2〜3 視点の合議 →
  分岐結果をコメント → 合議で LGTM がそろった時のみ `status:approved` へ
- **既存との関係**: `agent-reviewer` の並列観点と重複する部分があるため、
  `council-system` は「観点合意の SYNTHESIS」だけを担当（観点自体は agent-reviewer）
- **インパクト**: ★★ / **コスト**: 低（スキルは既存） / **依存**: `council-system`
- **対応**: B-2（事故率低下経由で人間負荷を削減）

### 4.3 B-3（軌道修正）系

#### H-7. `drift-watch`（実装中の意図逸脱検知）

WIP（`status:in-progress`）イシューを定期スキャンし、**ワーカーが書き換えたイシュー本文**
（III-1 ライブ仕様書）と元の本文との差分を取り、
意味のある変更が起きたら **リクエスター daemon に即時プロンプト送信**して人間判断を仰ぐ。

- **トリガー**: イシュー本文の更新イベント、`scout-map` コメント投稿、
  `## 実装スコープ` / `## 受け入れ条件` 行のテキスト変化
- **判定**: 差分を LLM に渡して「設計意図に対する逸脱度」を 0〜1 で採点（閾値 ≥ 0.5 で警告）
- **アクション**:
  - 通常: イシューにコメント `⚠️ 仕様変更を検出` を残してリクエスターに通知
  - 高逸脱: kiro-loop 経由でリクエスターセッションに即プロンプトを送り
    「イシュー #N の仕様変更を確認してください」を投入
- **波及効果**: 「レビュー時に初めて気づいて全差し戻し」を避けられる
- **インパクト**: ★★★ / **コスト**: 中 / **依存**: `issue-mailbox` 拡張
- **対応**: B-3

#### H-8. `compass-prompt`（北極星リマインダー）

`tools/multi-agent-shogun-kiro` の `north_star` 概念を gitlab-idd に転用。
プロジェクトルートに `NORTH_STAR.md` を置き、ワーカー Phase 4 実装ループの最初に
ハーネスが「現在の WIP がこの北極星から逸れていないか」を 1 行で再確認させる。

- **設置場所**: ハーネス側で worker-role.md 直前に prepend する形のプロンプト注入
  （SKILL.md 本体は変更しない）
- **頻度**: Phase 4 のループ毎回ではなく、ループ N 回ごとに 1 回
- **インパクト**: ★ / **コスト**: 低 / **依存**: なし
- **対応**: B-3

#### H-9. `scope-creep-guard`（サイズ膨張ガード）

実装中のブランチ diff を定期取得し、当初の S/M/L 見積もりを超えたら自動で
**派生イシューへの分割提案コメント**を投稿する。

- **動作**: WIP ブランチの `git diff --shortstat` を 1 時間ごとに取得し、
  受け入れ条件と照らして「スコープ外の変更が含まれていないか」を LLM に評価させる
- **アクション**: 膨張検知時はワーカー daemon に「派生イシュー化を検討してください」を送る
  または `priority:high` で派生イシュー化スキルを起動
- **インパクト**: ★★ / **コスト**: 中 / **依存**: gl.py + ローカル clone
- **対応**: B-3（前段で B-2 の差し戻し率も下がる）

### 4.4 横断系

#### H-10. `operator-console`（運用ダッシュボード）

人間が「何が今走っていて、何が自分の判断待ちなのか」を 1 画面で把握できる TUI（または最小 Web UI）。

- **画面要素**:
  - ノード一覧（誰の kiro-loop が生きているか、最終アクティビティ）
  - WIP イシュー（status:in-progress, 経過時間, ワーカー node_id, 直近コミット）
  - レビュー待ち（status:review-ready）
  - 承認済（status:approved、＝マージ待ち）
  - drift-watch / scope-creep-guard の警告キュー
- **アクション**: 各行から「マージする」「リオープン」「@self に振り直し」を即時実行
- **データソース**: `gl.py` + 各ハーネスのログ
- **インパクト**: ★★★ / **コスト**: 中〜高 / **依存**: なし
- **対応**: B-1〜B-3 すべて（人間の状況把握コストを直接削減）

#### H-11. `webhook-bridge`（GitLab webhook → kiro-loop）

`issue-mailbox` のポーリングを **GitLab webhook** に置き換えるブリッジ。
レイテンシ削減 + API レート消費削減。

- **動作**: 軽量 HTTP サーバ（FastAPI 等）が webhook を受信 → 内容を分類 →
  既存ハーネス（drift-watch / clarification-autoresponder / approval-digest）に分岐
- **配置**: 各ノードのローカル or 共有 VM のどちらでも可
- **副次効果**: H-4 の通知や H-7 の警告が「数分〜数十分遅れ」から「数秒以内」になる
- **インパクト**: ★★ / **コスト**: 中 / **依存**: 公開 IP or トンネル（cloudflared 等）
- **対応**: B-1〜B-3 横断

#### H-12. `meta-loop`（ハーネス自身のメタオーケストレーション）

上記 H-1〜H-11 はそれぞれ独立 daemon にすると管理が辛い。
`kiro-loop` のワークスペース機構を流用して、**ハーネス自体も kiro-loop の管理対象**にする。

- **構造**: `kiro-loop.yaml` の `prompts:` に各ハーネスを定期プロンプトとして登録
  - 例: 「issue-mailbox を 5 分ごとに駆動」「drift-watch を 15 分ごとに駆動」
- **副次効果**: タスクスケジューラ／systemd を増やさず、既存の運用に乗る
- **インパクト**: ★（土台） / **コスト**: 低 / **依存**: 既存 `kiro-loop`
- **対応**: 運用基盤

## 5. 優先度（短期で着手すべき順）

| 優先度 | 案 | 理由 |
|---|---|---|
| **P0** | **H-4 approval-digest** + **H-5 merge-risk-classifier** | 「承認済を捌く時間」が最も読みやすく、効果が大きい。スキル本体に影響しない |
| **P0** | **H-7 drift-watch** | 軌道修正の遅延は B-3 の中核損失。レビュー差し戻し率の改善にも直結 |
| **P1** | **H-1 intent-inbox**（最小版: Slack/Teams DM のみ） | 要件入口を 1 つ増やすだけでも工数削減効果が大きい |
| **P1** | **H-10 operator-console**（最小版: TUI） | 他ハーネスの可視化先になるため、追加実装の動機が湧きやすくなる |
| **P2** | H-11 webhook-bridge | H-4/H-7 が動き始めてから低レイテンシ化として導入 |
| **P2** | H-3 clarification-autoresponder / H-9 scope-creep-guard | 上位案の動作下地が整ってから |
| **P3** | H-6 second-eye-council / H-8 compass-prompt / H-2 roadmap-seeder | 効果は確実だが緊急度低 |
| **基盤** | H-12 meta-loop | P0 着手と同時に kiro-loop.yaml の prompts セクションを整備 |

## 6. スキル本体との分離原則（保守性のための制約）

`docs/plans/2026-05-09-gitlab-idd-improvement-unified.md` の前提 P-2「単純化とセットでないと拡張できない」を踏まえ、
本ハーネス群は **`.github/skills/gitlab-idd` の SKILL.md・references を 1 行も増やさない** ことを設計制約とする。

- ハーネスは `tools/<harness-name>/` に独立配置する
- GitLab とのやり取りは **既存 `scripts/gl.py` を呼ぶ** だけに留める（重複実装しない）
- スキルへの「指示」が必要な場合は kiro-loop のプロンプト送信を経由する（直接フックは作らない）
- ハーネス追加 = スキル単純化、を満たさない案は採用しない

## 7. 推奨ファーストステップ（実装着手案）

最小コストで体感効果を得る順序として、次の 2 本を 1 週間スプリントで作る:

1. **`tools/approval-digest/`**
   - `gl.py` で `status:approved` を全件取得 → MR diff サマリーを LLM で生成 →
     Teams / Outlook へ HTML 表で送信。CLI 単発実行で OK。
   - kiro-loop.yaml に「1 日 2 回起動」プロンプトとして登録（H-12）

2. **`tools/drift-watch/`**
   - GitLab API の `issues/{id}/resource_state_events` と本文 diff を 15 分ごとに取得 →
     `## 実装スコープ` / `## 受け入れ条件` の文字列変化を抽出 →
     リクエスター daemon にプロンプト投入。
   - 内部は `gl.py` を再利用、追加依存は最小。

この 2 本の運用ログをもとに、H-1（intent-inbox）と H-10（operator-console）の要件を固めて
次スプリントで実装する。

## 8. 想定 KPI

| 指標 | 期待される変化 |

## 9. 残課題・未検討事項

- **責任分界**: 自動マージは禁止のままだが、`H-4 approval-digest` から 1 タップマージするスマホ UI を作る場合、
  ユーザー認証と監査ログの設計が別途必要。
- **モデル選択**: drift-watch / merge-risk-classifier の LLM 呼び出しコスト。
  Haiku 4.5 をデフォルトとして、リスク高のみ Opus へ昇格させる戦略を検討。
- **チーム間配置**: 各ノードがハーネスをローカル起動するか、共有 1 ノードに集約するか。
  P0 では「リクエスター本人のノードに集約」を推奨（権限の二重化を避ける）。
- **既存 P-3 との整合**: 「タスク実行・レビューはオーケストレーター内に抱え込まない」原則を守るため、
  H-5 / H-7 の判定ロジックは **スキル外** に置く（gitlab-idd 自体には実装しない）。
