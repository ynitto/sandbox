# Agent Dashboard: ユーザー中心UI再設計

## 背景

現在の Agent Dashboard は必要な機能を備えている一方、設計時の分類名、内部状態、免責的な補足説明が日常画面へ露出している。利用者は進捗確認や承認をしたいだけでも、run、heartbeat、lease、bus、CLI などの実装概念を読み解く必要がある。

既存の「概要から詳細へ進む」「一覧から1件を選ぶ」という情報構造は維持しつつ、主画面を日常の進捗確認と判断に集中させる。

## 対象ユーザーと目的

主な利用者は、内部実装を知らず、日々の進捗確認、承認、指示、成果確認を行う人とする。

- 今の状態と次に必要な行動を短時間で把握できる。
- 操作結果がボタンの文言から予測できる。
- 内部情報を知らなくても主要機能を利用できる。
- 開発・障害調査に必要な情報は失わず、独立したサブ画面から確認できる。

## 検討した案

## 採用設計

### 情報構造

主画面は「いま何が起きているか」「自分に何が必要か」「成果は出たか」を中心に構成する。

- **概要**: 現在の状態、次に必要な操作、最近の成果を優先する。
- **タスク**: 作業予定の確認、追加、並び替えに集中する。
- **要対応**: 利用者の判断や入力が必要な項目だけを目立たせる。
- **実行**: 進行中の作業、進捗、次に起きることを主表示する。
- **成果**: 完了した内容と変更点を中心に表示する。
- **定期・定型作業**: `Cowork` のような実装・製品都合の名称ではなく、利用目的が分かる名称を使う。

タブ構成と既存機能への到達性は維持する。データ取得、状態判定、IPC、ファイル操作は変更しない。

### 文言と説明

- ボタンは「対応する」「再開」「成果を見る」など、操作結果が分かる短い動詞に統一する。
- 内部種別名や設計用語を見出し・主要ラベルへ出さない。
- 常時表示されている免責、仕組みの説明、例外条件の列挙は削除する。
- 補足が必要な場合は、操作時のエラー、短いヘルプ、開発者ツールへ移す。
- 空状態は「問題なし」「まだ成果はありません」など、状況と次の行動だけを伝える。
- 絵文字記号を操作アイコンとして使わず、既存SVGまたはテキストラベルへ置き換える。

### 開発者ツール

内部状態と高度な設定は、プロジェクト設定から開く独立したサブ画面「開発者ツール」へまとめる。

- run ID、内部ステータス、タスクグラフ
- heartbeat、lease、リトライ履歴
- bus、lock、CLI、同期方式
- 生ログ、原文、診断情報
- GitLab連携と状態共有の詳細設定

日常画面から情報を完全に削除するのではなく、必要な利用者が一段深い画面で確認できる段階的開示とする。障害時に必要な主要操作は元の画面にも残す。

### 視覚設計

- 装飾的な大カードや過剰な色分けを減らし、余白、見出し、罫線で階層を示す。
- 強調色は主要操作と要対応状態に限定する。
- 文字サイズを一貫した段階へ整理し、本文の可読性を確保する。
- 色だけで状態を伝えず、状態名と短い説明を併記する。
- 主要操作は44px相当の操作領域と可視フォーカスを持つ。
- 375pxから1440pxで横スクロールと操作の重なりを発生させない。

### エラーハンドリング

- エラーは発生箇所の近くで、原因の要約と次に取れる操作を示す。
- 技術的な詳細は開発者ツールまたは展開可能な詳細へ分離する。
- データ未取得、対象なし、未設定をエラー扱いせず、それぞれ短い空状態として表示する。
- 既存の確認ダイアログ、トースト、危険操作の保護は維持する。

### テスト

- 既存の操作、タブ遷移、設定、診断、同期機能が維持されること。
- 日常画面から内部用語と長い免責説明が除かれていること。
- 開発者ツールから移動した情報と設定へ到達できること。
- 主要操作のラベルが結果を表し、アイコンのみの操作にアクセシブルな名前があること。
- キーボードフォーカス、見出し順、フォームラベル、状態通知を確認すること。
- 375px、768px、1024px、1440px相当でレイアウトを確認すること。
- 既存ユニットテストと Electron の起動確認を行うこと。

## 既存スキルとの関係

- `ui-designer`: 段階的開示、操作階層、アクセシビリティ、レスポンシブ設計に使用する。
- `webapp-testing`: 実装後の画面幅、主要導線、開発者ツールへの遷移確認に使用する。
- `react-frontend-coder`: 対象がプレーン Electron のため使用しない。

## 実装方針

1. 現在の表示文言を日常情報、補足、内部情報に分類する。
2. 共通レイアウト、タイポグラフィ、ボタン、状態表示を整理する。
3. 概要、タスク、要対応、実行、成果の順に主表示を簡素化する。
4. 内部情報と高度な設定を開発者ツールへ移す。
5. 既存テストを更新し、機能維持と文言・導線を検証する。
6. Electronを主要画面幅で確認し、視認性と操作性を調整する。
