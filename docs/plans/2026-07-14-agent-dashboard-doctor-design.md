# Agent Dashboard Doctor・全出力表示 UI 設計

> **由来**: 対応する `docs/plans/*kiro-projects-viewer*` を置換せずクローンし、
> `agent-dashboard` / `agent-project` / `agent-flow` 名称へ改称した系統。
> 改称方針: [`docs/designs/agent-tools-rename-design.md`](../designs/agent-tools-rename-design.md)。

## 背景

要対応タブの失敗理由は概要としては有用だが、検証出力が約 200 文字へ圧縮されるため、原因の判断に必要な全体像を失うことがある。一方、生ログを最初から展開すると、概要から必要な詳細へ進む Viewer の情報設計を崩してしまう。

また、画面に表示された状態を読み解くには agent-project / agent-flow の知識が必要であり、「何が起きているか」「次に何をすべきか」をその場で確認できる読み取り専用の助言機能が必要である。

既存の設計決定「概要ハブから詳細タブへ深掘りする」を継承する。

## 採用案

「コンテキスト Doctor + 生出力の遅延読み込み」を採用する。

### 情報階層

要対応は次の順に深掘りする。

1. 状態別件数
2. 要対応一覧
3. 選択した要対応の判断概要・要約・操作
4. 関連 run の出力全体

「出力全体を見る」を押したときだけ、選択中 needs に対応するタスクの `last_run` を特定し、run 内の各工程の出力を取得する。関連 run が無い場合は needs 原文を全表示し、追加ログが存在しないことを明示する。UI に表示する原文は省略しない。

### Doctor ボタン群

サイドバー上部の通常操作とは分けて、次の Doctor グループを一行で配置する。

- 同期を修復
- AI に相談

既存の同期修復ボタンは Doctor グループへ移す。「AI に相談」は現在のアクティブタブと選択状態を文脈として扱う。

### AI に渡す画面文脈

- プロジェクト名、稼働状態、タスク件数、要対応件数
- 現在のタブ
- 要対応タブ: 選択中 needs の概要・原文・関連 run 出力
- 実行タブ: 選択中 run、内部ビュー、選択中工程
- タスク・成果履歴タブ: 現在の絞り込みと画面上の要点

設定トークンやアクセストークンなどの秘密情報は渡さない。CLI へ渡せる長さを超える場合、表示用の全出力は維持したまま、助言文脈だけ先頭・末尾を残して制限し、その事実をプロンプトに明示する。

### 助言の出力

回答は次の構造を要求し、Markdown としてダイアログへ表示する。

1. 現在起きていること
2. 次にすること
3. 判断の根拠

Doctor は読み取り専用であり、コマンド実行、ファイル編集、状態変更を行わない。実行中はボタンを無効にし、進行状態を `aria-live` で通知する。エラーは同じダイアログ内に表示する。

## Viewer アシスタント設定

既存の charter AI 補完と Doctor は共通設定を使う。設定セクション名を「Viewer アシスタント」に変更する。

選択可能な CLI:

- `kiro`（既定）
- `claude`
- `copilot`
- `codex`
- `cursor`
- `ollama`

モデルとタイムアウトも共通とする。既存の空 CLI 設定は `kiro` として扱い、既存ユーザーの設定を壊さない。Ollama はモデル未指定時に入力を促すエラーを返す。

Doctor 実行では各 CLI の読み取り専用設定を強制する。

| CLI | Doctor 実行方針 |
|---|---|
| kiro | `--trust-tools=` でツールを許可しない |
| claude | `--permission-mode plan`、セッション非永続 |
| copilot | 利用可能ツールを空に制限 |
| codex | `exec --sandbox read-only --ephemeral` |
| cursor | `--print --mode ask` |
| ollama | `ollama run <model>`、プロンプトを標準入力 |

charter 補完は既存挙動を維持しつつ、同じ 6 CLI のコマンド構築へ拡張する。

## コンポーネントとデータフロー

1. Renderer が現在画面の構造化コンテキストを作る。
2. 関連出力が必要なら既存 flow API で run 全体を遅延取得する。
3. Preload 経由で `agent:doctor` IPC を呼ぶ。
4. Main の agent 層が共通設定を解決し、Doctor 用の読み取り専用コマンドを構築する。
5. 応答本文だけを Renderer へ返し、Doctor ダイアログへ表示する。

ファイル書き込み API や既存のタスク操作 API は Doctor から呼ばない。

## エラーハンドリング

- CLI が未導入: CLI 名と PATH 確認を案内する。
- 認証切れ・モデル不明: stderr の末尾を安全な長さで表示する。
- タイムアウト: 設定時間と再実行方法を表示する。
- 関連 run が無い: needs 原文だけを表示し、Doctor もその範囲で助言する。
- 連打: 実行中の再実行を禁止する。
- タブや選択が変わった場合: 実行開始時のスナップショットに対する回答であることを表示する。

## レスポンシブ・アクセシビリティ

- 375px では Doctor グループを横幅内で均等配置する。
- 出力全体は横スクロール可能な等幅領域とし、ページ全体へ横スクロールを漏らさない。
- ボタンには可視フォーカスと 44px 相当の操作領域を確保する。
- 実行中・成功・失敗を色だけに依存せずテキストでも表す。
- `prefers-reduced-motion` を尊重する。

## テスト

- CLI ごとのコマンド構築と読み取り専用フラグ
- CLI 設定の既定値・既存設定互換・Ollama モデル必須
- needs から関連 `last_run` を解決し、全工程の出力を省略せず返すこと
- 関連 run が無い場合の needs 原文フォールバック
- 現在タブ・選択対象から Doctor 文脈を構築すること
- Doctor の多重実行防止とエラー表示
- 1440 / 1024 / 768 / 375px で水平オーバーフローがないこと
- 既存の要対応操作、charter 補完、同期修復が退行しないこと

## 却下案

### Doctor 専用タブ

診断情報は集約できるが、現在見ている対象から離れるため文脈を失いやすい。概要から詳細へ進む既存設計とも重複する。

### 要対応カードだけの AI ボタン

実装は小さいが、実行タブや概要タブで同じ疑問が生じたときに使えず、「現在開いている画面を相談する」という目的を満たさない。
