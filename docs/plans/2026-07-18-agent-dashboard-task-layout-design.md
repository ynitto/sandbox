# Agent Dashboard: タスク画面の固定レイアウト設計

## 背景

現行のタスク画面は、ID、タイトル、状態、優先度、再試行、検証、属性を横長のテーブルへ同時に表示している。タイトルや属性の長さ、バージョン数、画面幅によって列幅と操作位置が変わり、日常的な進捗確認に不要な内部情報も一覧へ露出している。

既存のタスク詳細ダイアログをサブ画面として活用し、メイン画面は一覧で判断するための情報だけに絞る。

## 目的

- タスクの状態、内容、優先度、次の行動を短時間で把握できるようにする。
- 詳細情報は既存のタスク詳細ダイアログで確認できるようにする。
- データ件数、文字量、状態、バージョン数が変わっても一覧の骨格を維持する。
- 375pxから1440pxまで横スクロールや操作の重なりを発生させない。
- 既存のフィルター、タスク追加、再計画、詳細操作、IPCを維持する。

## 採用設計

### 情報構造

メイン画面の各タスクには次の4項目だけを表示する。

1. **状態** — 日常利用者向けの短い状態名。色だけに依存せず文字を併記する。
2. **タスク名** — 一覧では最大2行に制限し、全文は詳細ダイアログで表示する。
3. **優先度** — 数値と短い重要度表現を固定幅で表示する。
4. **次の行動** — 完了までに何が起きるか、または利用者が何をすべきかを1行で表示する。

ID、再試行回数、検証コマンド、属性、依存関係、実行履歴、ファイルパスはメイン一覧から外し、既存のタスク詳細ダイアログへ集約する。関連実行への導線も詳細ダイアログに維持する。

### 固定レイアウト

- デスクトップは `状態 / タスク / 優先度 / 次の行動 / 詳細導線` の固定グリッドを使用する。
- 状態、優先度、詳細導線は固定幅、タスク名と次の行動は比率指定と最小幅を持たせる。
- 長文は省略し、セルの高さは最大2行を基準に固定する。全文を失わず詳細ダイアログで確認できるようにする。
- フィルター行と主要操作は一覧とは別のツールバーに置き、データ量に左右されない位置を保つ。
- バージョンフィルターが増えた場合は専用行で折り返し、主要操作を押し流さない。
- 空状態、読み込み中、0件のフィルターにも一覧と同じ領域を確保し、更新時のレイアウトシフトを抑える。

### レスポンシブ

- 768px未満では同じ情報順序を保った固定カードへ切り替える。
- モバイルカードも状態、タスク名、優先度、次の行動だけを表示し、詳細内容を重複させない。
- タスク選択は行またはカード全体で行えるようにし、44px以上の操作領域を確保する。
- 横スクロールは導入しない。

### ナビゲーションと操作

- 行またはカードを選択すると、既存のタスク詳細ダイアログを開く。
- 一覧要素はbutton相当のキーボード操作を可能にし、可視フォーカスを付ける。
- 行内に複数の競合する操作を置かず、詳細画面への入口を明確にする。
- 「タスクを追加」「計画を作り直す」はツールバー右側の主要操作として維持する。
- フィルターと件数はツールバー左側にまとめる。

### データと状態

表示は既存の `state.project.backlog`、`state.project.archive`、`state.backlogFilter`、`state.backlogCharter` を使う。新しい永続データやIPCは追加しない。

一覧用の表示モデルを純関数へ分離し、状態表示、優先度表示、次の行動、関連runの有無を一箇所で組み立てる。文字量や欠損値がDOM構造を変えないよう、各項目は常に同じ要素を出力する。

### エラーハンドリング

- タスクがない場合は、現在のフィルター条件と次に可能な操作を短文で示す。
- 欠損したタイトル、優先度、状態は安全な代替表示を使い、列自体は削除しない。
- 更新で件数が変化しても、ツールバーと一覧ヘッダーの位置を維持する。
- 詳細対象が更新で消えた場合は既存どおりダイアログを開かず、次の更新後も一覧を利用可能にする。

### アクセシビリティ

- 状態を色だけで表現しない。
- 行またはカードにアクセシブルな名前を付け、Enter/Spaceで詳細を開けるようにする。
- フィルターの選択状態を `aria-pressed` で示す。
- 省略した本文は詳細ダイアログで完全に読めるようにする。
- `:focus-visible` と `prefers-reduced-motion` の既存方針を維持する。

### テスト

- 一覧表示モデルが状態、優先度、次の行動を正しく生成すること。
- ID、再試行、検証、属性がメイン一覧に表示されず、詳細ダイアログには残ること。
- 長いタイトル、長い次の行動、欠損値でも同じDOM構造になること。
- フィルター、バージョン絞り込み、追加、再計画、詳細ダイアログが既存どおり動くこと。
- 375px、768px、1024px、1440pxで横スクロールとレイアウトシフトがないこと。
- 既存の `npm test` が成功すること。

## 既存スキルとの関係

- `ui-designer`: 段階的開示、長文の省略と展開、モバイルでのカード化、非同期更新時の領域確保を反映した。
- `webapp-testing`: 実装後の画面幅、フィルター、キーボード操作、詳細ダイアログ遷移の確認に利用する。
- `react-frontend-coder`: 対象がプレーンElectronのため使用しない。

## 実装計画

1. タスク一覧の表示モデルとHTML生成関数を純関数として追加し、表示項目を固定する。
2. `renderBacklog` を固定ツールバーと要約一覧へ置き換え、既存イベントを接続する。
3. デスクトップの固定グリッドとモバイルカード用CSSを追加する。
4. 一覧のアクセシビリティ属性とキーボード操作を追加する。
5. 表示モデル、情報分離、既存導線のテストを追加・更新する。
6. 全テストと主要画面幅の表示を検証する。
