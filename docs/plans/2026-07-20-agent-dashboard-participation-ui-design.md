# Agent Dashboard 参加UI設計

- 日付: 2026-07-20
- 状態: 承認済み
- 対象: `agent-project` から起動された `agent-flow` と `agent-amigos`
- 関連: `2026-07-19-delegation-contract-design.md`

## 目的

利用者が内部の委譲契約や実行方式を理解しなくても、現在参加できる仕事を見つけ、
このPCの設定済みエージェントを参加させられるようにする。

独立した旧「委譲」画面が扱っていた公示・落札・受け入れ・中止は復活させない。
公示と受け入れはミッション／要対応、進行状況は実行に残し、新しい画面は参加だけを扱う。

## 採用案

目的別の独立タブ「参加」を追加する。`agent-flow` と `agent-amigos` の募集中項目を
同じカード一覧へ正規化し、主操作を「参加する」1つに限定する。

タブは参加可能な項目がある場合に表示する。参加可能な項目がない場合にタブを開いているときは、
「現在参加できる仕事はありません」と表示する。

## 画面構成

各カードに表示する情報は次に限定する。

- 仕事またはミッションの名前
- 達成することの短い説明
- 種類: 「プロジェクト作業」または「ミッション」
- 参加先: `agent-flow` は次に実行可能な工程、`agent-amigos` は役割名
- 主操作: 「参加する」

`workload`、`busDir`、`executor`、`claim`、`owner-picks`、roles JSON、run IDなどの
内部用語は通常表示しない。診断に必要な識別情報はエラー詳細へだけ含める。

非同期処理中は押したカードのボタンを「参加しています…」として無効化する。
成功時はカードを再取得し、参加済みまたは募集終了の項目を一覧から除く。
失敗時はカード内に原因と再試行方法を表示し、スクリーンリーダー向けに `role="alert"` を付ける。

## agent-flowの参加動作

`agent-flow daemon` が対象runと同じローカルバスまたはGitバスを監視している場合、daemonは
claim可能な工程数に応じてワーカーを自動起動する。このためdaemon運用中は通常、手動参加は不要である。

手動の「参加する」は補助導線とし、このPC上でrun限定のワーカーを1つ起動する。

```text
agent-flow --bus <bus> --run-id <run-id> work \
  --node-id dashboard-<host>-<suffix> --idle-exit
```

ワーカーは依存関係を満たした次の工程を既存の決定的claimで取得し、作業がなければ終了する。
UIは工程を直接予約しない。予約だけを作ると実行主体のないclaimが残り、工程を止めるためである。

対象は、非終端runのうち未取得で依存関係を満たした工程が1件以上あるものとする。
`agent-project` 由来かはrun IDから関連タスクを導出し、カードへプロジェクト名を表示するために使う。
任意のagent-flow runも、同じバス上で参加可能なら同じ動作を使える。

## agent-amigosの参加動作

既存のミッション一覧と `amigos:claim` IPCを再利用する。必須または任意ロールのうち、
未充足で手動参加可能な役割をカードとして表示する。

参加時はホームのcommandsへ既存の `claim` 指示を投函し、常駐daemonに処理を委ねる。
参加方式が応募後選定の場合は、ボタン文言を「参加を申し込む」とし、確定前であることを成功表示で伝える。

## WindowsからWSLを使う場合

Windows版Dashboardでは、flowワーカーをWindowsネイティブで探さず `wsl.exe` 経由で起動する。
既存のWSL実行ヘルパーと同じ規則を共有する。

- `\\wsl.localhost\<distro>\...` / `\\wsl$\<distro>\...` はUNCからdistroとLinuxパスを導出する
- Windowsドライブパスは `/mnt/<drive>/...` へ変換する
- UNCにdistroが無ければ既定のWSL distroを使用する
- 引数は配列または安全に引用した `sh -lc` スクリプトとして渡す
- WSL側の `LANG` / `LC_ALL` は `C.UTF-8` にし、日本語エラーの文字化けを防ぐ

起動前にWSL内で `command -v agent-flow` を確認する。見つからない場合は参加処理を開始せず、
「WSLにagent-flowがインストールされていません」と回復方法付きで表示する。

プロセスはDashboardの画面更新から独立して継続させる。ただし起動結果を返せるまでは監視し、
即時終了した場合はstderrを利用者向けエラーへ変換する。長時間の作業ログは既存の実行画面で確認する。

## コンポーネントとデータフロー

1. main側が既存のdelegation正規化ビューとプロジェクトrun情報から参加候補を返す。
2. rendererは候補をカードへ描画し、参加可能なときだけタブを表示する。
3. 「参加する」で workload別IPCを呼ぶ。
4. flowはローカルまたはWSLで1回限りのworkerを起動する。
5. amigosは既存のclaim commandを投函する。
6. 成功後に候補を再取得し、表示を更新する。

書き込みは既存のエンジン契約だけを通し、rendererからバスへ直接ファイルを書かない。

## エラー処理

- daemonが既に自動参加済み: 最新状態へ更新し「すでに作業が始まっています」
- 参加競合で別ノードが先着: エラーではなく「別の参加者が開始しました」として一覧更新
- flow停止／一時停止: 「全体設定でagent-flowを再開してください」への案内
- WSL未導入: 「WSLが利用できません」
- WSL内にagent-flowがない: インストール先とPATH確認を案内
- bus/runが消えた: 「募集は終了しました」として一覧から除く
- amigos daemon停止: 投函は受理し、処理待ちであることを表示

## テスト

- 参加候補があるときだけ「参加」タブが表示される
- UIに内部用語や旧委譲操作が戻らない
- flow参加は `work --idle-exit` をrunとbus付きで起動する
- daemonの自動参加と手動参加が競合しても二重実行しない
- amigos参加は既存claim IPCへ正しいmission/roleを渡す
- Windowsパス、WSL UNC、distro指定、既定distroを正しく変換する
- agent-flow不在、WSL不在、即時終了を利用者向け文言に変換する
- ボタンのbusy状態、カード内エラー、キーボード操作を確認する

## 対象外

- UIからdaemonを常駐起動・停止する機能
- 公示、落札者選定、受け入れ、差し戻し、中止
- flowの工程を利用者が直接選ぶ機能
- flowへの応募後選定方式の追加
- 作業ログを参加画面へ複製すること
