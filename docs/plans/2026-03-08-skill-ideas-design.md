# スキル拡張・新規スキル案（2026-03-08）

## 背景

現在 33 個のスキルが `.github/skills/` に存在する。
`docs/feature-proposals.md` には多くの案が記録されているが、以下は**まだ未実装**または**未提案**の領域。
本ドキュメントでは、それらを整理し優先度付きで提案する。

---

## 1. 新規スキル案

### 1.1 `security-auditor` — セキュリティ監査スキル 🔴 新規提案

**概要**: コードベースのセキュリティリスクを体系的に検出・報告し、修正を支援する。

**背景**: 現在、セキュリティ専門のスキルが存在しない。`code-reviewer` はセキュリティ観点も含むが、深掘りには不足している。

**主な機能**:
- OWASP Top 10 に基づく脆弱性パターン検出（インジェクション・XSS・認証欠陥等）
- 依存関係の既知 CVE チェック（npm audit / pip-audit 等の解釈支援）
- シークレット漏洩（ハードコードされた API キー等）の検出
- セキュリティ修正の優先度付きレポート（Critical / High / Medium / Low）
- セキュリティテスト戦略の提案（SAST / DAST / ペネトレーションテスト計画）

**トリガー例**: 「セキュリティ診断して」「脆弱性を確認して」「OWASP に準拠してるか確認して」

### 1.2 `db-schema-designer` — リレーショナル DB スキーマ設計スキル 🟡 未実装（既提案）

**概要**: リレーショナルデータベースのスキーマ設計・正規化・マイグレーション戦略を支援する。

**背景**: `dynamodb-designer` は NoSQL 特化。RDBMS（PostgreSQL・MySQL・SQLite）向けの設計支援が不足している。

**主な機能**:
- Mermaid erDiagram によるER図生成
- 正規化（1NF〜3NF・BCNF）と非正規化のトレードオフ分析
- インデックス設計（カバリングインデックス・複合インデックス）の推奨
- マイグレーションスクリプト（Flyway / Liquibase / Alembic / Prisma Migrate）のテンプレート生成
- パーティショニング・シャーディング戦略の提案

**トリガー例**: 「DBスキーマを設計して」「ER図を作って」「正規化を確認して」「マイグレーションを設計して」

### 1.3 `test-strategy-planner` — テスト戦略プランナー 🟡 未実装（既提案）

**概要**: プロジェクト全体のテスト戦略（単体・結合・E2E・非機能）を設計し、CI への組み込み計画を立てる。

**背景**: `react-frontend-unit-tester` は React 単体テストに特化。テストピラミッド全体を俯瞰した戦略設計スキルがない。

**主な機能**:
- テストピラミッドに基づくカバレッジ目標策定（Unit 70% / Integration 20% / E2E 10% 等）
- E2E テストフレームワーク選定ガイド（Playwright vs Cypress vs Selenium の比較）
- テストデータ管理戦略（フィクスチャ・ファクトリ・シーディング）
- 非機能テスト計画（パフォーマンス・アクセシビリティ・セキュリティテスト）
- CI/CD パイプラインへのテスト統合設計（並列実行・フェールファスト戦略）

**トリガー例**: 「テスト戦略を設計して」「テストピラミッドを整理して」「どのテストフレームワークを使えばいい？」

### 1.4 `python-backend-coder` — Python バックエンド実装スキル 🔴 新規提案

**概要**: Python を使ったバックエンド API の実装を、要件確定済みの前提で進める。`react-frontend-coder` の Python バックエンド版。

**背景**: 実装スキルが React + TypeScript に偏っている。Python は AI/ML 系プロジェクトや API 開発で最も需要が高い言語。

**主な機能**:
- FastAPI / Django REST Framework / Flask での API 実装ガイド
- 認証・認可（JWT・OAuth2・APIキー）の実装パターン
- データベース接続（SQLAlchemy / Tortoise ORM / Django ORM）
- 非同期処理（asyncio / Celery）の実装支援
- `api-designer` が生成した OpenAPI スキーマからの実装への変換

**トリガー例**: 「PythonでAPIを実装して」「FastAPIでバックエンドを作って」「Djangoでエンドポイントを追加して」

### 1.5 `nextjs-fullstack-coder` — Next.js フルスタック実装スキル 🔴 新規提案

**概要**: Next.js App Router を使ったフルスタック実装を支援する。`react-frontend-coder` の Next.js 特化版。

**背景**: `react-frontend-coder` は Vite + Zustand ベースで SPA に特化。Next.js App Router（Server Components・Server Actions）に特化したスキルが不足している。

**主な機能**:
- App Router（layouts・loading・error UI）の実装パターン
- Server Components vs Client Components の選択基準
- Server Actions を使ったフォーム・データ変更の実装
- Route Handlers による API 実装
- next/image・next/font・Metadata API の最適活用

**トリガー例**: 「Next.jsで画面を作って」「App Routerで実装して」「Server Actionsでフォームを作って」

### 1.6 `i18n-localizer` — 国際化・多言語対応スキル 🔴 新規提案

**概要**: アプリケーションの国際化（i18n）実装と多言語ローカライゼーション（l10n）を支援する。

**背景**: グローバル展開が必要なプロダクトで i18n 対応をゼロから設計する際の指針となるスキルが存在しない。

**主な機能**:
- i18n ライブラリ選定（react-i18next / next-intl / vue-i18n 等の比較）
- 翻訳キー命名規約と JSON/YAML 構造設計
- 日付・数値・通貨フォーマット（Intl API 活用）の実装
- RTL（Right-to-Left）レイアウト対応ガイド
- 翻訳漏れ検出とCI統合（i18n-lint 等）

**トリガー例**: 「多言語対応して」「i18nを実装して」「英語と日本語に対応させて」

### 1.7 `accessibility-auditor` — アクセシビリティ監査スキル 🔴 新規提案

**概要**: Web アプリのアクセシビリティ（a11y）をレビューし、WCAG 2.1 AA 準拠に向けた改善を支援する。

**背景**: `ui-designer` はレイアウト・スタイルに特化しており、アクセシビリティの体系的な監査機能がない。障害者差別解消法改正・EU EAA 等の法的要件への対応ニーズも高まっている。

**主な機能**:
- WCAG 2.1 チェックリスト（知覚可能・操作可能・理解可能・堅牢）に基づくレビュー
- ARIA ロール・ランドマーク・ライブリージョンの適切な使用確認
- キーボードナビゲーション・フォーカス管理の検証
- 色コントラスト比の計算と改善提案（AA: 4.5:1 / AAA: 7:1）
- スクリーンリーダー向けコンテンツの最適化ガイド

**トリガー例**: 「アクセシビリティをチェックして」「WCAG準拠を確認して」「a11yをレビューして」

### 1.8 `performance-profiler` — パフォーマンスプロファイリングスキル 🔴 新規提案

**概要**: アプリケーションのパフォーマンスボトルネックを特定し、最適化手順をガイドする。

**背景**: `react-best-practices` は React 最適化ルール集だが、実際のプロファイリング手順・ボトルネック特定・改善優先度付けを体系的にガイドするスキルがない。

**主な機能**:
- Core Web Vitals（LCP・FID/INP・CLS）の計測と改善ガイド
- React Profiler / Chrome DevTools を使ったレンダリングボトルネック特定
- バンドルサイズ分析（webpack-bundle-analyzer / rollup-plugin-visualizer）
- バックエンド API レイテンシ分析（N+1問題・スロークエリ検出）
- パフォーマンスバジェットの策定と CI チェック

**トリガー例**: 「パフォーマンスを改善して」「なんで遅い？」「Core Web Vitalsを改善して」

### 1.9 `incident-responder` — インシデント対応スキル 🔴 新規提案

**概要**: 本番障害発生時の調査フロー・暫定対応・RCA（根本原因分析）・ポストモーテム作成を支援する。

**背景**: `systematic-debugging` は開発環境でのデバッグに特化。本番インシデントの対応（トリアージ・オンコール対応・振り返り）を体系化するスキルがない。

**主な機能**:
- インシデントトリアージフロー（影響範囲・重大度判定）
- ログ・メトリクス・トレースからの原因特定ガイド（オブザーバビリティ3本柱）
- 暫定対応（ロールバック・フィーチャーフラグ無効化・トラフィック切り替え）の手順
- タイムライン記録テンプレートと関係者向けステータスアップデート文章生成
- ポストモーテムドキュメント（5Whys・再発防止策）の作成支援

**トリガー例**: 「障害が発生した」「インシデント対応して」「ポストモーテムを書いて」「RCAを整理して」

### 1.10 `migration-planner` — 技術移行計画スキル 🔴 新規提案

**概要**: フレームワーク・言語・DB・インフラの技術移行計画を段階的に設計する。

**背景**: レガシーシステムのモダナイゼーション（Vue2→Vue3、Class Component→Hooks、モノリス→マイクロサービス等）を支援する専用スキルがない。

**主な機能**:
- 現状分析（依存関係マップ・リスク箇所の特定）
- 移行戦略の選択肢提案（Big Bang / Strangler Fig / Branch by Abstraction）
- 段階的移行計画（フェーズ分け・並行稼働期間の設計）
- 移行完了基準と検証チェックリスト
- ロールバック計画の策定

**トリガー例**: 「Vue 2からVue 3に移行したい」「クラスコンポーネントをHooksに書き直して」「モノリスをマイクロサービスに分割したい」

## 2. 既存スキル機能拡張案

### 2.1 `scrum-master` v2 — 並列スプリント・依存グラフ対応

**概要**: 依存関係のないタスクを並列サブエージェントで同時実行できるようにする。

**背景**: 現在は逐次実行のみ。バックログのタスク間に依存関係がない場合でも直列実行しているため、スループットに改善余地がある。

**拡張内容**:
- バックログ解析時に依存関係グラフ（DAG）を構築
- 独立タスクを並列エージェントに委譲（最大 N 並列）
- スプリントの残り時間見積もりを依存グラフ経路長から算出
- スプリントプランニング時に「クリティカルパス」を表示

---

### 2.2 `code-reviewer` v2 — AI コード特有パターン検出

**概要**: AI 生成コードに特有のアンチパターン（過度な汎用化・重複ロジック・意図不明な抽象化）を検出する観点を追加する。

**背景**: AI エージェントによるコード生成が増加した現在、「動くが意図が読めない」「同じロジックが複数箇所に点在する」等の問題が増えている。

**拡張内容**:
- AI 生成コード特有の臭い検出ルール追加
  - 過度に汎用なインターフェース（将来的な拡張を見越しすぎた設計）
  - 使われない引数・オプション（YAGNI 違反）
  - コピー&ペースト的な重複（バリアント違いの類似コード）
- GitHub PR コメントとしてフォーマットされた出力オプション
- `.copilot-review-rules.md` から追加ルールを読み込む拡張ポイント

---

### 2.3 `git-skill-manager` v2 — スキル依存グラフ可視化・ヘルスチェック

**概要**: インストール済みスキルの依存関係を Mermaid グラフで可視化し、定期ヘルスチェックを実行する。

**背景**: スキル数が 30 超になり、どのスキルがどこで使われるかの全体像が把握しにくくなっている。

**拡張内容**:
- `SKILL.md` の `depends_on` フィールド（feature-proposals.md §2.1 参照）を解析してグラフ生成
- スキルの「使用中・未使用・破損」ステータスの定期ヘルスチェック
- バージョン追跡の強化：`central_version` との差分を `git diff` 風に表示
- `--graph` オプションで依存グラフを Mermaid flowchart として出力

---

### 2.4 `ltm-use` v4 — 記憶クラスタリング・類似記憶推薦

**概要**: 蓄積された記憶を意味的にクラスタリングし、新しい記憶保存時に類似の既存記憶を提示して重複を防ぐ。

**背景**: 記憶が増えると類似内容が重複して蓄積される問題がある。現在は重複チェックがなく、`cleanup` 操作も手動。

**拡張内容**:
- 保存時に既存記憶との類似度スコアを計算し、閾値以上なら「類似記憶あり」と警告
- `recall` 時に意味的に近い記憶をランキング表示（キーワード完全一致だけでなく文意で検索）
- `cleanup` を自動化するスケジューラ：評価スコアが低く古い記憶を自動アーカイブ
- 記憶のタグを自動付与（登録テキストから主要トピックを抽出）

---

### 2.5 `brainstorming` v2 — 競合分析・トレードオフマトリクス機能

**概要**: アイデア検討時に既存の競合ソリューションとの比較・トレードオフマトリクスを自動生成する。

**背景**: 現在は「2〜3のアプローチ提案」で止まっているが、各アプローチの具体的な比較マトリクスがあるとユーザーの意思決定が早まる。

**拡張内容**:
- アプローチごとの比較表（工数・リスク・保守性・拡張性・学習コスト）を自動生成
- 既存スキルとの重複・シナジー分析（「このアイデアは X スキルでカバーできる可能性あり」）
- 決定ログ（Decision Record）を設計ドキュメントに自動埋め込み

---

### 2.6 `document-reviewer` v2 — AI 生成文書の品質チェック強化

**概要**: AI エージェントが生成した設計書・仕様書に特有の問題（曖昧な主体・矛盾・ハルシネーション的記述）を検出する観点を追加する。

**背景**: AI 生成文書が増えた現在、「一見もっともらしいが実装できない要件」「矛盾した制約」が含まれるリスクが高まっている。

**拡張内容**:
- 主体の曖昧さ検出（「システムが判断する」→「何がどのロジックで判断するか」を要求）
- 相互矛盾する要件の自動検出
- 実装可能性チェック（「リアルタイム」「瞬時に」等の非現実的な要件のフラグ立て）
- 非機能要件の欠落チェック（セキュリティ・パフォーマンス・可用性要件）

## 3. フレームワーク機能強化案（未実装項目の詳細化）

### 3.1 スキル依存関係の明示的管理（feature-proposals.md §2.1 の詳細化）

**SKILL.md フロントマター拡張**:

```yaml
---
name: react-frontend-unit-tester
description: "..."
metadata:
  version: "1.0.0"
  depends_on:
    - name: react-frontend-coder
      reason: "テスト対象のコンポーネントが存在する前提"
    - name: react-best-practices
      reason: "パフォーマンス観点のテスト戦略"
  recommends:
    - name: code-reviewer
      reason: "テスト完了後のコードレビューに推奨"
---
```

**実装コンポーネント**:
- `git-skill-manager` での依存解決ロジック（install 時に depends_on を自動インストール）
- `scrum-master` でのロード順序最適化
- 依存未充足時の警告メッセージ

---

### 3.2 スキルコンポジション（feature-proposals.md §2.4 の詳細化）

**composite スキルのフォーマット**:

```yaml
---
name: fullstack-feature-workflow
type: composite
description: "要件定義からフルスタック実装まで一気通貫で進めるワークフロー"
metadata:
  version: "1.0.0"
steps:
  - id: s1
    skill: requirements-definer
    output_file: "requirements.json"
  - id: s2
    skill: api-designer
    depends_on: [s1]
    input_file: "requirements.json"
    output_file: "api-spec.yaml"
  - id: s3
    skill: db-schema-designer
    depends_on: [s1]
    input_file: "requirements.json"
    output_file: "schema.md"
  - id: s4
    skill: react-frontend-coder
    depends_on: [s2]
    input_file: "api-spec.yaml"
  - id: s5
    skill: python-backend-coder
    depends_on: [s2, s3]
    input_files: ["api-spec.yaml", "schema.md"]
  - id: s6
    skill: react-frontend-unit-tester
    depends_on: [s4]
  - id: s7
    skill: test-strategy-planner
    depends_on: [s4, s5, s6]
---
```
