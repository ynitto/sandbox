# gitlab-idd: Design Doc 運用統合

> 作成日: 2026-05-11

## 概要

gitlab-idd v4.3.0 でイシューを **living design doc（生きた設計書）** として運用するための拡張。
リクエスター・ワーカー・レビュワーの全ロールにわたって、設計の記録と評価を組み込む。

---

## 背景・動機

現行の gitlab-idd では、イシュー本文は「タスク仕様書」（目的・受け入れ条件・スコープ）として機能しているが、
設計の記録場所が定まっていない。ワーカーはコメントにスカウトマップを投稿するが、
イシュー本文が設計文書として機能していないため次の問題が生じていた:

- 「なぜこの設計を選んだか」がレビュー時に追えない
- 要件不明確なまま実装が進み、後工程で手戻りが発生する
- 設計の品質がレビュー評価に含まれていない

このセクションの削除・改名は絶対に禁止。design doc セクションはすべて**このセクションより前**に配置する。

## ロール別の変更詳細

### リクエスター（イシュー投稿）

**追加: ステップ 3-0 — イシューテンプレートの確認**

`_body.md` 作成前に GitLab の project issue template を確認する:

```bash
python scripts/gl.py list-issue-templates
# テンプレートあり → get-issue-template --name "{name}" で内容取得
# テンプレートなし → references/design-doc-template.md のデフォルトを使用
```

**変更: `_body.md` テンプレートの拡張**

既存セクションを維持しつつ、design doc セクションを前に追加:

```
## 背景・目的          ← 新規（旧「## 目的」を拡張）
## 問題定義            ← 新規
## 提案ソリューション   ← 新規
## 代替案              ← 新規
## リスクと対策         ← 新規
── 以下は既存セクション（変更なし）──
## 受け入れ条件
## 実装スコープ
## 技術制約
## ターゲットブランチ  ← 必須・削除禁止
## 依存イシュー
## 参考情報
```

---

### ワーカー（イシュー実行）

**追加: ステップ 3-6 — イシュー本文に設計を記録**

スカウトマップ（ステップ 3-5）完了後、コードベース探索で得た情報を元に design doc セクションを補完する:

1. `get-issue {issue_id} --get description` で現在の本文を確認
2. 空欄・不足セクションを補記（`_design_update.md` 作成）
3. 要件が不明確な箇所は `⚠️ 推測:` プレフィックスで明示
4. `update-issue {issue_id} --body-file _design_update.md` で本文更新
5. 設計記録コメントを投稿

**推測マークの使用例**:
```markdown
## 提案ソリューション

⚠️ 推測: イシューの記述から X モジュールへの変更と解釈。
`path/to/file.py` の Y 関数を対象に実装する方針とする。
要件に誤りがあればイシュー本文を直接修正してください。
```

---

### レビュワー（レビュー・評価）

**追加: ステップ 3-2b — Design Doc 品質評価**（requester-review.md および non-requester-review.md）

| チェック項目 | 基準 |
|------------|------|
| `## 背景・目的` が明確か | 「なぜこれが必要か」が 1 文で説明できるか |
| `## 問題定義` が具体的か | 現状の課題が列挙されているか |
| `## 提案ソリューション` の根拠があるか | 「なぜこの方法か」が説明されているか |
| 代替案が検討されているか | 少なくとも 1 つの比較候補があるか（S サイズは省略可） |
| リスクが識別されているか | 技術的・ビジネス的リスクが 1 つ以上記載されているか |
| 推測（`⚠️ 推測:`）が多すぎないか | 主要セクションの半数以上が推測なら設計として不十分 |
| 実装と設計の整合性 | コード変更が提案ソリューションに沿っているか |

design doc セクションが存在しない場合（旧フォーマット）は全項目をスキップする（後方互換）。

レビューコメントテンプレートに `### Design Doc 評価` セクションを追加（内容がない場合は省略可）。

---

## 後方互換性

- **既存イシュー**: design doc セクションのない旧フォーマットのイシューはそのまま動作する。ワーカーのステップ 3-6 ではセクションが存在しない場合に追加するだけで、既存の `## 受け入れ条件` や `## ターゲットブランチ` は変更しない。
- **レビュー評価**: design doc セクションが存在しない場合は品質評価をスキップする。

---

## 検証手順

1. `python scripts/gl.py list-issue-templates` が JSON を返すことを確認
2. リクエスターロールで新規イシューを作成し、`_body.md` に design doc セクションが含まれることを確認
3. ワーカーロールでイシューを拾い、ステップ 3-6 でイシュー本文が更新されることを確認
4. レビュワーロールで review コメントに `### Design Doc 評価` が含まれることを確認
5. 旧フォーマット（`## 目的` のみ）のイシューで `## ターゲットブランチ` が正常に読み取れることを確認
