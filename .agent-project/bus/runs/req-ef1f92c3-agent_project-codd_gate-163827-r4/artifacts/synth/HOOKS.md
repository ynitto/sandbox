# 汎用フック境界の設計（agent_project）

本体が外部ツールへ依存せず、差し込み点だけを持つための境界。**本体は「能力」だけを知り、それを
満たす module が何という名前かは知らない。** 名前の所有権は設定 `hooks:` にあり、未指定なら
sibling ディレクトリを能力で走査して引き当てる。

対象は Python module として差し込むフック。外部 CLI の差し込み点（`verify` / `regression_cmd` /
`intake_cmd` / `inbox` / `notify_cmd` / `--executor`）の正典は
[`agent-project-design.md`](../../docs/designs/agent-project-design.md) §4.1 で、本書はその隣に立つ
別種の口を定義する。

## 決定

設定明示を第一・sibling 能力スキャンを第二とする**単一解決器**（`agent_project/hooks.py` の
`_hook_provider`）へ一本化した。二択のどちらに寄せても要件が片方落ちる。

| 案 | プロバイダ名を本体から追い出せる | 未設定の既存環境で連携が保つ | 判定 |
|---|---|---|---|
| A. sibling を固有名で直に import する | ✗ 本体が外部の版管理下にある名前を握る | ✓ | 却下 |
| B. 設定への明示だけを受ける | ✓ | ✗ 既存インストールは `hooks:` を書いていない | 却下 |
| C. 設定 → sibling 能力スキャン | ✓ | ✓ | **採用** |

分けるべき問いは二つあった。**「名前を誰が持つか」（設定）と「既定でどう見つけるか」（能力
スキャン）** で、これを分けると両方立つ。A のままだとプロバイダ側が module 名を変えただけで本体が
壊れる。名前を設定へ譲れば、本体が知るのは「`detect_wiring` を持つ何か」だけになる。

### 却下した案

- **環境変数でプロバイダを指定する。** 設定が yaml と env の二箇所へ散り、doctor が「なぜ解決に
  失敗したか」を説明できなくなる。agent-project は設定の単一アンカーを root 直下の yaml に置く。
- **sibling 側にマーカー属性を足して宣言させる。** 発見は明示的になるが、プロバイダ側の編集が要る。
  「本体は差し込み点のみ」と言いながら差し込む側に本体都合の記述を強制することになる。必須属性の
  有無が事実上のマーカーとして既に機能しており、足す必要がない。
- **debt にも module フックを置いて3系統を対称にする。** debt の差し込み点はすでに `intake_cmd`
  （プロセス境界）にあり、本体が受けるのは JSON テキストだけ。module を足すと差し込み点が二重に
  なり、片方は必ず遊ぶ。パースは本体同梱の `_parse_intake_records`（検出器非依存）で完結する。

## 自動配線は復活させない

`build_config` は `regression_cmd` / `intake_cmd` を補わない。設定ファイル・CLI・既定の値をその
まま使う。これ以前の版は `repos.json` が実在すると起動時にこの2キーを自動で補っていたので、本版は
振る舞いとして**後方非互換**になる。

| 案 | プロバイダ名が本体から消える | 既存環境で連携が保つ | 判定 |
|---|---|---|---|
| D. 自動配線を `_hook_provider` 経由で復活させる | ✓ | ✓ | 却下 |
| E. 補わない。未結線は doctor が見せ、設定は人が書く | ✓ | ✗ | **採用** |

D は grep 条件を満たす。それでも採らなかったのは、**消したかったのが名前ではなく結合そのもの**
だから。起動時に外部ツールの有無を見て自分の中核設定を書き換える構造は、固有名を隠しても残る。
設定ファイルに書いていない値が実行時に生えてくると、なぜそのコマンドが走っているのかを設定から
追えない。フック経由にすると、その追えなさに「本体は名前を知らない」という体裁が付くぶん、かえって
たちが悪い。

代償は移行の一手間で、これは1回で終わる。導線は doctor の未結線所見（推奨コマンド文字列を `fix:`
へ出す）と README の移行ノート。

未結線所見の severity は `info` に据え置いた。所見を作っているのはプロバイダ側で、本体は
`judgment` を不透明として扱う以上そこへ手を入れる口が無い。加えて `info` は新規インストールに
とっては正しい——連携は任意機能で、使っていない環境で warn を出し続ければ誤報になる。warn が
欲しいのは移行の瞬間だけで、それは所見の severity ではなくリリースノートの担当と判断した。

## 能力表

`hooks.py` の `HOOK_CAPABILITIES` が**本体が外部へ求める契約の全部**。ここに無い属性へは触らない。

| 能力キー | 必須属性 | 呼び出し元 | シグネチャ |
|---|---|---|---|
| `wiring.detect` | `detect_wiring` | `doctor_wiring_findings` | `detect_wiring(regression_cmd=, intake_cmd=, repos_path=, which=, run=) -> judgment` |
| `wiring.findings` | `doctor_findings` | `doctor_wiring_findings` | `doctor_findings(judgment) -> list[dict]` |

`judgment` は本体にとって**不透明**。型も属性も検査しない。配線と findings を別キーにしたのは、
片方だけ持つプロバイダを許すためではなく、`detect_wiring` だけ改名されたときに「配線系統だけ
no-op、findings 系統は生存」という半端な状態で走らせないため。両方揃ったときだけ動く。

## 解決順序

`_hook_provider(capability, cfg=None)` が唯一の入口。例外を投げない。

1. **キャッシュ**（`_HOOK_CACHE`）。能力キー毎にプロセス内で 1 回だけ解決する。不在（`None`）も
   キャッシュする——プロバイダの居ない環境で毎回ディスクを舐めない。
2. **設定明示**。`cfg.hooks` をフルキー（`wiring.detect`）→ 前半キー（`wiring`）の順に引く。
   非空なら `importlib.import_module`。必須属性を全部持てば採用。**失敗しても 3 へ落ちない**——
   人が名前を書いた以上、その意図を黙って別物で置き換えない。
3. **sibling 能力スキャン**。`tools/agent-project/` 直下の `*.py` を**名前の昇順**で走査する。
   `_` 始まりと非識別子（`agent-project.py`）を名前で落とし、残りはソーステキストへ
   `^def <属性>\s*\(` を当てて前置フィルタしてから import する。契約の関数を定義していないファイルを
   総当たり import すると、その副作用（グローバル初期化・パス操作・プロセス起動）を本体が浴びる。
4. 全滅なら `None`。呼び出し側は no-op へ縮退する。

`cfg` 省略時は 2 を飛ばす（cfg を組み立てる前の経路向け）。**呼び出し側は `cfg` を渡すこと**——
省略すると設定明示が効かず、テストの patch も `lambda cap, cfg=None: ...` の形になる。

昇順を固定するのは、候補が複数ある環境（差し替えの途中・別実装の同居）で採用先が glob の返す順＝
ファイルシステム依存になると、同じツリーでもマシンによって配線先が変わるため。

**現行との差（意図した縮小）**: 固有名で import していた頃は sys.path 全体を探せた。能力スキャンは
sibling ディレクトリしか見ない——名前を知らない以上、sys.path 全体を能力で舐めるのはコストと副作用が
非現実的になる。逃げ道が `hooks:` で、名前を書けば経路 2 が sys.path 全体から解決する。

## フォールバック

| # | 状況 | `_hook_provider` | 本体 | doctor |
|---|---|---|---|---|
| 1 | 明示あり・import 成功・契約充足 | module | 通常 | なし |
| 2 | 明示あり・import 失敗 | `None` | no-op | **warn**「指定した配線プロバイダを解決できない」 |
| 3 | 明示あり・必須属性が欠ける | `None` | no-op | **warn**（2 と同じ title。evidence に必須属性名） |
| 4 | `hooks:` が対応表でない／値が module 名でない | 明示なし扱い＝スキャンへ | 通常（当たれば） | **warn**「hooks の設定が読めない」 |
| 5 | 設定なし・スキャン命中 | module | 通常 | なし（採用 module 名を journal へ 1 行） |
| 6 | 設定なし・命中 0 件 | `None` | no-op | **なし**（任意機能の不在は正常） |
| 7 | 候補の import が例外を投げる | 次候補へ。全滅で `None` | no-op | なし |
| 8 | sibling ディレクトリが無い（zipapp 等） | `None` | no-op | なし |
| 9 | 同名の無関係 module へ解決される | 契約チェックで棄却 → `None` | no-op | なし |
| 10 | プロバイダ呼び出しが例外を投げる | — | その系統だけ空へ畳む | なし |

**2〜4 は warn、6 は無言。** ここが要点で、「明示した意図が果たされなかった」と「任意機能を使って
いない」を区別する。固有名で import していた頃はどちらも一律無言で、設定ミスが観測できなかった。

4 が「no-op」でなく「通常」なのは、設定の型不正を `build_config` が正規化で吸収して既定（自動検出）
へ落とすため。**自動検出がたまたま当たっても warn は出す**——書いた設定が効いていない事実は、
代わりが見つかったかどうかとは無関係だから。判断の材料は3層に分かれる。

| 層 | 見るもの | 残すもの |
|---|---|---|
| `configfile._normalize_hooks` / `_hooks_config_error` | 生の設定値の型 | `cfg.hooks` / `cfg.hooks_error` |
| `hooks._hook_resolution_error` | 読めた名前が解決できたか | 理由文字列 |
| `doctor._hook_misconfig_findings` | 上の2つ | finding（warn） |

型不正の検出が configfile にあるのは、生の設定値を持っているのがこの層だけだからで、cfg には
正規化後の対応表しか残らない。両方の層で型を見ると同じ warn を二経路が作り、どちらが出したのかが
運用からもテストからも区別できなくなる。

## テストからの注入点

**`_hook_provider` を patch すれば全系統が差し替わる。** これが唯一の注入点。

| 差し替えたいもの | patch |
|---|---|
| プロバイダ全体 | `mock.patch.object(km, "_hook_provider", lambda cap, cfg=None: fake)` |
| プロバイダ不在 | `lambda cap, cfg=None: None` |
| 系統ごとの片欠け | `lambda cap, cfg=None: fake if cap == "wiring.findings" else None` |
| 解決順序そのもの | `km._HOOK_CACHE.clear()` してから `cfg.hooks` を差し替える |
| 走査対象 | `mock.patch.object(km, "_hook_sibling_dir", lambda: <tmpdir>)` |
| 走査順 | `_hook_scan_siblings(required, <glob を差し替えたディレクトリ>)` を直に呼ぶ |
| 実測 I/O | `doctor_wiring_findings(cfg, which=…, run=…)`（既存の注入引数） |

fake は `detect_wiring` / `doctor_findings` を持つ `SimpleNamespace` で足りる。`_HOOK_CACHE` は
module 辞書として公開する（隠すと patch できない）。テスト間で漏れるので `setUp` で `clear()` する。

## 検証で守る性質

差し込み点は壊れても静かに no-op へ縮退する＝黙って機能が消える。テストが空振りしていても気づけない
ため、次の性質はミューテーション（1つずつ壊して該当テストが赤くなるか）で確認している。

明示指定の失敗で自動検出へ落ちない／キャッシュが効く／前置フィルタが無関係 sibling を import しない／
必須属性チェックが契約不足を棄却する／前半キーを引く／走査順が名前の昇順／`_` 始まりと非識別子を
落とす／片方だけ解決ではプロバイダを呼ばない／プロバイダの例外を畳む／設定ミスの warn が出る／
設定の型不正を記録する。

`- verify:` / `regression_cmd` / `intake_cmd` 側の性質（`id` の冪等キー・レコード単位の検証）は
`TestIntake` が持つ。`id: 0` のような falsy な冪等キーを「id 無し」と誤読すると、同じ発見が毎周期
backlog へ積み上がる——パーサの出力にしか現れず、2 周目以降の件数でしか見えない退行になる。
