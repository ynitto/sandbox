## 処理パイプライン

### Stage 0 — 入力全体の前処理
`strip_ansi(out)` で ANSI エスケープを除去する。カラーコード付き出力だとフェンス開始
記号 ` ``` ` も先頭トークンも認識できなくなるため、最初に必ず行う。

### Stage 1 — コードフェンス優先スキャン
`_code_fence_lines(out)` で Markdown コードフェンス内の行を出現順に集める。
- 開始フェンス判定は行末が ` ```(言語タグ任意) ` であること（「実行してください: ` ```bash `」
  のような前置き散文の同居も開始行として認識）。
- 閉じフェンスが無ければ入力末尾までをそのブロックの内容とみなす。
フェンス内が1行でもあれば `_first_executable_line(fenced, require_shell_syntax=False)` で判定する。
**sh -n 構文チェックを課さない** — フェンスで明示された時点で LLM の意図（これはコマンドである）
を信頼する設計。ヒットすればそれを返し、以降のフォールバックには進まない。

### Stage 2 — フェンス外フォールバック
フェンスがゼロ、またはフェンス内に条件を満たす行が無かった場合のみ実行する。
各生行を `_has_command_like_leading_token(_strip_leading_command_label(_strip_leading_shell_prompt(line.strip())))`
で事前フィルタし（既知コマンド語 / `./` `../` `/` プレフィックス / ハイフン入り CLI 名パターン）、
通過した行だけを `_first_executable_line(candidates, require_shell_syntax=True)` に渡す。
こちらは **sh -n 構文チェックを追加で課す** — 自然文（散文）が単純コマンドとして誤採用されるのを防ぐため。

## 行単位の正規化（両パス共通、`_first_executable_line` 内で適用順）
1. `raw_line.strip()` — 前後の空白除去
2. `_strip_code` — ANSI 再除去 + 行**全体**が1組のバッククォートで囲まれている場合のみ両端を除去
   （インラインコードスパンの全体一致のみが対象。部分一致は対象外）
3. `_strip_leading_shell_prompt` — 行頭が `$` + 空白1文字以上ならその `$ ` を1回だけ除去
   （`$(...)` `$VAR` は `$` 直後が空白でないため対象外＝誤剥離しない）
4. `_strip_leading_command_label` — 行頭ラベル `検証コマンド:`（全角コロン `：` も可）を
   `^.*?検証コマンド\s*[:：]\s*` の非貪欲マッチで**変化がなくなるまで反復**除去
   - 非貪欲 `.*?` によりラベル前の散文同居（「以下を実行してください。検証コマンド: `<command>`」）にも対応
   - 反復適用で二重・多重ラベル（「検証コマンド: 検証コマンド: `<command>`」）も収束
   - 対象は固定ラベル語のみ（行内の任意のコロンではない）— `git commit -m "note: fix bug"` の
     ようなコマンド内コロンを誤って割らない

## 採否判定（正規化後の行に対して）
以下すべてを満たす行を「最初の実コマンド行」として確定し、正規化後の文字列をそのまま返す:
- 空文字でない（→ **空行**はスキップ）
- `#` で始まらない（→ **コメント行**はスキップ）
- casefold が `bash`/`console`/`sh`/`shell`/`zsh` のいずれとも一致しない（→ フェンス言語タグ単独行を除外）
- フェンス内: 追加条件なし（sh -n チェック免除）
- フェンス外: `_looks_like_shell_command` の sh -n 構文チェックを通過すること（→ **散文**はここでスキップされる）

満たさない行はスキップして次候補行へ進む。全行が不適合、あるいはフェンス外で事前フィルタを
通過する行が1つも無ければ `None`。

## verbatim 性の保証
正規化ステップ（ANSI 除去・行全体を囲むバッククォート除去・行頭 `$ `・行頭ラベル）は
いずれも行の**外側**しか触らない。行内部の引用符（`"$KIRO_BASE_REV"`）や `$` 変数参照は
どの正規表現の対象にもならず一切変更されない。返却値は「除去後に残った文字列」そのもの、
すなわち verbatim である。

## 除外規則とタスク文言の対応
| タスク文言 | 実装上の根拠 |
|---|---|
| ラベル行（例 `検証コマンド:` のみの行）をスキップ | `_strip_leading_command_label` 除去後に空文字化 → 「空文字でない」判定で弾かれ次行へ |
| 空行をスキップ | 採否判定の空文字チェック |
| コメントをスキップ | 採否判定の `#` 始まりチェック |
| 散文をスキップ | フェンス外は sh -n 構文チェック（`_looks_like_shell_command`）で除外。フェンス内は免除（後述） |
| 最初の実コマンド行を verbatim で返す | 正規化は行の外側のみ操作するため引用符・`$` 変数は無傷のまま返る |

フェンス内で散文チェックを免除する点は「散文をスキップ」という文言と一見矛盾するが、
これは意図的な設計判断（LLM がコードフェンスで明示的に囲んだ時点で意図を信頼する）であり、
t1/t2/t3 いずれの検証でも既存回帰と整合している。矛盾ではなく仕様として明記する。

## 既知の穴（今回スコープ外・t1/t2/t3 一致）
`検証コマンド: $ codd-gate ...`（ラベルと `$` プロンプトが**同一行**）は `None` になる。

原因（適用順から特定): `_strip_leading_shell_prompt` が `_strip_leading_command_label` より
**先に**適用される合成関数の並びのため、この時点では行頭がまだ「検証コマンド」で始まっており
`$` プロンプト判定にマッチしない。ラベル除去後に残る `$ command` は再度プロンプト剥がしに
かけられない（1 パス構成のため）。結果として `$ command` がそのまま候補になり、フェンス外なら
sh -n 構文チェックで弾かれ、フェンス内ならその文字列がそのまま返る——いずれも
「ラベルとコマンドが同一行」という直感的期待とはズレる。

本タスクの完了条件（ラベルとコマンドが別行）および既存回帰 33 件のいずれの範囲にも
含まれないため、今回は仕様として明記するに留め、コード修正は行わない。
