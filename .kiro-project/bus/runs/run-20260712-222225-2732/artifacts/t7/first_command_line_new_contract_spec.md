# `_first_command_line` 新パース契約仕様

対象: `tools/kiro-project/kiro-project.py` `_first_command_line`（現 L2953-2965）
根拠: t1（逐語コード）, t2（呼び出し契約）, t3（テスト規約）, t4（8カテゴリ実測）, t6（クロスチェック）

## 0. 前提の確定（依存タスク間の食い違いの解消）

- t1〜t3 が扱った「フェンス内コマンドを拾えない」旧不具合は HEAD (`9fcf0e9`) で**解消済み**（カテゴリ1-4,7,8 が実測PASS）。本仕様はこれを前提として維持し、再修正しない。
- t4/t6 が独立に確認した**未解消の失敗**（カテゴリ5 `$ ` プロンプト記号、カテゴリ6 箇条書き番号）を本仕様の主題として扱う。以下のルールはこの2カテゴリを解消するための追加規則である。
- t6 が指摘した `synth_verify`docstring と `ensure_verify` の空文字時挙動の不整合は、`_first_command_line` 自体の入出力契約ではなく呼び出し側 (`ensure_verify`) の申し送り事項であり、本仕様のスコープ外として扱う（別タスクでの解消を推奨）。

## 1. 入力契約

- 入力は `out: str`（kiro-cli が返す生テキスト1本）。前置き散文・コードフェンス・箇条書き・見出し・末尾説明文が混在しうる。
- `None` / 空文字も許容する（`(out or "").splitlines()` で空リスト化）。

## 2. 出力契約

- 戻り値は `Optional[str]`。成功時は「装飾を除いた実行可能コマンド1行」、それ以外は `None`。
- 例外は送出しない（現行契約を維持）。

## 3. 決定的処理手順（優先順位: フェンス内 > 素のコマンド行）

1. `_code_fence_lines(out)` でコードフェンス内の行を出現順に収集する（複数フェンスは連結、未閉じフェンスは末尾まで）。**フェンスが1つでも存在すれば、フェンス外は一切見ない。**
2. フェンス内候補がある場合、収集した行を先頭から走査し、各行について §4 の除外ルールと §5 の装飾プレフィックス除去を適用した上で、残った最初の非空行を**無条件で**（`sh -n` 構文チェックなしで）採用する。
3. フェンスが1つも無い場合のみ、`out` の全行に §4・§5 を適用し、`_has_command_like_leading_token` を満たし、かつ `_looks_like_shell_command`（`sh -n` 構文チェック＋全角句読点除外）を通過する最初の行を採用する。
4. 1つも採用できなければ `None`。
5. 複数フェンスがある場合、常に「最初に見つかったフェンス」の先頭採用行を返す（意味的にどのフェンスが本来の検証コマンドかを文脈判定する機能は持たない＝現行仕様を維持、§7 参照）。

## 4. 除外ルール（散文・見出し・箇条書き記号）

候補行として**採用しない**もの:

- **散文**: 全角句読点（`。、！？；：「」『』（）`）を含む行。フェンス外では `_looks_like_shell_command` が担保、フェンス内でも同一基準を適用する。
- **見出し・コメント**: `#` で始まる行（Markdown見出し／シェルコメントの双方を同一基準で除外。フェンス内は既存の `#` コメント除外がそのままカバーする）。
- **空行・言語タグ残骸**: 空行、および `_SHELL_FENCE_LANGUAGE_TAGS`（`bash`/`console`/`sh`/`shell`/`zsh`）に一致する行（現行仕様を維持）。
- **箇条書き記号のみの行**: §5 でプレフィックスを除去した後に残りが空、または既知コマンド語・パス始まりに該当しない行（例: `2. 結果がグリーンになることを確認` はプレフィックス除去後も日本語散文のため除外される）。

これらは行単位の**フィルタ**であり、該当行はスキップして次の候補行へ進む（`None` 即確定ではない）。

## 5. 新規ルール: 装飾プレフィックスの除去（$ プロンプト記号・箇条書き番号）

候補行の除外判定・構文チェックを行う**前**に、各行の先頭から以下を1回だけ剥がす（フェンス内・フェンス外の両経路で共通適用）:

- **シェルプロンプト記号**: 行頭が `$` + 1個以上の空白（正規表現 `^\$\s+`）のみを剥がす。`$(...)` のようなコマンド置換や `$VAR` はこの形に一致しないため保持される（誤剥離しない）。
- **箇条書き番号・記号**: 行頭の `^(?:[-*+]|\d+[.)])\s+`（例: `1. `, `1) `, `12. `, `- `, `* `）を剥がす。数字の直後に空白を挟まない語（`2to3` 等）には一致しないため安全。

剥がした残りの文字列に対して §3・§4 の判定（フェンス内は無条件採用、フェンス外は構文チェック）を適用する。

**適用範囲外（明示的にスコープ外）**: プレフィックス除去後もなお散文と判定される行（インラインコード〔バッククォート〕が文中に埋め込まれ、前後に日本語散文を伴うケース。例: `` 1. `pytest -q` を実行してテストが通ることを確認する。 ``）からのコマンド抽出は行わない。行単位選択という現行アーキテクチャの前提を超える（文中スパン抽出が必要）ため、本仕様では対応しない。

## 6. 後方互換の保証範囲

- 既存 `TestVerifyAssist` の 12 件（`pytest -k first_command_line`）は全て緑を維持する。特にカテゴリ1-4・7・8（bashフェンス／言語タグなしフェンス／前置き散文＋フェンス／前置き散文のみ／複数フェンス／末尾説明文）の戻り値は一切変更しない。
- フェンス内「無条件採用」方針（`sh -n` 構文チェックを課さない）はそのまま維持する。§5 は「除去してから無条件採用する」という前段ステップの追加であり、採用基準そのものを厳格化するものではない。
- フェンス外の `_looks_like_shell_command`（`sh -n` 構文チェック）は必須のまま維持し、判定対象を「装飾プレフィックス除去後の文字列」に置き換えるのみ。
- `_first_command_line` の戻り値型（`Optional[str]`）・無例外契約・`synth_verify` 側の3段フィルタ（None→再試行、非シェル→再試行、恒真式→再試行）は変更しない。
- `_code_fence_lines` / `_strip_code` / `_SHELL_FENCE_LANGUAGE_TAGS` / `_KNOWN_COMMAND_WORDS` の既存契約は変更しない（§5 のプレフィックス除去は `_first_executable_line` の候補行前処理として追加する新規ステップという位置づけ）。

## 7. スコープ外・既知の非対応（意図的に対象外とする範囲）

- 複数フェンスからの意味的選別（どのフェンスが本来の検証コマンドか）は行わない。現行どおり最初のフェンスの先頭採用行を返す。
- 散文中に埋め込まれたインラインコードスパンの抽出（§5末尾参照、カテゴリ6サンプルB相当）は対象外。
- `synth_verify`docstring と `ensure_verify` の空文字時の即時人手回付有無の不整合（t6指摘）は `_first_command_line` の契約外であり、本仕様では扱わない。
- `-k first_command_line` の既存テスト集合に §5 で新規に解消するカテゴリ5・6の回帰テストが未追加であること（t6指摘の検証ギャップ）自体は本タスク（synthesize）の範囲外。実装タスクで `TestVerifyAssist` に `test_first_command_line_<状況>` 命名で追加することを推奨する。
