# gate1 verify: t6〜t9 原因仮説の敵対的検証

## 判定

**verify=fail**

理由: t6〜t9 の各仮説は「機構の再現」や「コード読解」による裏付けはあるが、要求された「実際の失敗スタックトレース行との照合」が欠落している。run 内一次情報（t1/t4/t5, events）では失敗テスト自体が確認できず、仮説を「4件の失敗原因」として成立させる証拠が不足しているため。

## 独立検証で確認した事実

1. 完了条件コマンドを割当worktreeで実行し、`900 passed in 121.30s`、終了コード 0。
2. `artifacts/t1/pytest_result.md` は `900 passed / 0 failed`、`artifacts/t4/pytest_failures_detail.md` も「失敗0件・個別再実行未実施」。
3. run の `events/*.jsonl` を `Traceback|FAILED|=+ FAILURES =+|AssertionError|RuntimeError|timeout` で検索しても該当なし。
4. したがって「4件失敗」を示す nodeid/トレースバック一次証拠が run 内に存在しない。

## t6〜t9 の仮説ごとの評価

1. **t6（/var と /private/var 差）**  
   最小再現ログはあるが、対象4件の失敗トレースバックと結びつく証拠なし。しかも結論自体が「現実装は realpath 防御済みで不成立」。
2. **t7（APFS 大小文字非区別）**  
   実験ログはあるが、対象4件の失敗トレースバックとの結合証拠なし。現行失敗が存在しないため因果を立証不可。
3. **t8（git config 差分）**  
   config 採取はあるが、失敗トレースバック照合なし。結論も「差分はあるが現行テスト失敗原因の証拠なし」。
4. **t9（プラットフォーム前提マッピング）**  
   実装読解ベースで、失敗トレースバック照合なし。前提4の潜在リスク指摘はあるが実害実証なし。

## 根本原因の判定（4件共通か/別々か）

**「4件のテスト失敗」という事象はこの run の証拠上は成立しない。**  
成立している異常は「verify 判定系のタイムアウト境界（120s）と実行時間（約121〜129s）の不整合」による誤NGの疑い（t5 の主張）で、これは **単一の共通原因候補**。  
一方 t6〜t9 は、失敗実体がないため「4件の原因分類（共通/別々）」を確定する土台を欠く。

## 再作業に必要な具体的修正指示

1. 「4件失敗」主張を維持するなら、まず該当 nodeid 4件と `--tb=long` の生トレースバックを artifacts に追加すること。
2. その後、各仮説（t6〜t9）を **トレースバックの該当フレーム行** と 1:1 で対応付けること（「どの行がどの仮説を支持/反証するか」を明記）。
3. 失敗が再現しない場合は、原因仮説タスクの結論を「テスト失敗原因」ではなく「誤NG（verify timeout）検証」に切り替えること。
